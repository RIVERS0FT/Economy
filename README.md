# Economy

Economy 是一款网页端多人在线经济模拟、产业经营、统一资产交易与藏品／商品／工厂单项或捆绑竞价拍卖游戏。

- 项目：`RIVERS0FT/Economy`
- 生产网页：`https://game.riversoft.top/economy/`
- 管理员页面：`https://game.riversoft.top/economy/admin`
- 封禁管理：合入 `https://game.riversoft.top/economy/admin` 的“账号封禁”分区
- 客户端状态版本：`16`
- 世界状态版本：`14`
- 市场需求模型版本：`9`

> 人口经济管理规则：管理员后台可在“概况”内查看完整当前参数和持续时间、应用有期限的稳定需求政策或按政策立即补充人口钱包缺口；参数不设业务上限，只保留下限、安全整数和计算结果范围校验。写操作继续要求幂等，但不要求管理备注、不生成调控记录，且禁止任意编辑人口余额、收入状态或既有订单。


## 当前核心循环

```text
工作获得基础资金
→ 建设同类型工厂数量
→ 选择工厂生产配方并开启持续生产
→ 产成品直接进入共享仓库
→ 在统一资产订单簿交易商品和工厂
→ 消费需求形成最终品订单并沿生产链派生上游流动性，市场储备提供双边流动性
→ 通过不可拆分资产包拍卖交易藏品、商品和工厂
→ 邀请新玩家获得独立宝石
→ 在商店按固定汇率兑换普通货币
→ 调整产业链、库存、资金与收藏
→ 提升服务器计算的总资产和排行榜名次
```

## 当前关键规则

- 当前目录共 31 种商品和 21 种工厂类型；水果、饮料、鱼类、预制餐、纸品和家电均已进入完整生产与需求链。
- 产业链包括 `小麦 → 面粉 → 食品／预制餐`、`甘蔗 → 砂糖 + 水果／奶 → 饮料`、`木材 → 木板／纸浆 → 家具／纸品`、`棉花／毛 → 纺织品 → 服装`、`铁矿石 → 钢材 → 机械`、`原油 → 塑料 + 铜矿石 → 铜材 → 电子产品`，并由 `机械 + 电子产品 → 家电` 汇合为最终耐用品。
- 商品订单只允许玩家订单、消费需求订单和市场储备订单；`food` 食品市场与内部 ID 保持为 `household` 的社会消费市场只生成终端消费买单，31 种商品全部至少属于一个直接需求类别；市场储备使用真实资金和库存同时生成商品买单与卖单，不创建系统工厂订单。
- 消费需求由基础人口、技术人口和专业人口三个真实账户提供资金；每类人口根据收入、储蓄和消费状态计算五分钟预算，不再按活跃玩家数量、库存价值或成交活跃度发行预算。
- 玩家库存数量和库存价值不得扩大市场需求总预算，也不得直接提高某商品的需求份额。
- 三类人口使用真实余额和冻结资金；就业收入预算继续保持 70% 用于最终消费的直接需求，30% 用于沿正式配方反向推导的派生流动性。模型 9 额外提供总基础预算 12% 的稳定需求补充，三周期目标钱包、每周期最多补一周期缺口，稳定预算按 85% 直接需求／15% 派生流动性分配。人口消费成交不再发行普通货币。
- 市场储备每 5 分钟撤销并重挂双边商品订单；买单额度取当期消费预算 25% 与本组基础预算 5% 的较大值，且不得超过储备可用资金；卖单只能冻结真实储备库存，基础总价差为 8%、限制在 4%～24%，每项最多挂目标库存的 25%，并保留 20% 安全库存。储备报价必须避让当前系统盘口。
- 市场需求模型 9 在三类人口真实钱包上增加受控稳定需求补充、最低储备买盘与低成交证据置信度；升级时撤销旧系统订单并释放冻结，保留既有储备资产、玩家订单和玩家资产，世界版本保持 14。
- 模型 9 将原材料、中间品和工业品纳入社会终端消耗：棉花、毛和纺织品对应穿着维护，木材、木板和纸浆对应装修包装，原油和塑料对应能源与日用材料，矿石、金属和机械对应建设维修；所有直接需求共享既有人口真实预算，总基础预算仍为 5700，不因商品数量增加而扩张。
- 直接需求采用需求类别和替代关系；主食、肉蛋奶鱼、水果奶饮料、食品预制餐以及电子产品和家电按相对有效价格、可成交深度和替代弹性平滑调整份额，替代品总预算不增加。
- 同一输出商品存在多条正式配方时形成生产路径替代；同一正式配方包含多个输入时形成互补关系，并通过互补门控优先补足瓶颈输入。
- 生产链采用双向价格传导：上游成本通过成本推动滞后影响下游，下游成交价值和最终需求通过净回值滞后影响上游；多路线成本锚点按路线份额与输入可购性加权，每条生产边至少延迟一个需求周期，并使用最近三个周期 60%／30%／10% 信号、阻尼、涨跌幅限制和基础价回归防止循环放大。
- 商品价格角色由直接需求目录和正式配方图自动判断；水果、鱼类、奶、面粉和电子产品可以同时承担直接消费和下游输入，机械通过家电链自动成为派生中间品，不维护静态角色 ID 名单。
- 商品初始参考价、生产数量、周期秒数和周期成本全部保持整数；周期成本是玩家固定支付的基础生产成本，人口生产工资按该成本乘管理员生产工资系数计算，参考分钟利润仍按固定成本和工厂复杂度固定为 C1=1、C2=3、C3=6、C4=6、C5=8、C6=10、C7=12。
- 农场的小麦、水稻、棉花和甘蔗配方保持同参数：周期 120 秒、单座产量 4、单座周期成本 6；水果由独立果园生产，鱼类由独立渔场生产。
- `steelworks` ID 永久保留，正式名称改为冶炼厂，同时支持铁矿石炼钢和铜矿石炼铜；不新增铜冶炼厂。
- 畜牧场可切换生产肉、蛋、奶和毛；磨坊可生产面粉或砂糖；饮料厂可选择乳制或果汁路线；电子工厂和家电厂分别执行塑料+铜材、机械+电子产品的多输入配方。
- 正式配方统一使用 `inputs[]`；同一周期所有输入必须同时满足并原子扣除，任一输入不足时不得扣除其他原料。
- 工厂按 `facilityTypeId + count` 保存，不存在单座工厂实例、实例 ID、内部仓库或领取产成品。
- 同类型工厂共用运行意图、三种顶层状态、进度、当前配方和周期结算；开启后只能持续生产。
- 工厂异常时保留开启意图；资金、原料或仓库条件恢复后，从新的完整周期自动恢复。
- 所有工厂集群统一使用服务器正式配方；单配方工厂显示唯一配方，多配方工厂可切换，运行中切换在下一完整周期生效且不重置当前进度。
- 工厂详情的生产公式只展示集群参数：运行中按 `participatingCount` 显示本周期输入、输出和成本，停止或异常按 `nextCycleCount` 显示启动后或恢复后的集群参数；周期时长不乘工厂数量，实际结算口径与公式一致。
- 建设卡只显示建造费用和施工时间，并同时显示工厂类型与配方名称；不显示生产周期、单座周期产量或单座周期成本。
- 商品和工厂共用统一限价订单结构，并统一调用 `server/src/order-matching.js` 执行价格优先、同价时间优先、maker price、部分成交、订单状态推进、逐笔 fill 和卖方手续费；商品与工厂模块只提供资产结算和行情记录适配器。不同主体且价格交叉的可成交订单必须立即按 maker price 撮合；同一玩家的新商品或工厂订单若会与自己的反向订单交叉，必须在冻结资产前拒绝。
- 系统订单之间继续禁止互相成交；消费需求与市场储备不能通过虚假自成交消除交叉，储备报价必须重新定价或撤掉对应方向。市场处理完成后，同一商品同时存在系统买卖盘时必须满足最高系统买价严格低于最低系统卖价。
- 订单簿按同资产、同方向、同价格匿名聚合为价格档位，只累加有效订单的当前剩余数量；服务器订单、同价时间优先、逐单撤销、成交明细和单张卖单手续费保持独立。
- 玩家通过统一订单簿卖出商品或工厂时，按单张卖单累计成交总额精确收取 1% 手续费，使用向下取整的累计差额且不设最低手续费；手续费按基础人口 20%、技术人口 60%、专业人口 20% 形成市场服务就业收入。
- 工厂运行中可以提交卖单；冻结数量立即退出当前参与数量，并停止产生商品和运营成本。
- 商品和工厂估值均使用最近一次统一订单簿真实成交价；模拟历史、挂单、基础价、系统价值和拍卖价格不得参与估值，从未真实成交时估值为 0。
- 增长榜、生产榜和交易榜按北京时间 `Asia/Shanghai` 每周一 00:00 结算；交易榜成绩等于周期内统一订单簿实际卖出成交额，撤单未成交剩余量、买入成交额和拍卖成交均不计入。
- 工作冷却固定为 3 秒，不随连续工作次数增加；每次有效点击继续直接发行新普通货币，不读取通胀、全服货币量或玩家资产。
- 商店兑换继续按固定汇率直接发行普通货币；礼品码和管理员发放也按各自规则记录发行。系统不设置人口侧货币回收、余额衰减或发行总量上限；稳定需求补充只按人口钱包缺口发行，并受当前目标钱包参数与同周期补充比例约束。
- 玩家支付的生产周期成本保持目录固定，不乘生产工资系数；人口生产工资按固定成本乘不低于 50% 且不设业务上限的临时政策系数计算，高于固定成本的差额记为工资补贴发行，低于固定成本的差额记为工资扣留回收。工厂建造费、仓库扩容费和市场卖出手续费仍全部转为人口就业收入；建造业固定按基础人口 60%／技术人口 30%／专业人口 10% 分配；生产运营岗位按 C1～C7 复杂度分配。
- 市场页面不得增加人口经济区域；管理员“概况”内增加只读人口经济区域，展示三类人口钱包、就业来源、施工托管、生产复杂度工资、稳定需求补充、货币发行统计，以及完整当前人口政策参数、总持续时间、已持续时间和剩余时间。
- 共享仓库允许无限扩容；当前等级 `L` 的容量增量为 `250 + 50 × (L - 1)`，扩容费用为 `150 + ceil((当前实际总容量 - 500) × 0.6)`，费用按基础人口 50%、技术人口 40%、专业人口 10% 形成仓储建设就业收入。
- 共享仓库只显示可用或冻结数量大于零的商品，并复用统一商品 SVG 图标；商品内容区使用容器查询，在不同宽度下每行显示 2／3／4／5／6 项。
- 生产管理区采用工厂集群主从布局：大于等于 1600px 时建设卡、两列工厂集群选择器和自然高度的当前详情卡紧凑排列，详情宽度上限为 680px；1381px–1599px 保持三列单列选择器，961px–1380px 保持建设卡左列、选择器与详情右列，低于 960px 改为单列。移动端不展开全部详情；移动端不显示独立“当前工厂”栏；工厂卡固定三列，名称位于左上、统一工厂 SVG 居中、右下角仅显示数量数字，绿色／红色／灰色分别表示运行／异常／停止，点击卡片直接打开底部详情且不保留选中态。
- 工厂详情选择与建设类型选择使用独立客户端状态；状态轮询只刷新详情数据，不重置当前工厂或关闭已打开的移动悬浮框。当前工厂不再拥有时回退到正式目录中的第一种已拥有工厂，无工厂时关闭悬浮框。
- 共享仓库、建设新工厂、工厂集群选择器和桌面详情卡属于生产页同一一级平面；大于 720px 时四边内边距统一为 16px，不大于 720px 时统一为 12px，标题左上锚点保持一致。移动悬浮详情框属于 Overlay，只允许自身内容区纵向滚动。
- 顶部状态栏直接由固定玻璃胶囊承载五列状态内容，不包含 `ScrollArea`、内部滚动视口或自绘滚动条；窄桌面和移动端使用紧凑数值与自适应字号，状态栏最上层 `::after` 内描边必须在卡片后方滚动时保持连续。
- 滚动条使用全局统一尺寸并按最近输入方式工作：鼠标／触控板模式提供可操作的横纵轨道，触控模式完全隐藏横向轨道并保留原生横滑，只在实际纵向滚动后显示可触摸操作且会自动隐藏的纵向轨道；市场资产目录禁止吸附并支持无级滑动，本地成交使用单一双轴虚拟视口。
- 状态栏、导航、商品和工厂资产使用统一内联 SVG；工厂厂房图标与机械商品齿轮图标保持独立。
- 服务器只保存权威经济状态；玩家活动和成交展示日志保存在当前浏览器。
- 游戏状态使用全局世界修订号排序，并按目录、玩家、市场、拍卖、排行榜五个分区增量同步。首次进入返回五个完整分区；后续请求同时提交全局修订号与各分区内容哈希，只返回哈希变化的分区。全局修订变化但当前玩家视图未变化时只返回新的全局修订号和空补丁；同全局修订号轮询继续在进入 SQLite 事务前返回轻量确认。每个 `GET state` 响应都在分区 envelope 顶层携带响应生成时的 `serverNow`，该字段不进入世界 JSON、`EconomyState` 或分区哈希。
- 客户端状态版本当前值与上一版本兼容下限统一定义在 `server/shared/economy-state-version.js`；服务器序列化、浏览器分区合并、类型声明、README 与权威设计由 `scripts/verify-client-state-version.mjs` 强制保持一致。
- 权威动作响应固定只返回 `{ result: { ok, message }, revision }`，不附带完整状态、分区补丁、分区修订、`unchanged`、`serverNow` 或动作内部结算字段；浏览器收到确认后立即使用动作前的全局修订号和现有分区哈希请求 `GET state`，再合并变化分区。
- 客户端默认每 5 秒轮询状态，可选 3／5／10 秒；空闲 `GET state` 不得写回未变化的世界 JSON。全局游戏模型不得使用每秒时钟驱动整页重渲染，倒计时只在实际需要的页面局部维护。局部倒计时统一从 `serverNow` 校准的共享单调服务器时钟读取当前时间，迟到或较旧响应不得让工作冷却和其他倒计时回退，`lastProcessedAt` 不得在每次轮询时被重新解释为当前服务器时间。
- 客户端只接受不低于当前值的状态修订号；权威动作期间必须取消旧轮询并暂停新轮询，工作按钮立即显示“处理中”。动作已经提交但补拉失败时不得改写为操作失败，恢复轮询后继续补拉；分区修订使用状态内容哈希，仅用于判断正文是否需要传输，不替代全局修订号的迟到响应排序。
- `/economy-api/game/` 的大型 JSON 响应必须使用 gzip 压缩，避免状态补丁占满服务器出口带宽。
- 主页账号认证使用单进程短缓存：状态读取 10 秒、普通写操作最多 2 秒、管理员不使用缓存；同会话并发认证合并，LRU 上限 5,000 条，缓存键只保存 Cookie 的 SHA-256 摘要。
- Economy 注册完成时点是统一账号第一次创建 Economy 玩家档案；邮箱验证码和已登录主页账号首次进入共用相同建档、邀请归因与注册 IP 检测逻辑。
- 宝石是独立于普通货币的整数资产，不参与市场、生产、总资产或排行榜。每名玩家拥有永久 8 位邀请码和专属分享链接；新玩家通过分享链接首次建档，或注册后 24 小时内在设置页填写邀请码时，邀请人立即获得 10 宝石，被邀请人不获得宝石。宝石可在独立“商店”按服务器固定汇率 1 宝石兑换 10 普通货币，单次 1～100 宝石，只允许单向兑换且不可撤销。
- 一个账号只能绑定一次邀请关系，分享链接与手动邀请码互斥。玩家不提供经济状态重置能力，资金、库存、工厂、订单、统计、宝石、邀请码、邀请关系、宝石流水和封禁记录均持续保留。
- 同一个注册 IP 指纹出现多个 Economy 账号时，该指纹下全部账号封禁且邀请奖励被阻断；普通游戏接口返回 423。管理员可在独立封禁页面解禁单个账号或整个事件，所有操作保留审计，解禁不自动补发被拦截的邀请宝石。
- 邮箱验证码只保存 HMAC，10 分钟过期、60 秒禁止重发、错误 5 次作废、不可重复使用，且发送与提交 IP 指纹必须一致。
- 邮箱验证码终态记录保留 30 天，注册请求最多每小时触发一次过期标记和历史清理；永久 Economy 注册记录不受影响。
- 发送验证码前必须先通过主页账号服务仅限回环的邮箱存在性接口查重；已注册邮箱返回 409 并引导登录，不创建验证码记录，也不调用 Resend。
- 生产验证码使用服务器上的 `RESEND_API_KEY` 和大写 `EMAIL_FROM`。systemd 先加载共享 `/etc/riversoft-email.env`，再加载 `/etc/riversoft-economy-api.env`；邮件密钥不存入 GitHub Actions。独立配置工作流只验证运行进程已加载两个变量；缺少配置时客户端明确显示“邮箱验证码服务未配置，请联系管理员”，不得显示为整个游戏服务器不可用。
- 管理员可以在单个事务中一次生成最多 50,000 个礼品码；礼品码业务表只保存 SHA-256 哈希，明文仅用于本次响应、TXT 下载和现有 24 小时幂等重试缓存。
- 管理员礼品码与兑换记录按游标分页，默认每页 100 条、最多 200 条；首屏不得全表传输，页面通过“加载更多”继续读取。
- 藏品是服务器记录归属的唯一资产实例，图片由芝加哥艺术博物馆 IIIF 外链渲染，不复制到游戏服务器。
- 管理员只能导入明确标记为公版且具有 `imageId` 的芝加哥艺术博物馆藏品；图片 URL 由服务器固定生成，不接受任意外链。
- 藏品、商品和工厂可单独或混合组成最多 20 项的不可拆分资产包公开竞价；卖方资产与最高出价资金同时冻结，但冻结资产仍归原玩家所有并计入总资产。商品最高出价按包内全部商品数量预占仓库，工厂冻结数量退出生产；服务器原子结算整包资产，拍卖成交价不写入订单簿行情或估值。
- 藏品目前不计入商品／工厂估值和总资产排行榜，避免没有公开买盘时产生任意估值。

## 页面

```text
概览｜市场｜生产｜资产｜藏品｜拍卖｜排行｜商店｜设置
```

- 概览：工作、行情、生产与财富摘要。
- 市场：商品和工厂连续双行资产目录、明确禁用原因、自然高度 5+5 单列价格档位订单簿、宽行情主列、逐单撤单与本地成交。
- 生产：共享仓库、建设卡、工厂集群选择器、当前集群详情，以及移动端底部悬浮详情框。
- 资产：现金、商品和工厂资产汇总，以及当前浏览器中的资产变化记录。
- 藏品：当前账号持有的艺术藏品、图片、来源与归属。
- 拍卖：使用资产包编辑器组合藏品、商品和工厂，整包竞价、取消无出价拍卖并查看自动结算结果。
- 排行：服务器总资产排行榜。
- 商店：按服务器固定汇率将宝石单向兑换为普通货币，并查看最近兑换记录。
- 设置：玩家资料、偏好、宝石、分享链接、邀请码、礼品兑换和退出。客户端偏好只显示已实现的“紧凑数字”和 3s／5s／10s 状态刷新频率。

- 前端依赖使用精确版本与 `package-lock.json`，CI、Web Build 和 Deploy 固定 Node 24.4.0 并使用 `npm ci`；CI 额外运行 Chromium 浏览器测试，覆盖存储拒绝与设置页运行时行为。

## 固定压力测试账号

压力测试统一复用 `tests/stress/accounts.json` 中的 24 个普通玩家槽位，避免每轮测试注册新邮箱。账号在主页账号服务中一次性预置后保持不变；Node 测试脚本通过 `tests/stress/loadAccounts.mjs` 读取，并用 `offset` / `limit` 分配并发槽位。

仓库只记录测试邮箱和逻辑槽位，不保存密码、Cookie 或 Token。运行前在测试环境注入 `ECONOMY_STRESS_TEST_PASSWORD`；压力脚本不得调用注册接口，也不得把运行时凭据写入日志或 artifact。
