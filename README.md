# Economy

Economy 是一款网页端多人在线经济模拟、产业经营、统一资产交易与藏品／商品／工厂单项或捆绑竞价拍卖游戏。

- 项目：`RIVERS0FT/Economy`
- 生产网页：`https://game.riversoft.top/economy/`
- 管理员页面：`https://game.riversoft.top/economy/admin`
- 封禁管理：合入 `https://game.riversoft.top/economy/admin` 的“账号封禁”分区
- 客户端状态版本：`15`
- 世界状态版本：`13`
- 市场需求模型版本：`2`

## 当前核心循环

```text
工作获得基础资金
→ 建设同类型工厂数量
→ 选择工厂生产配方并开启持续生产
→ 产成品直接进入共享仓库
→ 在统一资产订单簿交易商品和工厂
→ 市场需求形成最终品订单并沿生产链派生上游流动性
→ 通过不可拆分资产包拍卖交易藏品、商品和工厂
→ 邀请新玩家获得独立宝石
→ 在商店按固定汇率兑换普通货币
→ 调整产业链、库存、资金与收藏
→ 提升服务器计算的总资产和排行榜名次
```

## 当前关键规则

- 当前目录共 31 种商品和 21 种工厂类型；水果、饮料、鱼类、预制餐、纸品和家电均已进入完整生产与需求链。
- 产业链包括 `小麦 → 面粉 → 食品／预制餐`、`甘蔗 → 砂糖 + 水果／奶 → 饮料`、`木材 → 木板／纸浆 → 家具／纸品`、`棉花／毛 → 纺织品 → 服装`、`铁矿石 → 钢材 → 机械`、`原油 → 塑料 + 铜矿石 → 铜材 → 电子产品`，并由 `机械 + 电子产品 → 家电` 汇合为最终耐用品。
- 商品订单只允许玩家订单和市场需求订单；市场需求固定分为 `food` 食品市场和 `household` 家庭消费市场，不生成系统卖单、企业采购或固定价格流动性。
- 食品市场基础预算为 1,000／5 分钟，家庭消费市场基础预算为 900／5 分钟；预算按近 7 天经济活跃玩家平方根增长，最多 6 倍，并结合真实成交活跃度和满足率缺口调整。
- 玩家库存数量和库存价值不得扩大市场需求总预算，也不得直接提高某商品的需求份额。
- 每组周期预算中 85% 用于最终消费的直接需求，15% 用于沿正式配方反向推导的派生流动性；两者共享同一总预算，未使用预算不结转且当期只能发行一次。
- 新世界首次状态处理和模型 1 升级通过 `marketDemand.modelVersion = 2` 补齐新增目录并立即生成当前周期市场买单；世界版本继续保持 13。
- 直接需求采用需求类别和替代关系；主食、肉蛋奶鱼、水果奶饮料、食品预制餐以及电子产品和家电按相对有效价格、可成交深度和替代弹性平滑调整份额，替代品总预算不增加。
- 同一输出商品存在多条正式配方时形成生产路径替代；同一正式配方包含多个输入时形成互补关系，并通过互补门控优先补足瓶颈输入。
- 生产链采用双向价格传导：上游成本通过成本推动滞后影响下游，下游成交价值和最终需求通过净回值滞后影响上游；每条生产边至少延迟一个需求周期，并使用最近三个周期 60%／30%／10% 信号、阻尼、涨跌幅限制和基础价回归防止循环放大。
- 商品价格角色由直接需求目录和正式配方图自动判断；水果、鱼类、奶、面粉和电子产品可以同时承担直接消费和下游输入，机械通过家电链自动成为派生中间品，不维护静态角色 ID 名单。
- 商品初始参考价、生产数量、周期秒数和周期成本全部保持整数；当前平衡目标为基础原料 1／分钟、中间产品 3／分钟、最终产品 6／分钟。
- 农场的小麦、水稻、棉花和甘蔗配方保持同参数：周期 120 秒、单座产量 4、单座周期成本 6；水果由独立果园生产，鱼类由独立渔场生产。
- `steelworks` ID 永久保留，正式名称改为冶炼厂，同时支持铁矿石炼钢和铜矿石炼铜；不新增铜冶炼厂。
- 畜牧场可切换生产肉、蛋、奶和毛；磨坊可生产面粉或砂糖；饮料厂可选择乳制或果汁路线；电子工厂和家电厂分别执行塑料+铜材、机械+电子产品的多输入配方。
- 正式配方统一使用 `inputs[]`；同一周期所有输入必须同时满足并原子扣除，任一输入不足时不得扣除其他原料。
- 工厂按 `facilityTypeId + count` 保存，不存在单座工厂实例、实例 ID、内部仓库或领取产成品。
- 同类型工厂共用运行意图、三种顶层状态、进度、当前配方和周期结算；开启后只能持续生产。
- 工厂异常时保留开启意图；资金、原料或仓库条件恢复后，从新的完整周期自动恢复。
- 所有工厂集群统一使用服务器正式配方；单配方工厂显示唯一配方，多配方工厂可切换，运行中切换在下一完整周期生效且不重置当前进度。
- 工厂卡的生产公式只展示集群参数：运行中按 `participatingCount` 显示本周期输入、输出和成本，停止或异常按 `nextCycleCount` 显示启动后或恢复后的集群参数；周期时长不乘工厂数量，实际结算口径与公式一致。
- 建设卡只显示建造费用和施工时间，并同时显示工厂类型与配方名称；不显示生产周期、单座周期产量或单座周期成本。
- 商品和工厂共用统一限价订单结构，支持价格优先、同价时间优先、部分成交、撤单和禁止自成交。
- 订单簿按同资产、同方向、同价格匿名聚合为价格档位，只累加有效订单的当前剩余数量；服务器订单、同价时间优先、逐单撤销、成交明细和单张卖单手续费保持独立。
- 玩家通过统一订单簿卖出商品或工厂时，按单张卖单累计成交总额收取 1% 手续费，向上取整且最低为 1；买方仍支付成交总额，卖方获得扣费后净额，手续费由系统回收。卖给市场需求同样收费，拍卖不收取该手续费。
- 工厂运行中可以提交卖单；冻结数量立即退出当前参与数量，并停止产生商品和运营成本。
- 商品和工厂估值均使用最近一次统一订单簿真实成交价；模拟历史、挂单、基础价、系统价值和拍卖价格不得参与估值，从未真实成交时估值为 0。
- 工作冷却固定为 10 秒，不随连续工作次数增加。
- 共享仓库允许无限扩容；当前等级 `L` 的容量增量为 `250 + 50 × (L - 1)`，扩容费用为 `150 + ceil((当前实际总容量 - 500) × 0.6)`，费用由服务器按实际容量计算。
- 共享仓库只显示可用或冻结数量大于零的商品，并复用统一商品 SVG 图标；商品内容区使用容器查询，在不同宽度下每行显示 2／3／4／5／6 项。
- 桌面生产管理区由左侧常驻建设卡和右侧工厂列表组成；大于 1380px 时右侧工厂列表固定四列，同一网格行中的卡片等高、不同网格行按内容自适应；961px–1380px 使用自然高度单列。工厂名称与开关位于第一行，状态位于第二行，运行数量摘要位于第三行，市场入口始终固定在卡片底部。
- 共享仓库、建设新工厂和工厂集群属于生产页同一一级平面；大于 720px 时四边内边距统一为 16px，不大于 720px 时统一为 12px，标题左上锚点保持一致。
- 状态栏、导航、商品和工厂资产使用统一内联 SVG；工厂厂房图标与机械商品齿轮图标保持独立。
- 服务器只保存权威经济状态；玩家活动和成交展示日志保存在当前浏览器。
- 游戏状态使用世界修订号轮询；首次进入或修订号变化时返回完整状态，世界修订号未变化时只返回轻量确认。服务器在进程内缓存已迁移世界与修订号，由单一全局调度器最多每秒推进一次生产、拍卖和排行榜；同修订号轮询不得进入 SQLite 事务、重新解析、遍历或重新序列化整份世界。
- 客户端默认每 5 秒轮询状态，可选 3／5／10 秒；空闲 `GET state` 不得写回未变化的世界 JSON。全局游戏模型不得使用每秒时钟驱动整页重渲染，倒计时只在实际需要的页面局部维护。
- 客户端只接受不低于当前值的状态修订号；权威动作期间必须取消旧轮询并暂停新轮询，工作按钮立即显示“处理中”。
- `/economy-api/game/` 的大型 JSON 响应必须使用 gzip 压缩，避免完整状态占满服务器出口带宽。
- 主页账号认证使用单进程短缓存：状态读取 10 秒、普通写操作最多 2 秒、管理员不使用缓存；同会话并发认证合并，LRU 上限 5,000 条，缓存键只保存 Cookie 的 SHA-256 摘要。
- Economy 注册完成时点是统一账号第一次创建 Economy 玩家档案；邮箱验证码和已登录主页账号首次进入共用相同建档、邀请归因与注册 IP 检测逻辑。
- 宝石是独立于普通货币的整数资产，不参与市场、生产、总资产或排行榜。每名玩家拥有永久 8 位邀请码和专属分享链接；新玩家通过分享链接首次建档，或注册后 24 小时内在设置页填写邀请码时，邀请人立即获得 10 宝石，被邀请人不获得宝石。宝石可在独立“商店”按服务器固定汇率 1 宝石兑换 10 普通货币，单次 1～100 宝石，只允许单向兑换且不可撤销。
- 一个账号只能绑定一次邀请关系，分享链接与手动邀请码互斥。玩家不提供经济状态重置能力，资金、库存、工厂、订单、统计、宝石、邀请码、邀请关系、宝石流水和封禁记录均持续保留。
- 同一个注册 IP 指纹出现多个 Economy 账号时，该指纹下全部账号封禁且邀请奖励被阻断；普通游戏接口返回 423。管理员可在独立封禁页面解禁单个账号或整个事件，所有操作保留审计，解禁不自动补发被拦截的邀请宝石。
- 邮箱验证码只保存 HMAC，10 分钟过期、60 秒禁止重发、错误 5 次作废、不可重复使用，且发送与提交 IP 指纹必须一致。
- 邮箱验证码终态记录保留 30 天，注册请求最多每小时触发一次过期标记和历史清理；永久 Economy 注册记录不受影响。
- 发送验证码前必须先通过主页账号服务仅限回环的邮箱存在性接口查重；已注册邮箱返回 409 并引导登录，不创建验证码记录，也不调用 Resend。
- 生产验证码使用服务器上的 `RESEND_API_KEY` 和大写 `EMAIL_FROM`。systemd 先加载共享 `/etc/riversoft-email.env`，再加载 `/etc/riversoft-economy-api.env`；邮件密钥不存入 GitHub Actions。独立配置工作流只验证运行进程已加载两个变量；缺少配置时客户端明确显示“邮箱验证码服务未配置，请联系管理员”，不得显示为整个游戏服务器不可用。
- 管理员可以在单个事务中一次生成最多 50,000 个礼品码；礼品码业务表只保存 SHA-256 哈希，明文仅用于本次响应、TXT 下载和现有 24 小时幂等重试缓存。
- 管理员礼品码与兑换记录按游标分页，默认每页 100 条、最多 200 条；首屏不得全表传输，页面通过“加载更多”继续读取。
- 藏品是服务器记录归属的唯一资产实例，图片由芝加哥艺术博物馆 IIIF 外链渲染，不复制到游戏服务器。
- 管理员只能导入明确标记为公版且具有 `imageId` 的芝加哥艺术博物馆藏品；图片 URL 由服务器固定生成，不接受任意外链。
- 藏品、商品和工厂可单独或混合组成最多 20 项的不可拆分资产包公开竞价；卖方资产与最高出价资金同时冻结，但冻结资产仍归原玩家所有并计入总资产。商品最高出价按包内全部商品数量预占仓库，工厂冻结数量退出生产；服务器原子结算整包资产，拍卖成交价不写入订单簿行情或估值。
- 藏品目前不计入商品／工厂估值和总资产排行榜，避免没有公开买盘时产生任意估值。

## 页面

```text
概览｜市场｜生产｜资产｜藏品｜拍卖｜排行｜商店｜设置
```

- 概览：工作、行情、生产与财富摘要。
- 市场：商品和工厂连续双行资产目录、明确禁用原因、自然高度 5+5 单列价格档位订单簿、宽行情主列、逐单撤单与本地成交。
- 生产：共享仓库、左侧建设卡和工厂类型集群。
- 资产：现金、商品和工厂资产汇总，以及当前浏览器中的资产变化记录。
- 藏品：当前账号持有的艺术藏品、图片、来源与归属。
- 拍卖：使用资产包编辑器组合藏品、商品和工厂，整包竞价、取消无出价拍卖并查看自动结算结果。
- 排行：服务器总资产排行榜。
- 商店：按服务器固定汇率将宝石单向兑换为普通货币，并查看最近兑换记录。
- 设置：玩家资料、偏好、宝石、分享链接、邀请码、礼品兑换和退出。客户端偏好只显示已实现的“紧凑数字”和 3s／5s／10s 状态刷新频率。

- 前端依赖使用精确版本与 `package-lock.json`，CI、Web Build 和 Deploy 固定 Node 24.4.0 并使用 `npm ci`；CI 额外运行 Chromium 浏览器测试，覆盖存储拒绝与设置页运行时行为。

## 本地开发与完整检查

```bash
npm install
npm run dev
npm run build
```

`npm run build` 会执行文档权威性、邮箱验证码注册、宝石邀请与同 IP 封禁、页面职责、UI 架构、产业目录与整数经济平衡、统一资产市场、液态玻璃、仓库、工厂集群、市场需求、礼品码批量生成、藏品拍卖、Nginx、服务器测试、TypeScript 和 Vite 生产构建检查。

- 普通玩家成交回报只展示本人订单的成交结果；订单簿、状态接口和浏览器本地记录不得暴露成交来源、去向、市场需求组、对手身份或对手订单 ID。
