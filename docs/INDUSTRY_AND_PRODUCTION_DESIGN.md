## 1. 不可回退决策

- 工厂资产按 `facilityTypeId + count` 保存。
- 不存在单座工厂实例、工厂实例 ID、实例列表、设施槽位、内部商品、内部容量或领取操作。
- 每种 `facilityTypeId` 只有一个工厂集群。
- 同类型工厂共享运行意图、三种顶层状态、周期进度、当前配方和启停操作。
- 新建、买入或解冻的数量在运行中集群的下一周期加入，不重置当前周期，也不获得部分周期产出。
- 工厂卖单冻结数量立即不参与生产。
- 同一玩家同时只能施工一座工厂。
- 产成品完成后直接进入共享仓库。
- 所有正式目录和生产参数由服务器 `industry-catalog.js` 提供；客户端和需求模型不得维护另一套正式数值。
- 商品初始参考价、配方输入输出数量、生产周期秒数和周期成本必须保持整数。
- 同一工厂的不同正式配方可以拥有不同周期和周期成本；集群结算必须读取当前配方，而不是用工厂默认配方覆盖其他路线。
- 每种工厂必须在正式目录中声明 `complexity`，复杂度固定使用 `C1`～`C7`，并按复杂度约束建造费、施工时间和工厂系统参考值。
- 工厂复杂度同时约束建设门槛和参考分钟利润；正式梯度固定为 C1=1、C2=3、C3=6、C4=9、C5=12、C6=15、C7=18。复杂度不得改变市场需求预算。

正式多配方规则：

- 农场包含 `wheat-crop`、`rice-crop`、`cotton-crop`、`sugarcane-crop`；棉花和甘蔗不得拆成独立设施。
- 果园独立生产水果，不把水果继续塞入农场，避免粮食、工业纤维、糖料和果树共用一个过度膨胀的配方列表。
- 矿场包含 `mine-default` 与 `copper-ore-mining`；铜矿石不得拆成独立铜矿场。
- 畜牧场包含肉、蛋、奶、毛四个替代配方；渔场独立生产鱼类。
- `mill` 是永久设施 ID，正式显示名称为“磨坊”，同时包含面粉和砂糖两条配方。
- `steelworks` 是永久设施 ID，正式显示名称为“冶炼厂”；包含钢材和铜材两条配方，不新增铜冶炼厂。
- 纺织厂包含棉纺和毛纺两条替代配方；棉花和毛不是同一周期的同时输入。
- 食品厂包含食品和预制餐两条配方；预制餐必须同时消耗面粉和鱼类。
- 饮料厂包含乳制饮料和果汁饮料两条生产路径；乳制路线消耗砂糖和奶，果汁路线消耗水果和砂糖，两条路线初始单位成本相同。
- 电子工厂只有一个正式配方，必须同时消耗 1 塑料和 1 铜材；家电厂必须同时消耗 1 机械和 1 电子产品。

服务器 `PRODUCT_CATALOG` 和 `FACILITY_TYPE_CATALOG` 是正式来源。正式配方统一使用 `inputs[]`，无原料使用空数组，单原料使用一个元素，多原料必须在同一周期全部满足并原子扣除。任一输入不足时不得扣除其他输入。所有工厂类型都必须显式提供 `defaultRecipeId` 和至少一个正式 `recipes[]` 配方；客户端不得按工厂 ID 拼接名称或维护另一套参数。

### 2.1 建造参数

生产参数与建造参数继续分字段维护，避免调整利润时误改资产建设节奏；但两者都必须按同一个 `complexity` 等级接受验证。复杂度由产业链深度、原料组合、设备规模和技术门槛共同决定，不能只按工厂所属商品类别推断。

施工时间区间固定为：

- C1 为 30 秒～1 分钟；
- C2 为 5～10 分钟；
- C3 为 30 分钟～1 小时；
- C4～C7 为 1～2 小时。

C4～C7 共用同一小时级区间，但必须继续按产业复杂度递增，不得让更高复杂度工厂比更低复杂度工厂更早完成。当前高级工业阶梯固定为机械厂 100 分钟、电子工厂 110 分钟、家电厂 120 分钟。

工厂系统参考值用于新世界工厂市场初始参考、工厂订单价格范围和兼容字段，不代替最近一次真实订单簿成交价。其规则固定为：

```text
系统参考值 = ceil(建造费 × 130% ÷ 5) × 5
```

## 3. 商品与工厂目录扩展规则

新增商品和工厂时必须：

1. 保持 ID 唯一。
2. 为新增商品提供生产来源或明确系统来源。
3. 保证工厂输入、输出均引用正式商品目录。
4. 旧存档自动补齐新增商品库存、市场价格历史、价格传导状态和工厂集群空状态。
5. 旧存档自动补齐新增商品库存时不得重置资金、既有库存、订单、工厂数量、仓库等级或排行榜数据。
6. 客户端按服务器目录动态渲染，不得写死 6 个商品 ID、6 种工厂或固定商品名称。
7. 新数值必须保持整数，并通过目录验证脚本重新计算参考分钟利润。
8. 任一复杂度的目标利润改变时，必须整体检查所有同级工厂和完整产业链，不能只修补单个亏损商品。
9. 新工厂必须声明 C1～C7 复杂度，施工时间必须落在对应区间，且所有正式配方必须精确达到该复杂度的参考分钟利润；调整现有复杂度时必须同步审查利润、同级工厂及相邻等级的时间递增关系。
10. 调整建造费时必须同步按 130% 向上取整到 5 更新 `systemValue`，并更新本节表格与 `verify-industry-catalog.mjs` 的精确断言。

## 4. 工厂集群状态

```ts
interface FacilityGroup {
  facilityTypeId: string;
  count: number;
  participatingCount: number;
  pendingJoinCount: number;
  listedCount: number;
  auctionedCount: number;
  frozenCount: number;
  availableCount: number;
  nextCycleCount: number;
  enabled: boolean;
  status: 'running' | 'stopped' | 'error';
  statusReason?: FacilityStatusReason;
  cycleStartedAt?: number;
  lifetimeOutput: number;
  activeRecipeId: string;
  pendingRecipeId?: string;
}
```

三种顶层状态：

- `running`：开启且当前完整周期正在运行。
- `stopped`：玩家关闭运行意图。
- `error`：运行意图仍开启，但资金、原料、仓库或可用数量条件不足。

`enabled` 表示玩家运行意图，不等同于当前是否正在生产。异常条件恢复后由服务器自动恢复，不要求再次点击启动。

## 5. 统一周期

```text
实际周期产量 = 单座周期产量 × participatingCount
实际周期成本 = 单座周期成本 × participatingCount
每项实际周期原料 = 该项单座输入数量 × participatingCount
周期净仓库增长 = max(0, 实际周期产量 - Σ（每项实际周期原料）)
```

每个完整周期在一个 SQLite 事务中：

1. 检查运行意图、可用数量、资金、原料、仓库空间和当前配方。
2. 同时检查全部输入、资金与仓库条件；任一条件不满足时本周期不得产生部分扣款或部分扣料。
3. 扣除整个集群成本和全部输入原料。
4. 产成品直接增加到共享仓库。
5. 增加 `lifetimeOutput` 和玩家生产统计。
6. 推进周期时间。
7. 在周期边界合并 `pendingJoinCount`。
8. 应用 `pendingRecipeId`，然后按新配方进入下一周期。

不得让同一集群的一部分工厂单独生产。生产公式只展示集群参数：运行中使用 `participatingCount`，停止或异常使用 `nextCycleCount`；周期时间不乘数量，公式与服务器结算口径保持一致。

## 6. 下一周期加入

施工完成、工厂买单或工厂拍卖成交、订单簿卖单撤销或拍卖取消／流拍解冻发生在运行期间时：

```text
当前参与数量保持不变
新增数量进入 pendingJoinCount
当前周期按旧数量结算
下一周期合并新增数量
```

新增数量不得重置 `cycleStartedAt`，不得获得部分周期产出。运行中提交工厂卖单或工厂拍卖时，服务器先结算已经完成的完整周期，再从当前参与数量和待加入数量中冻结对应数量。订单簿卖单与拍卖冻结数量合并计算，并立即降低当前或下一周期产能；取消或流拍解冻后从下一完整周期加入。

## 7. 运行、停止与自动恢复

- 玩家打开开关后，`enabled = true`。
- 条件满足时状态为 `running`。
- 资金、原料、仓库或可用工厂不足时状态为 `error`。
- 条件恢复后自动恢复，从新的完整周期开始，不补算异常期间错过的周期。
- 玩家关闭开关后 `enabled = false` 且状态为 `stopped`，不得自动启动。
- 停止前只结算服务器时间上已经完成的完整周期；未完成周期不扣费、不耗料、不产出。

自动恢复必须由统一协调函数处理，不能由页面按钮或本地倒计时决定。

## 8. 持续生产与通用配方切换

- 工厂只有持续生产模式；开启后一直运行到玩家关闭或出现异常。
- 不存在定量生产、目标产量、生产计划、计划完成状态或计划自动保存。
- 每个工厂集群保存一个 `activeRecipeId`；多配方工厂可以选择其他正式配方，单配方工厂提交唯一配方时幂等成功。
- 停止或异常状态下切换配方立即写入 `activeRecipeId`，随后重新检查自动运行条件。
- 运行中切换配方只写入 `pendingRecipeId`；当前完整周期仍按旧配方结算，周期边界再提升为新的 `activeRecipeId`。
- 同一周期内连续切换只保留最后一次选择，不得重置 `cycleStartedAt`、重复产出或获得部分周期产量。
- 工厂作为资产交易时不携带卖方配方配置；买入数量加入买方既有工厂集群，并使用买方当前配方。
- 所有工厂详情统一显示“生产配方”选择器；多配方工厂可选择，单配方工厂显示唯一选项并禁用。

客户端开关只绑定 `group.enabled`，配方选择器只提交服务器正式配方 ID。旧 `/plan` 接口返回 `410 Gone`，不得静默接受旧生产计划。

## 9. 生产页面结构

页面顺序固定为：

```text
页面标题与运行／停止／异常／施工汇总
→ 共享仓库
→ 生产管理区：建设新工厂 + 工厂集群选择 + 当前工厂详情
```

共享仓库、建设新工厂、工厂集群选择和桌面当前工厂详情属于生产页同一一级平面，统一使用 `.production-surface` 与 `PagePanel`：大于 `720px` 时四边内边距统一为 `16px`，不大于 `720px` 时统一为 `12px`。业务 CSS 只管理轨道、列数、内部密度和 Overlay，不得重新声明一级卡片外层 `padding`。

### 9.1 建设卡

- 桌面独占生产管理区左列，宽度 `280px–320px`。
- 使用 `position: sticky`，只在生产管理区范围内常驻。
- 低于 960px 恢复普通文档流并位于工厂集群选择之前。
- 显示工厂类型、可选或固定配方名称、建造费用、施工时间、当前施工任务、建设按钮和单次施工规则。
- 建设卡不得显示生产周期、单座周期产量或单座周期成本；正式生产参数只在已拥有工厂的详情区域显示。
- 建设类型选择继续使用 `selectedFacilityTypeId`；当前详情工厂必须使用独立本地状态，不得因点击详情卡而改变建设下拉框。

### 9.2 工厂集群选择

- 客户端先按 `facilityTypeId` 建立已拥有集群映射，再严格遍历服务器 `game.facilityTypes` 生成选择卡；不得按名称、数量、运行状态、最近操作或客户端数组顺序重新排序。
- 默认详情工厂是正式目录顺序中的第一种已拥有工厂。首次进入移动端只选中默认工厂，不自动弹出详情悬浮框。
- 选择状态在五秒状态刷新期间保持；服务器快照只更新当前工厂的数据。所选工厂不再拥有时回退到正式目录中的第一种已拥有工厂；已经没有任何工厂时关闭悬浮详情并显示空状态。
- 选择卡只显示工厂名称、总数量、完整状态和“运行／待加入／冻结”三项摘要。整卡是单一真实 `button`，使用 `aria-pressed` 标记当前选择。
- 选择卡内部不得嵌套运行开关、配方选择器或市场按钮，避免嵌套交互和移动端误触。

### 9.3 桌面和平板主从布局

- 大于 `1380px` 时生产管理区使用三列：建设卡 `280px–320px`、工厂集群选择 `260px–300px`、当前工厂详情最小 `480px` 并占据剩余宽度。
- `961px–1380px` 时使用两列：建设卡位于左列，工厂选择和当前详情在右列上下排列。
- `721px–960px` 时恢复单列文档流，顺序为建设卡、工厂选择、当前详情；详情仍直接显示，不使用悬浮框。
- 桌面和平板只渲染一个当前工厂的完整详情，不再同时铺开所有完整工厂卡。

### 9.4 当前工厂详情

- 详情显示工厂名称和总数量、运行开关、完整状态、“运行中／下一周期加入／冻结中”三列摘要、生产配方、集群生产公式、生产进度和市场入口。
- 标题、开关、状态、摘要、配方和公式的语义与服务器状态保持不变；正式配方继续来自服务器目录。
- 集群生产公式支持无输入、单输入、多输入和单输出；客户端必须逐项渲染并显示 `+` 分隔。
- 输入位于进度条上方左侧；无输入统一显示“无”；每个输入项目下方显示对应当前可用库存。输出位于右侧，不显示输出库存。
- 输入与输出之间固定显示周期 SVG 图标、配方周期数值、成本 SVG 图标和当前集群周期成本；周期不乘以工厂规模，不得显示总工时。
- 生产公式只展示集群参数。运行中以 `participatingCount` 作为公式乘数并标记“本周期 × N”；停止或异常使用 `nextCycleCount`，分别标记“启动后 × N”或“恢复后 × N”。
- 当前周期的输入、输出和成本不得提前计入 `pendingJoinCount`；公式不得使用总持有 `count` 作为生产乘数。
- 生产进度条是公式容器最后一个可见元素；进度条下方不得显示当前周期、恢复运行、产出、成本或其他说明文字。
- 存在 `pendingRecipeId` 时只在配方标题区显示“下一周期切换为：配方名称”，当前公式继续展示本周期配方。
- 工厂集群开关点击区域与可见胶囊保持 `2.75rem × 1.6rem`。
- 市场入口固定为“前往市场交易该工厂 →”，并位于详情底部。

### 9.5 移动端详情悬浮框

- 不大于 `720px` 时页面内只显示当前选择栏和紧凑工厂选择网格；点击任意选择卡或“查看详情”后，从屏幕底部打开 Bottom Sheet，不在页面中继续渲染完整详情卡。
- 悬浮框使用 `role="dialog"`、`aria-modal="true"` 和标题关联；打开后焦点进入关闭按钮，`Tab` 焦点限制在悬浮框内，`Escape`、关闭按钮或点击遮罩可以关闭，关闭后焦点返回触发卡。
- 悬浮框打开期间锁定背景页面滚动；视口变为大于 `720px` 时必须自动关闭并解除滚动锁。
- 悬浮框最大高度为 `min(88dvh, 760px)`，顶部圆角，底部包含 `safe-area-inset-bottom`。悬浮框是唯一纵向滚动容器，生产公式、配方区域和内部卡片不得再创建嵌套纵向滚动。
- 标题和开关区域在悬浮框内容滚动时保持顶部可见，市场入口保持底部可见。启停和配方操作完成后悬浮框保持打开并原地刷新。
- `prefers-reduced-motion: reduce` 时关闭底部滑入和遮罩淡入动画。

```text
player.facilities
factory instance id
internalGoods
internalCapacity
facility listings as the active market model
trades
ledger
assetEvents
productionMode

completedQuantity

```

旧单座工厂数据必须汇总为 `facilityGroups`；旧内部商品无损迁移到对应输出商品的共享仓库；旧固定价格工厂挂牌不得重新成为当前交易来源。世界版本 7 的 `grain` 可用库存、冻结库存、玩家订单和价格历史必须完整迁移为 `wheat`，不得平均拆分或丢失订单优先级；旧已完成定量计划保持手动停止，旧运行中定量计划改为持续生产。新农场默认 `activeRecipeId = wheat-crop`。

本次 31／21 目录扩展不提升世界版本或客户端状态版本，市场需求模型升级为 `marketDemand.modelVersion = 2`。旧存档自动补齐新增商品零库存、市场、价格传导状态和新增工厂空集群语义，不修改现有资金、订单、库存、工厂数量、`activeRecipeId` 或周期开始时间。模型 1 的未完成市场需求订单被取消并立即按模型 2 重建；玩家订单、冻结资产和既有成交价格不得重写。

建设复杂度、建造费、施工时间与系统参考值属于目录参数变更，不提升世界版本或客户端状态版本。已经开始施工的任务保留其持久化 `completesAt`，不得因目录更新时间被缩短、延长或重新扣费；新提交的施工任务使用更新后的正式参数。既有玩家工厂数量、订单、成交价和资产不得重写。

## 11. 防回退

不得恢复：

- 单座工厂实例或设施槽位；
- 工厂内部仓库和领取产成品；
- 固定价格工厂挂牌市场；
- 手动修复异常后必须再次点击启动；
- 恢复定量生产、目标产量、生产计划或计划完成状态；
- 运行中切换配方立即重置周期或中途改变当前周期产出；
- 把一个玩家的同类工厂集群拆分成多种并行配方；
- 新数量中途加入当前周期；
- 让共享仓库、建设卡、工厂选择和桌面详情在同一断点使用不同一级外层内边距；
- 恢复同时铺开所有完整工厂卡、四列完整卡网格或瀑布流；
- 按名称、数量、状态或最近操作重排工厂选择卡；
- 让当前详情选择复用建设下拉框状态；
- 移动端首次进入自动弹出详情、在页面内直接铺开完整详情，或把悬浮框改成居中小弹窗；
- 移动端悬浮框关闭后丢失焦点返回、背景滚动锁或 `Escape` 关闭能力；
- 在悬浮框内部创建多层纵向滚动容器；
- 选择卡内部放入开关、配方下拉框或市场按钮；
- 顶部“正常生产中”重复说明；
- 独立持有／参与／待加入／冻结数量卡；
- “下一周期按若干座结算”辅助句；
- 移动端把运行中／下一周期加入／冻结中改为纵向排列；
- 恢复周期、产量、成本和原料四格规格区；
- 在生产公式中恢复箭头、可见“周期”标签或可见“运行成本”标签；
- 丢失多输入、多输出、逐输入库存、`+` 分隔能力；
- 让公式恢复单座输入、输出或成本，使用 `group.count` 作为公式乘数，或把 `pendingJoinCount` 提前计入当前周期；
- 运行中不按 `participatingCount` 展示，或停止／异常时不按 `nextCycleCount` 展示；
- 在生产进度条下方恢复当前周期、恢复运行、产出、成本或其他可见说明文字；
- 隐藏单配方工厂的配方选择器，或允许单配方选择器提交其他配方；
- 建设卡恢复生产周期、单座周期产量或单座周期成本；
- 客户端维护正式配方、成本或周期；
- 让生产周期乘以工厂规模、显示总工时，或让卡片四边使用不同内边距；
- 恢复白色市场说明“在统一订单簿中买卖该工厂”或旧入口“前往市场 →”；
- 使用小数商品价格、产量、输入量、周期秒数或周期成本；
- 只调一个产业链节点而不复算完整链条的参考分钟利润；
- 让目录扩展自动提高食品或家庭消费市场基础预算；
- 用静态商品 ID 名单判断价格角色，而不从直接需求与正式配方图自动判断；
- 删除工厂 `complexity`、把施工时间移出对应复杂度区间，或让高等级工业建设时间倒退；
- 调整建造费却不按 130% 向上取整到 5 同步更新 `systemValue`；
- 因目录参数更新而重算已经开始施工任务的 `completesAt` 或再次扣除建设费用。

### 拍卖冻结与工厂总量

工厂订单与拍卖冻结只减少可用于当前和下一周期生产的数量，不提前减少 `group.count`。同一资产包中的多种工厂分别按类型冻结并全部退出生产；发布、取消或流拍前后总持有数量不变，因此冻结工厂继续按最近一次订单簿真实成交价计入总资产。成交完成后才从卖方总量扣除并加入买方集群；整包任一项目异常时不得部分转移工厂。
