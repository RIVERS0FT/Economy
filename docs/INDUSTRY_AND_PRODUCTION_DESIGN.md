# Economy 多商品产业与生产计划设计

> 状态：当前生产、商品市场和工厂资产设计基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-11  
> 客户端状态版本：`version: 7`  
> 世界状态版本：`version: 3`

本文件取代旧版“一种基础商品、一种基础设施、设施槽位限制”的生产设计。涉及商品目录、工厂目录、生产配方、工厂启停、生产计划、库存、订单簿、设施交易和旧数据迁移时，以本文件为准。

服务器权威、事务、幂等、SQLite、systemd 和 Nginx 规则以 `docs/SERVER_ARCHITECTURE_AND_DEPLOYMENT_DESIGN.md` 为准；视觉规则以 `docs/UI_DESIGN_SYSTEM.md` 为准；订单页面归属以 `docs/MARKET_AND_ASSET_INFORMATION_ARCHITECTURE.md` 为准；日志保存位置以 `docs/LOCAL_ACTIVITY_LOG_DESIGN.md` 为准。

## 1. 不可回退的核心决策

- 玩家持有工厂数量不设上限。
- 不存在设施槽位、空闲槽位、槽位升级或购买工厂时的槽位检查。
- 同一玩家同时只能施工一座工厂，这是施工队节奏限制，不是资产持有上限。
- 所有工厂施工完成后保持停止状态，必须由玩家手动启动。
- 玩家可以随时手动停止运行中的工厂。
- 阻塞工厂在条件恢复后不得自动重启，必须再次手动启动。
- 每座工厂支持持续生产和定量生产两种计划。
- 定量计划达到目标后自动停止，停止原因记录为 `plan_complete`。
- 每种商品拥有独立库存、冻结库存、订单簿、最近成交价和价格历史。
- 不同商品订单不得互相撮合。
- 配方、费用、周期和产量由服务器目录决定，客户端不能修改正式结果。
- 旧玩家资产必须原地迁移，不得清零或自动兑换成不等值资产。
- 订单管理和撤单必须位于市场页，资金与资产变化必须位于资金页。
- 服务器不持久化玩家成交、流水或资产事件；用户日志仅保存在浏览器本地。

## 4. 工厂状态

正式状态：

```text
constructing
ready
running
paused
full
insufficient_funds
insufficient_input
listed
```

停止原因：

```text
manual
plan_complete
insufficient_funds
insufficient_input
output_full
listed
maintenance
```

规则：

- `ready`：施工完成，等待首次手动启动。
- `running`：服务器按周期生产。
- `paused`：玩家手动停止或定量计划完成。
- `full`：内部产成品无法再容纳一个完整周期产出。
- `insufficient_funds`：无法支付下一个完整周期运营费。
- `insufficient_input`：无法提供下一个完整周期原料。
- `listed`：正在挂牌，不能运行或修改计划。
- 领取产成品、补充资金或补充原料只解除客观条件，不自动启动工厂。

## 5. 手动启动与停止

启动前服务器检查：

- 不是施工中或挂牌中；
- 内部产成品仓库能容纳一个完整周期；
- 资金足够支付一个完整周期；
- 加工厂拥有一个完整周期所需原料；
- 定量计划尚未完成。

停止操作规则：

1. 操作进入服务器事务；
2. 先按服务器时间结算已经完成的完整周期；
3. 当前未完成周期作废；
4. 未完成周期不扣费、不消耗原料、不增加产出；
5. 工厂进入 `paused`，停止原因为 `manual`；
6. 客户端收到新权威状态后，在本地生成停止和资产变化记录。

## 6. 生产计划

每座工厂具有：

```ts
productionMode: 'continuous' | 'target';
targetQuantity?: number;
completedQuantity: number;
```

### 6.1 持续生产

- 不设置目标上限；
- 持续运行到玩家停止或出现阻塞；
- 修改为持续生产时清除旧目标和旧完成量。

### 6.2 定量生产

- `targetQuantity` 表示计划产出的最终商品数量；
- 目标必须是周期产量的整数倍；
- 保存新计划时 `completedQuantity` 归零；
- 达到目标后立即停止，不允许超产；
- 已完成计划不会自动重置或自动创建下一份计划；
- 修改计划前必须先停止工厂。

## 7. 批量离线结算

服务器不得逐秒模拟工厂。每次处理工厂时计算：

```text
可完成周期数 = min(
  已经过的完整周期,
  内部产成品容量允许周期,
  资金允许周期,
  原料允许周期,
  计划剩余允许周期,
  单次安全上限
)
```

然后在一次事务内批量：

- 扣除运营费；
- 扣除原料；
- 增加内部产成品；
- 增加累计产量；
- 增加计划完成量；
- 推进周期起始时间。

玩家离线前处于 `running` 的工厂继续结算；其他状态不产生离线产出。客户端下一次读取权威状态时，可以在当前浏览器本地生成聚合变化记录，但服务器不保存该日志。

## 8. 多商品库存

玩家库存结构：

```ts
inventories: Record<productId, {
  available: number;
  frozen: number;
}>;
```

共享仓库容量规则：

```text
所有商品可用数量
+ 所有商品冻结数量
+ 未完成买单预占数量
<= inventoryCapacity
```

加工工厂只消耗 `available` 原料。被冻结用于卖单的商品不能用于生产。

## 9. 多商品市场

订单必须包含：

```ts
productId: string;
```

撮合候选必须同时满足：

- 商品 ID 相同；
- 买卖方向相反；
- 价格交叉；
- 订单仍未完成；
- 不是同一玩家自成交。

每种商品独立维护最近成交价、价格历史、买一卖一、外部流动性订单、人口或企业需求和需求满足率。

订单页面规则：

- 市场页显示当前商品未完成订单并提供快捷撤单；
- 市场页显示全部商品未完成订单和冻结资产；
- 市场页显示当前浏览器本地成交记录；
- 不提供独立订单导航或 `RecordsPage`；
- 未完成订单数量显示在市场导航。

## 10. 工厂交易

- 工厂数量固定为 1，不允许部分成交；
- 挂牌前必须停止生产并领取全部内部产成品；
- 挂牌价保持在系统估值的 50%～200%；
- 买方只检查资金，不检查设施槽位；
- 玩家挂牌成交时转移原始唯一工厂 ID；
- 收购后工厂进入 `paused`，必须由买家手动启动；
- 收购后生产计划重置为持续生产；
- 客户端通过前后状态差异在本地记录工厂买卖，不在服务器保存交易历史。

## 11. 旧状态迁移

旧版世界状态在 SQLite 读取层调用 `migrateWorld`：

| 旧数据 | 新数据 |
|---|---|
| 基础商品 | `grain` 粮食 |
| `inventory` | `inventories.grain.available` |
| `frozenInventory` | `inventories.grain.frozen` |
| 单一市场价格和历史 | `markets.grain` |
| 旧商品订单 | 增加 `productId: grain` |
| 基础生产设施 | `facilityTypeId: farm` |
| 旧内部商品 | 农场内部粮食 |
| 旧运行设施 | 持续生产，保留运行状态和周期时间 |
| `facilitySlots` | 删除，不转化为其他限制 |
| 旧设施挂牌 | 农场挂牌 |
| `trades`、`ledger`、`assetEvents` | 从服务器世界状态删除，不迁移到新服务器日志 |

迁移必须可重复执行，不改变玩家资金、库存、工厂、订单、挂牌和市场状态。

## 12. 客户端状态版本 7

服务器返回：

- 六种商品目录；
- 六类工厂目录；
- 各商品库存和市场状态；
- 当前订单和挂牌；
- 当前工厂、生产计划、资金和统计。

不返回：

```text
trades
ledger
assetEvents
```

这些日志由 `src/utils/localActivityStore.ts` 在浏览器本地生成。

## 13. 服务器动作

稳定动作包括：

```text
buildFacility
startFacility
pauseFacility
setProductionPlan
collectFacility
listFacility
cancelFacilityListing
buyFacility
placeOrder
cancelOrder
```

所有写操作继续使用 SQLite `BEGIN IMMEDIATE` 和 `Idempotency-Key`。

## 14. 测试要求

必须覆盖：

1. 不同商品订单绝不互相撮合；
2. 工厂持有数量没有槽位限制；
3. 同一玩家同时只能施工一座工厂；
4. 加工厂正确消耗原料；
5. 定量计划不会超产；
6. 手动停止只结算完整周期；
7. 阻塞解除后不自动重启；
8. 工厂交易转移原始唯一资产；
9. 旧状态无损迁移；
10. API 状态不包含玩家日志；
11. SQLite 世界 JSON 不包含玩家日志；
12. 幂等请求不重复改变资产。

## 15. 不可回退规则

不得：

- 恢复设施槽位；
- 恢复单商品、单工厂模型；
- 自动启动施工完成或阻塞恢复的工厂；
- 删除手动停止；
- 删除定量计划；
- 允许不同商品互相撮合；
- 把玩家日志重新写入服务器；
- 把本地日志用于资产或排名计算；
- 绕过事务、幂等或服务器验证；
- 在未更新设计、迁移和测试的情况下改变生产或日志边界。

未更新设计、迁移和测试的生产或资产事件回退不应合并。
