# Economy 多商品产业与生产计划设计

> 状态：当前生产、商品市场和工厂资产设计基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-11  
> 客户端状态版本：`version: 7`  
> 世界状态版本：`version: 3`

本文件定义商品目录、工厂目录、生产配方、工厂启停、生产计划、共享仓库约束、工厂交易和旧数据迁移。页面归属以 `docs/PAGE_CONTENT_AND_NAVIGATION_DESIGN.md` 为准，仓库等级与扩容以 `docs/WAREHOUSE_EXPANSION_DESIGN.md` 为准。

## 1. 不可回退的核心决策

- 玩家持有工厂数量不设上限，不存在设施槽位。
- 同一玩家同时只能施工一座工厂。
- 工厂施工完成后保持停止状态，必须手动启动。
- 玩家可以手动停止运行中的工厂。
- 阻塞条件解除后不得自动重启。
- 每座工厂支持持续生产和定量生产。
- 定量计划达到目标后自动停止，不允许超产。
- 不同商品订单不得互相撮合。
- **所有工厂不设置内部产成品仓库或内部容量。**
- **每个完整生产周期完成后，产成品直接进入玩家共享仓库。**
- 共享仓库无法容纳下一个完整周期的净新增商品时，工厂停止，状态为 `full`，停止原因为 `output_full`。
- 不提供产成品领取操作、领取按钮或 `/collect` API。
- 服务器不持久化玩家活动日志。

## 4. 工厂状态

```text
constructing
ready
running
paused
full
insufficient_funds
insufficient_input
listed
```

停止原因：

```text
manual
plan_complete
insufficient_funds
insufficient_input
output_full
listed
maintenance
```

- `full`：共享仓库无法容纳下一个完整生产周期的净新增商品。
- `insufficient_funds`：无法支付下一个周期运营费。
- `insufficient_input`：无法提供下一个周期原料。
- `listed`：工厂正在挂牌，不能生产或修改计划。
- 仓库扩容、补充资金或补充原料只解除客观条件，不自动启动工厂。

## 5. 直接入仓结算

每个完整周期必须在同一个服务器事务中原子完成：

1. 检查共享仓库空间；
2. 检查运营资金；
3. 检查原料；
4. 扣除运营费；
5. 扣除原料；
6. 将产成品直接增加到 `inventories[outputProductId].available`；
7. 增加累计产量与计划完成量；
8. 推进周期时间。

工厂对象和工厂挂牌快照不得保存：

```text
internalGoods
internalCapacity
```

## 6. 共享仓库净增长规则

一个周期对仓库占用的净增长：

```text
netStoragePerCycle = max(0, outputPerCycle - inputPerCycle)
```

- 无原料工厂必须为完整周期产出预留空间。
- 加工工厂可以利用同一周期消耗原料释放的空间。
- 例如面粉厂在满仓时消耗 2 粮食并产出 1 面粉，仓库净减少 1，因此允许生产。
- 多座工厂按服务器处理顺序共享同一剩余容量，不能各自重复使用同一空间。
- 未完成买单预占继续计入共享仓库占用。

## 7. 手动启动与停止

启动前服务器检查：

- 工厂不是施工中或挂牌中；
- 共享仓库能容纳下一个完整周期的净新增商品；
- 资金足够支付一个周期；
- 加工厂拥有一个周期原料；
- 定量计划尚未完成。

手动停止时先结算服务器时间上已经完成的完整周期；当前未完成周期作废，不扣费、不消耗原料、不增加产出。

## 8. 生产计划

```ts
productionMode: 'continuous' | 'target';
targetQuantity?: number;
completedQuantity: number;
```

- 持续生产运行到手动停止或出现阻塞。
- 定量目标必须是周期产量的整数倍。
- 达到目标后立即停止，停止原因为 `plan_complete`。
- 修改计划前必须先停止工厂。

## 9. 批量离线结算

服务器不得逐秒模拟。可完成周期数：

```text
min(
  已经过的完整周期,
  共享仓库净增长允许周期,
  资金允许周期,
  原料允许周期,
  计划剩余允许周期,
  单次安全上限
)
```

离线前处于 `running` 的工厂继续按服务器时间结算。产出直接写入共享仓库，不产生待领取商品。

## 10. 工厂交易

- 工厂挂牌前必须停止生产。
- 不需要清空内部产成品，因为内部产成品不存在。
- 挂牌价必须在系统估值的 50%～200%。
- 买方只检查资金，不检查设施槽位。
- 收购后工厂进入 `paused`，生产计划重置为持续生产。
- 工厂挂牌快照不得包含内部容量或内部商品。

## 11. 旧状态迁移

旧数据迁移规则：

| 旧数据 | 新数据 |
|---|---|
| `facility.internalGoods` | 一次性无损增加到该工厂输出商品的共享仓库可用库存 |
| `facility.internalCapacity` | 删除 |
| 挂牌快照中的内部容量 | 删除 |
| `full`（旧内部仓库满） | 迁移为 `paused`、`manual` |
| 旧运行工厂 | 保留运行状态和周期时间，之后使用直接入仓结算 |
| `facilitySlots` | 删除 |
| `trades`、`ledger`、`assetEvents` | 从服务器世界状态删除 |

迁移允许临时造成仓库占用超过容量，以保证旧资产不丢失；超限期间不能增加净库存，玩家可出售商品或扩容。

迁移必须幂等：旧内部商品只转移一次，保存后的世界状态不得再包含内部存储字段。

## 12. 客户端与本地日志

客户端状态不得包含工厂内部存储字段，不得提供领取操作。

浏览器本地生产事件通过以下权威状态差异生成：

- `completedQuantity` 增量；
- 输出商品库存增量；
- 原料库存减少；
- 运营费扣除。

旧浏览器已经保存的“领取”历史可以继续只读显示，但不得触发任何服务器操作。

## 13. 测试要求

必须覆盖：

1. 原料型工厂产出直接进入共享仓库；
2. 仓库不足一个完整周期时不扣费、不生产并停止；
3. 加工厂可以利用原料消耗释放的空间；
4. 多工厂共享同一剩余容量；
5. 定量计划不超产；
6. 手动停止只结算完整周期；
7. 旧内部商品无损且仅迁移一次；
8. API、SQLite 和挂牌快照不包含内部存储字段；
9. 不存在领取路由、客户端动作或按钮；
10. 同一服务器时间重复处理不会重复生产。

## 14. 不可回退规则

不得：

- 恢复工厂内部商品或内部容量；
- 恢复产成品领取操作；
- 让产成品先进入工厂再由玩家领取；
- 绕过共享仓库空间检查；
- 只检查产出而忽略同周期原料消耗释放的空间；
- 让多个工厂重复使用同一剩余容量；
- 丢失旧内部商品；
- 让客户端决定正式生产结果；
- 绕过事务、幂等或服务器验证。

未更新设计、迁移、测试和防回退检查的生产规则修改不应合并。
