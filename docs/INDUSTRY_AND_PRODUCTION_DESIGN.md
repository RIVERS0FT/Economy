# Economy 产业目录与工厂集群设计

> 状态：当前商品目录、工厂数量资产与生产设计基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-12  
> 客户端状态版本：11  
> 世界状态版本：7

市场交易以 `UNIFIED_ASSET_ORDER_BOOK_DESIGN.md` 为准；共享仓库以 `WAREHOUSE_EXPANSION_DESIGN.md` 为准；页面归属以 `PAGE_CONTENT_AND_NAVIGATION_DESIGN.md` 为准。

## 1. 不可回退决策

- 工厂资产按 `facilityTypeId + count` 保存。
- 不存在单座工厂实例、工厂实例 ID、实例列表、设施槽位、内部商品、内部容量或领取操作。
- 每种 `facilityTypeId` 只有一个工厂集群。
- 同类型工厂共享运行意图、三种顶层状态、周期进度、生产计划和启停操作。
- 新建、买入或解冻的数量在运行中集群的下一周期加入，不重置当前周期，也不获得部分周期产出。
- 工厂卖单冻结数量立即不参与生产。
- 同一玩家同时只能施工一座工厂。
- 产成品完成后直接进入共享仓库。
- 所有正式目录和生产参数由服务器提供，客户端不得维护另一套正式数值。

## 4. 商品与工厂目录扩展规则

新增商品和工厂时必须：

1. 保持 ID 唯一。
2. 为新增商品提供生产来源或明确系统来源。
3. 保证工厂输入、输出均引用正式商品目录。
4. 旧存档自动补齐新增商品库存、市场价格历史、基础流动性和工厂集群空状态。
5. 旧存档自动补齐新增商品库存时不得重置资金、既有库存、订单、工厂数量、仓库等级或排行榜数据。
6. 客户端按服务器目录动态渲染，不得写死 6 个商品 ID、6 种工厂或固定商品名称。

## 5. 工厂集群状态

```ts
interface FacilityGroup {
  facilityTypeId: string;
  count: number;
  participatingCount: number;
  pendingJoinCount: number;
  listedCount: number;
  availableCount: number;
  nextCycleCount: number;
  enabled: boolean;
  status: 'running' | 'stopped' | 'error';
  statusReason?: FacilityStatusReason;
  cycleStartedAt?: number;
  productionMode: 'continuous' | 'target';
  targetQuantity?: number;
  completedQuantity: number;
  pendingProductionPlan?: PendingProductionPlan;
}
```

三种顶层状态：

- `running`：开启且当前完整周期正在运行。
- `stopped`：玩家关闭运行意图，或定量计划完成。
- `error`：运行意图仍开启，但资金、原料、仓库、计划或可用数量条件不足。

`enabled` 表示玩家运行意图，不等同于当前是否正在生产。

## 6. 统一周期

```text
周期产量 = 单座周期产量 × participatingCount
周期成本 = 单座周期成本 × participatingCount
周期原料 = 单座周期原料 × participatingCount
周期净仓库增长 = max(0, 周期产量 - 周期原料)
```

每个完整周期在一个 SQLite 事务中：

1. 检查运行意图、可用数量、资金、原料、仓库空间和计划。
2. 扣除整个集群成本。
3. 扣除整个集群原料。
4. 产成品直接增加到共享仓库。
5. 增加 `completedQuantity` 和玩家生产统计。
6. 推进周期时间。
7. 在周期边界合并 `pendingJoinCount`。
8. 判断当前定量计划是否完成。
9. 未完成时应用 `pendingProductionPlan` 并进入下一周期。

不得让同一集群的一部分工厂单独生产。

## 7. 下一周期加入

施工完成、工厂买单成交或卖单撤销解冻发生在运行期间时：

```text
当前参与数量保持不变
新增数量进入 pendingJoinCount
当前周期按旧数量结算
下一周期合并新增数量
```

新增数量不得重置 `cycleStartedAt`，不得获得部分周期产出。

运行中提交工厂卖单时，服务器先结算已经完成的完整周期，再从当前参与数量和待加入数量中冻结对应数量。冻结数量立即降低当前或下一周期产能。

## 8. 运行、停止与自动恢复

- 玩家打开开关后，`enabled = true`。
- 条件满足时状态为 `running`。
- 资金、原料、仓库、计划或可用工厂不足时状态为 `error`。
- 条件恢复后自动恢复，从新的完整周期开始，不补算异常期间错过的周期。
- 玩家关闭开关后 `enabled = false` 且状态为 `stopped`，不得自动启动。
- 停止前只结算服务器时间上已经完成的完整周期；未完成周期不扣费、不耗料、不产出。

自动恢复必须由统一协调函数处理，不能由页面按钮或本地倒计时决定。

## 9. 生产计划

- 持续运行：一直运行到玩家关闭或出现异常。
- 定量生产：达到目标后必须关闭开关，状态为 `stopped`，原因为 `plan_complete`。
- 运行中保存的新计划进入 `pendingProductionPlan`，在下一周期边界生效。
- 当前定量计划完成时，即使存在待生效计划，也必须先把待生效计划提升为新的当前计划，然后关闭开关；不得自动继续生产。
- 玩家再次开启开关后，使用已经提升的新当前计划运行。
- 数量变化导致剩余目标与下一周期产量不兼容时进入 `error`，原因为 `plan_adjustment_required`。
- 调整计划后可自动恢复，不允许超产或只运行部分工厂。

定量生产完成后关闭开关是服务器权威规则，客户端开关仅绑定 `group.enabled`。

## 10. 生产页面布局

页面顺序固定为：

```text
页面标题与运行／停止／异常／施工汇总
→ 共享仓库
→ 生产管理区：左侧建设新工厂，右侧工厂集群列表
```

建设卡：

- 桌面独占生产管理区左列，宽度 `280px–320px`。
- 使用 `position: sticky`，只在生产管理区范围内常驻。
- 低于 960px 恢复普通文档流并位于工厂列表之前。

工厂卡：

- 大于 960px 时固定高度 384px；移动端使用自然高度。
- 宽屏右侧双列，中等桌面单列，顶部对齐。
- 顶部是唯一状态说明区，异常原因直接写入状态标签。
- 运行状态不显示额外说明；剩余周期只在进度区出现。
- 标题显示总数量，顶部摘要只显示“运行中／下一周期加入／冻结中”。
- 不显示独立“持有”项、数量卡片或“下一周期按若干座结算”辅助句。
- 规格区保留周期、产量、成本、原料需求和当前库存。
- 计划区只保留当前计划、模式下拉框、定量输入和统一“保存计划”按钮。
- 不显示重复的“持续／定量”状态小卡片。
- 开关始终位于卡片右上角。
- 底部只提供前往统一订单簿的市场入口。

## 11. 迁移与版本

世界版本 7 不保存：

```text
player.facilities
factory instance id
internalGoods
internalCapacity
facility listings as the active market model
trades
ledger
assetEvents
```

旧单座工厂数据必须汇总为 `facilityGroups`；旧内部商品无损迁移到对应输出商品的共享仓库；旧固定价格工厂挂牌不得重新成为当前交易来源。

## 12. 防回退

不得恢复：

- 单座工厂实例或设施槽位；
- 工厂内部仓库和领取产成品；
- 固定价格工厂挂牌市场；
- 手动修复异常后必须再次点击启动；
- 运行中修改计划立即重置周期；
- 定量计划完成后自动继续待生效计划；
- 新数量中途加入当前周期；
- 工厂卡桌面自然高度或强制同行等高之外的任意高度分叉；
- 顶部“正常生产中”重复说明；
- 独立持有／参与／待加入／冻结数量卡；
- “下一周期按若干座结算”辅助句；
- 重复的持续／定量状态小卡片；
- 客户端维护正式配方、成本或周期。
