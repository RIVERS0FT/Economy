## 1. 不可回退决策

- 工厂资产按 `facilityTypeId + count` 保存。
- 不存在单座工厂实例、工厂实例 ID、实例列表、设施槽位、内部商品、内部容量或领取操作。
- 每种 `facilityTypeId` 只有一个工厂集群。
- 同类型工厂共享运行意图、三种顶层状态、周期进度、当前配方和启停操作。
- 新建、买入或解冻的数量在运行中集群的下一周期加入，不重置当前周期，也不获得部分周期产出。
- 工厂卖单冻结数量立即不参与生产。
- 同一玩家同时只能施工一座工厂。
- 产成品完成后直接进入共享仓库。
- 所有正式目录和生产参数由服务器 `industry-catalog.js` 提供；客户端和需求模型不得维护另一套正式数值。
- 商品初始参考价、配方输入输出数量、生产周期秒数和周期成本必须保持整数。
- 同一工厂的不同正式配方可以拥有不同周期和周期成本；集群结算必须读取当前配方，而不是用工厂默认配方覆盖其他路线。
- 每种工厂必须在正式目录中声明 `complexity`，复杂度固定使用 `C1`～`C7`，并按复杂度约束建造费、施工时间和工厂系统参考值。
- 工厂复杂度同时约束建设门槛和参考分钟利润；正式梯度固定为 C1=1、C2=3、C3=6、C4=6、C5=8、C6=10、C7=12。复杂度决定生产运营就业结构，但不得直接创造市场需求预算。
- 模型 8 的稳定需求补充只由人口钱包覆盖缺口触发，不读取工厂数量、产量、库存价值或复杂度；生产周期成本仍只形成就业收入。
- 玩家可见配方利润分析只读取客户端当前公开订单簿、可用库存和服务器正式配方，不得改变生产、市场、资产或统计状态。

### 2.1 周期成本与人口运营就业

每个完整成功周期的运营就业工资总额严格等于当前配方周期成本乘 `participatingCount`。资金、原料、仓库或运行条件任一不满足时，不扣周期成本、不产生工资、不产出商品。

| 复杂度 | 基础人口 | 技术人口 | 专业人口 |
|---|---:|---:|---:|
| C1 | 90% | 9% | 1% |
| C2 | 78% | 20% | 2% |
| C3 | 55% | 40% | 5% |
| C4 | 30% | 60% | 10% |
| C5 | 18% | 55% | 27% |
| C6 | 10% | 40% | 50% |
| C7 | 5% | 25% | 70% |

整数工资使用最大余数法分配，三类人口新增收入之和必须精确等于玩家实际支付的周期成本。

### 2.2 建造业就业

工厂复杂度只决定建造费总额、施工时间和建成后的运营岗位，不改变建造业就业比例。所有工厂建造费固定按基础人口 60%、技术人口 30%、专业人口 10% 分配。玩家开工时一次性支付并进入施工就业托管，服务器按 `floor(建造费 × 已施工时间 ÷ 总施工时间)` 逐步释放，完成时释放全部整数余数。模型 7 上线前已经支付的施工费不得追溯补发。

完整产业链：

```text
小麦 → 面粉 ─┬→ 食品
鱼类 ────────┴→ 预制餐
甘蔗 → 砂糖 + 奶 → 乳制饮料
甘蔗 → 砂糖 + 水果 → 果汁饮料
水果、鱼类、肉、蛋、奶、水稻、食品、饮料、预制餐 → 食品市场直接消费
棉花／毛 → 纺织品 → 服装
木材 ─┬→ 木板 → 家具
      └→ 纸浆 → 纸品
铁矿石 → 钢材 → 机械 ─────────┐
铜矿石 → 铜材 ─┐              ├→ 家电
原油 → 塑料 ────┴→ 电子产品 ──┘
```

直接需求商品由 `MARKET_DEMAND_GROUP_CATALOG.classes[]` 定义；基础原料和中间产品通过正式配方自动获得派生流动性。玩家可用库存、冻结库存、库存价值和挂牌价值不得扩大需求总预算，也不得直接提高任何商品份额；只有有效卖单深度可以影响可购性。价格沿正式配方双向、逐边、滞后传导，详细规则以 `PRODUCT_AND_GAMEPLAY_DESIGN.md` 为准。商品 ID 必须唯一，已有 ID 不得重命名或复用。

正式多配方规则：

- 农场包含 `wheat-crop`、`rice-crop`、`cotton-crop`、`sugarcane-crop`；棉花和甘蔗不得拆成独立设施。
- 果园独立生产水果，不把水果继续塞入农场，避免粮食、工业纤维、糖料和果树共用一个过度膨胀的配方列表。
- 矿场包含 `mine-default` 与 `copper-ore-mining`；铜矿石不得拆成独立铜矿场。
- 畜牧场包含肉、蛋、奶、毛四个替代配方；渔场独立生产鱼类。
- `mill` 是永久设施 ID，正式显示名称为“磨坊”，同时包含面粉和砂糖两条配方。
- `steelworks` 是永久设施 ID，正式显示名称为“冶炼厂”；包含钢材和铜材两条配方，不新增铜冶炼厂。
- 纺织厂包含棉纺和毛纺两条替代配方；棉花和毛不是同一周期的同时输入。
- 食品厂包含食品和预制餐两条配方；预制餐必须同时消耗面粉和鱼类。
- 饮料厂包含乳制饮料和果汁饮料两条生产路径；乳制路线消耗砂糖和奶，果汁路线消耗水果和砂糖，两条路线初始单位成本相同。
- 电子工厂只有一个正式配方，必须同时消耗 1 塑料和 1 铜材；家电厂必须同时消耗 1 机械和 1 电子产品。

服务器 `PRODUCT_CATALOG` 和 `FACILITY_TYPE_CATALOG` 是正式来源。正式配方统一使用 `inputs[]`，无原料使用空数组，单原料使用一个元素，多原料必须在同一周期全部满足并原子扣除。任一输入不足时不得扣除其他输入。所有工厂类型都必须显式提供 `defaultRecipeId` 和至少一个正式 `recipes[]` 配方；客户端不得按工厂 ID 拼接名称或维护另一套参数。

### 2.1 建造参数

生产参数与建造参数继续分字段维护，避免调整利润时误改资产建设节奏；但两者都必须按同一个 `complexity` 等级接受验证。复杂度由产业链深度、原料组合、设备规模和技术门槛共同决定，不能只按工厂所属商品类别推断。

施工时间区间固定为：

- C1 为 30 秒～1 分钟；
- C2 为 5～10 分钟；
- C3 为 30 分钟～1 小时；
- C4～C7 为 1～2 小时。

C4～C7 共用同一小时级区间，但必须继续按产业复杂度递增，不得让更高复杂度工厂比更低复杂度工厂更早完成。当前高级工业阶梯固定为机械厂 100 分钟、电子工厂 110 分钟、家电厂 120 分钟。

工厂系统参考值用于新世界工厂市场初始参考、工厂订单价格范围和兼容字段，不代替最近一次真实订单簿成交价。其规则固定为：

```text
系统参考值 = ceil(建造费 × 130% ÷ 5) × 5
```

## 3. 商品与工厂目录扩展规则

新增商品和工厂时必须：

1. 保持 ID 唯一。
2. 为新增商品提供生产来源或明确系统来源。
3. 保证工厂输入、输出均引用正式商品目录。
4. 旧存档自动补齐新增商品库存、市场价格历史、价格传导状态和工厂集群空状态。
5. 旧存档自动补齐新增商品库存时不得重置资金、既有库存、订单、工厂数量、仓库等级或排行榜数据。
6. 客户端按服务器目录动态渲染，不得写死 6 个商品 ID、6 种工厂或固定商品名称。
7. 新数值必须保持整数，并通过目录验证脚本重新计算参考分钟利润。
8. 任一复杂度的目标利润改变时，必须整体检查所有同级工厂和完整产业链，不能只修补单个亏损商品。
9. 新工厂必须声明 C1～C7 复杂度，施工时间必须落在对应区间，且所有正式配方必须精确达到该复杂度的参考分钟利润；调整现有复杂度时必须同步审查利润、同级工厂及相邻等级的时间递增关系。
10. 调整建造费时必须同步按 130% 向上取整到 5 更新 `systemValue`，并更新本节表格与 `verify-industry-catalog.mjs` 的精确断言。

## 4. 工厂集群状态

```ts
interface FacilityGroup {
  facilityTypeId: string;
  count: number;
  participatingCount: number;
  pendingJoinCount: number;
  listedCount: number;
  auctionedCount: number;
  frozenCount: number;
  availableCount: number;
  nextCycleCount: number;
  enabled: boolean;
  status: 'running' | 'stopped' | 'error';
  statusReason?: FacilityStatusReason;
  cycleStartedAt?: number;
  lifetimeOutput: number;
  activeRecipeId: string;
  pendingRecipeId?: string;
}
```

三种顶层状态：

- `running`：开启且当前完整周期正在运行。
- `stopped`：玩家关闭运行意图。
- `error`：运行意图仍开启，但资金、原料、仓库或可用数量条件不足。

`enabled` 表示玩家运行意图，不等同于当前是否正在生产。异常条件恢复后由服务器自动恢复，不要求再次点击启动。

## 5. 统一周期

```text
实际周期产量 = 单座周期产量 × participatingCount
实际周期成本 = 单座周期成本 × participatingCount
每项实际周期原料 = 该项单座输入数量 × participatingCount
周期净仓库增长 = max(0, 实际周期产量 - Σ（每项实际周期原料）)
```

每个完整周期在一个 SQLite 事务中：

1. 检查运行意图、可用数量、资金、原料、仓库空间和当前配方。
2. 同时检查全部输入、资金与仓库条件；任一条件不满足时本周期不得产生部分扣款或部分扣料。
3. 扣除整个集群成本和全部输入原料。
4. 产成品直接增加到共享仓库。
5. 增加 `lifetimeOutput` 和玩家生产统计。
6. 推进周期时间。
7. 在周期边界合并 `pendingJoinCount`。
8. 应用 `pendingRecipeId`，然后按新配方进入下一周期。

不得让同一集群的一部分工厂单独生产。生产公式只展示集群参数：运行中使用 `participatingCount`，停止或异常使用 `nextCycleCount`；周期时间不乘数量，公式与服务器结算口径保持一致。

## 6. 下一周期加入

施工完成、工厂买单或工厂拍卖成交、订单簿卖单撤销或拍卖取消／流拍解冻发生在运行期间时：

```text
当前参与数量保持不变
新增数量进入 pendingJoinCount
当前周期按旧数量结算
下一周期合并新增数量
```

新增数量不得重置 `cycleStartedAt`，不得获得部分周期产出。运行中提交工厂卖单或工厂拍卖时，服务器先结算已经完成的完整周期，再从当前参与数量和待加入数量中冻结对应数量。订单簿卖单与拍卖冻结数量合并计算，并立即降低当前或下一周期产能；取消或流拍解冻后从下一完整周期加入。

## 7. 运行、停止与自动恢复

- 玩家打开开关后，`enabled = true`。
- 条件满足时状态为 `running`。
- 资金、原料、仓库或可用工厂不足时状态为 `error`。
- 条件恢复后自动恢复，从新的完整周期开始，不补算异常期间错过的周期。
- 玩家关闭开关后 `enabled = false` 且状态为 `stopped`，不得自动启动。
- 停止前只结算服务器时间上已经完成的完整周期；未完成周期不扣费、不耗料、不产出。

自动恢复必须由统一协调函数处理，不能由页面按钮或本地倒计时决定。

## 8. 持续生产与通用配方切换

- 工厂只有持续生产模式；开启后一直运行到玩家关闭或出现异常。
- 不存在定量生产、目标产量、生产计划、计划完成状态或计划自动保存。
- 每个工厂集群保存一个 `activeRecipeId`；多配方工厂可以选择其他正式配方，单配方工厂提交唯一配方时幂等成功。
- 停止或异常状态下切换配方立即写入 `activeRecipeId`，随后重新检查自动运行条件。
- 运行中切换配方只写入 `pendingRecipeId`；当前完整周期仍按旧配方结算，周期边界再提升为新的 `activeRecipeId`。
- 同一周期内连续切换只保留最后一次选择，不得重置 `cycleStartedAt`、重复产出或获得部分周期产量。
- 工厂作为资产交易时不携带卖方配方配置；买入数量加入买方既有工厂集群，并使用买方当前配方。
- 所有工厂详情统一显示启用的“生产配方”选择器；多配方工厂可选择其他正式配方，单配方工厂显示唯一选项并保持启用。重复选择当前正式配方不得提交经济动作、重置周期、刷新进度或显示成功提示。

客户端开关只绑定 `group.enabled`，配方选择器只提交服务器正式配方 ID。旧 `/plan` 接口返回 `410 Gone`，不得静默接受旧生产计划。

### 9.4 当前工厂详情

- 详情显示工厂名称和总数量、运行开关、完整状态、“运行中／下一周期加入／冻结中”三列摘要、生产配方、集群生产公式、生产进度、玩家可见配方利润分析和市场入口。

- 集群生产公式支持无输入、单输入、多输入和单输出；客户端必须逐项渲染并显示 `+` 分隔。

- 生产公式只展示集群参数。运行中以 `participatingCount` 作为公式乘数并标记“本周期 × N”；停止或异常使用 `nextCycleCount`，分别标记“启动后 × N”或“恢复后 × N”。
- 当前周期的输入、输出和成本不得提前计入 `pendingJoinCount`；公式不得使用总持有 `count` 作为生产乘数。
- 生产进度条是公式容器最后一个可见元素；进度条下方不得显示当前周期、恢复运行、产出、成本或其他说明文字。利润分析是公式后的独立同级区域，不得塞入进度条容器。

- 工厂集群开关点击区域与可见胶囊保持 `2.75rem × 1.6rem`。
- 市场入口固定为“前往市场交易该工厂 →”，并位于详情底部。

### 9.5 玩家可见配方利润分析

利润分析是客户端只读经营辅助，不是服务器承诺收益，也不参与生产结算、订单撮合、估值、排行榜或市场需求。运行中没有待切换配方时按当前 `participatingCount` 和当前配方分析；停止或异常时按 `nextCycleCount` 分析；存在 `pendingRecipeId` 时改为按下一周期配方和 `nextCycleCount` 分析，避免把新配方套在当前周期数量上。

计算只读取当前状态快照中的商品 `open`／`partial` 公开订单、正式配方、可用库存和建造费，并排除当前玩家自己的反向订单。采购原料按卖价从低到高逐档吃单，出售原料或产成品按买价从高到低逐档吃单；同价时保持发布时间和订单 ID 的稳定顺序。不得使用基础价、参考价或未成交价格外推，也不得用最后成交价填补缺失盘口。

固定展示以下口径：

```text
补料成本 = 只购买当前可用库存未覆盖的输入数量
产出净额 = 产出逐档卖出总额 − 单张玩家卖单预计手续费
本轮现金利润 = 产出净额 − 周期运营成本 − 补料成本
全量采购利润 = 产出净额 − 周期运营成本 − 全部输入逐档采购成本
相比直接卖原料 = 产出净额 − 周期运营成本 − 各输入逐档直卖净额
```

玩家卖方手续费按统一订单簿规则估算：成交额大于零时为 `max(1, ceil(累计成交额 × 1%))`。产出按一张卖单估算；多种输入直接出售时每种商品视为独立卖单并分别计算最低手续费。买入原料不收取卖方手续费。

任一所需输入卖盘或产出买盘无法覆盖完整数量时，对应完整利润必须显示为不可估算并明确展示已覆盖数量，不得把部分成交额等比例放大。原料买盘不足时“相比直接卖原料”不可估算。每分钟利润按当前配方周期折算；有输入配方使用“相比直接卖原料”的加工增值，无输入配方使用生产现金利润。静态回本使用当前分析范围对应的建造成本除以每分钟利润；利润不为正或盘口不足时不得显示虚假回本时间。

界面必须展示补料成本、产出净额、卖方手续费、本轮现金利润、全量采购利润、相比直接卖原料、每分钟利润、静态建造回本、逐输入库存与补买深度，以及盘口不足、预计亏损或加工倒挂提示。状态轮询更新订单簿、库存或配方后本地重算；不得新增利润专用轮询、服务器字段、数据库写入或经济动作。

窄屏利润分析保持紧凑而不删减信息：摘要使用两列，本轮现金利润占满一行并优先显示；其余指标保持两列，标签与数值同行、说明位于下一行。不得在移动端恢复三项摘要逐项单列、隐藏说明或创建利润卡内部滚动区。

### 9.6 移动端详情悬浮框

- 不大于 `720px` 时页面内只显示固定三列的紧凑工厂选择网格，不显示独立“当前工厂”栏或“查看详情”按钮；点击任意选择卡后直接从屏幕底部打开 Bottom Sheet，不在页面中继续渲染完整详情卡。极窄屏仍保持三列，不降为两列或单列。
- 悬浮框使用 `role="dialog"`、`aria-modal="true"` 和标题关联；打开后焦点进入可程序化聚焦的对话框容器，第一次 `Tab` 再进入开关、配方选择器或市场按钮，焦点始终限制在悬浮框内。`Escape`、点击遮罩或有效向下拖动可以关闭，关闭后焦点返回触发卡。
- 悬浮框使用“固定头部／唯一 `ScrollArea` 正文／固定底部操作区”三段结构。头部包含拖动把手、标题、标题下方状态、运行开关和三项数量摘要，不包含顶部关闭按钮；正文只包含配方、公式、进度和利润分析；底部只包含市场入口。`src/styles/facility-detail-sheet.css` 是移动悬浮框布局、拖动反馈和固定头尾的最终 CSS 权威。
- 点击遮罩和按下 `Escape` 必须与有效下拉关闭共用同一收起流程：先向下移动悬浮框并同步淡出遮罩，动画结束后再卸载；动画期间忽略重复关闭请求。`prefers-reduced-motion: reduce` 时允许立即关闭。
- 悬浮框打开期间锁定背景页面滚动，并抑制背景 `.page-scroll-area` 的覆盖式纵向轨道；关闭后只允许页面在后续实际滚动时重新显示轨道。视口变为大于 `720px` 时必须自动关闭并解除滚动锁与轨道抑制。
- 悬浮框最大高度为 `min(88dvh, 760px)`，顶部圆角，固定底部包含 `safe-area-inset-bottom`。正文共享 `ScrollArea`，设置 `axis="y"` 与 `scrollbarVisibility="adaptive"`，并作为唯一纵向滚动容器；生产公式、配方区域、利润卡和底部操作区不得再创建嵌套纵向滚动。
- 触控模式下，悬浮框纵向轨道只在正文 `scrollTop` 实际变化后显示；显示期间允许触摸拖动滑块与点击轨道翻页，停止滚动并空闲 `1600ms` 后淡出。触摸按下但没有滚动不得显示，触控横向轨道继续隐藏，不得复制共享滚动条尺寸、计时器或活动判断。
- 从拖动把手或固定头部非交互空白处可以向下拖动；从正文开始时，只有正文 `scrollTop = 0` 且向下位移超过 `8px` 并大于横向位移的 `1.2` 倍，才接管为关闭手势。正文未到顶时不得抢走正常滚动；按钮、链接、输入、选择器、运行开关、滚动条滑块与轨道不得成为关闭拖动起点。
- 拖动跟手反馈只通过 DOM ref、CSS 自定义属性和 `requestAnimationFrame` 更新 `translateY` 与遮罩强度，不得在每次移动时写 React state。向下距离达到悬浮框高度的 `25%`（至少 `96px`、至多 `160px`），或距离至少 `40px` 且释放速度达到 `0.75px/ms` 时关闭；否则回弹原位。
- 固定头部与底部使用不透明主表面。市场入口不得通过正文内部负 `bottom` 或 `position: sticky` 覆盖利润内容，最后一项利润数据必须能够完整滚动到底部操作区上方；底部市场按钮占满宽度且最小高度为 `48px`。
- 启停和配方操作完成后悬浮框保持打开并原地刷新。`prefers-reduced-motion: reduce` 时保留跟手拖动，但释放后立即关闭或复位，并关闭底部滑入、遮罩淡入、惯性和回弹过渡。

## 11. 防回退

不得恢复：

- 单座工厂实例或设施槽位；
- 工厂内部仓库和领取产成品；
- 固定价格工厂挂牌市场；
- 手动修复异常后必须再次点击启动；
- 恢复定量生产、目标产量、生产计划或计划完成状态；
- 运行中切换配方立即重置周期或中途改变当前周期产出；
- 把一个玩家的同类工厂集群拆分成多种并行配方；
- 新数量中途加入当前周期；
- 让共享仓库、建设卡、工厂选择和桌面详情在同一断点使用不同一级外层内边距；
- 恢复同时铺开所有完整工厂卡、四列完整卡网格或瀑布流；
- 在 `1600px` 及以上把工厂集群选择恢复为单列，或让当前详情列继续扩张到超过 `680px`；
- 让桌面详情壳恢复交叉轴拉伸、`min-height: 100%`、弹性空白行或可见占位 spacer；

### 拍卖冻结与工厂总量

工厂订单与拍卖冻结只减少可用于当前和下一周期生产的数量，不提前减少 `group.count`。同一资产包中的多种工厂分别按类型冻结并全部退出生产；发布、取消或流拍前后总持有数量不变，因此冻结工厂继续按最近一次订单簿真实成交价计入总资产。成交完成后才从卖方总量扣除并加入买方集群；整包任一项目异常时不得部分转移工厂。

## 市场需求 V6 生产路径规则

- 派生需求读取上一完整周期结算后的有效下游拉动：实际成交数量 + 50% 新未满足数量 + 15% 更早积压数量。
- 同一输出商品的生产路线份额同时读取单位生产成本和各输入公开卖单覆盖率；理论便宜但无法获得原料的路线不得主导派生需求。
- 多路线输出的成本锚点按当前路线份额加权，不得继续直接选择理论最低成本路线。
- 每周期仍只允许沿一条正式生产边传播，互补输入继续使用其他输入最低覆盖率门控。
