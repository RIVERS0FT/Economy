# Economy 人口消费与基础买单设计

> 状态：多人市场需求侧设计基线  
> 适用范围：网页端多人实时交易版本  
> 更新时间：2026-07-10

## 1. 设计目的

共享市场不能只依赖玩家相互买卖，否则在玩家较少、生产者多于贸易商或服务器刚开放时，容易出现缺少买单、商品无法出售和经济循环停滞。

游戏通过模拟城市人口产生持续消费需求，为工厂生产的基础商品提供有经济含义的基础买单。

人口消费不是固定价格商店，也不是无限回收系统。人口买单必须来源于人口规模、居民预算、商品需求、市场价格和需求满足率。

## 2. 核心经济循环

```text
玩家拥有工厂
-> 工厂生产基础商品
-> 玩家向统一市场提交卖单
-> 城市人口产生消费需求
-> 人口需求形成基础商品买单
-> 统一订单簿撮合
-> 商品成交后被居民消费并退出流通
-> 卖方获得货币
-> 玩家继续扩建、升级或收购工厂
```

人口解决基础需求侧问题，玩家工厂解决供给侧问题。

## 3. 核心原则

### 3.1 人口是真实需求来源

每笔人口买单都对应一个消费周期中的需求和预算。

### 3.2 不保证玩家盈利

居民只会在价格承受范围内购买。玩家报价过高时，卖单可以长期无法成交。

### 3.3 不无限购买

每个周期的目标需求和预算有限。需求满足后，不再无限购买。

### 3.4 共用商品订单簿

人口买单进入与玩家相同的基础商品订单簿，遵循价格优先、同价时间优先、部分成交和订单过期规则。

### 3.5 商品成交后退出流通

人口购买的基础商品视为已经消费，不进入系统库存，也不能再次出售。

### 3.6 需求变化可解释

人口、预算、价格、短缺和城市事件改变需求时，页面应向玩家展示主要原因和趋势。

## 4. 城市人口模型

城市聚合状态包括：

- 总人口
- 有效消费人口
- 居民平均收入
- 周期可支配消费预算
- 基础商品目标需求
- 需求满足率
- 生活成本指数
- 城市繁荣度
- 当前人口事件

MVP 不逐个模拟居民，只维护聚合人口与需求数据。

## 5. 消费周期

MVP 建议一个消费周期为 5 分钟。

每周期执行：

1. 计算人口可用消费预算。
2. 计算基础商品目标需求。
3. 扣除本周期已满足需求。
4. 根据价格承受能力生成少量买单。
5. 实时与玩家商品卖单撮合。
6. 周期结束时过期或重新评估未成交人口订单。
7. 记录消费量、均价、预算使用量和需求满足率。

买单生成后可立即成交，不需要等到周期结束。

## 6. 需求与预算

```text
周期目标需求 = 有效消费人口 × 人均基础需求
```

```text
周期消费预算 = 有效消费人口 × 人均可支配预算 × 基础商品预算占比
```

人口所有成交总量不能超过剩余目标需求，成交总额不能超过剩余预算。

未使用预算和未满足需求可以少量结转，但必须设置上限，避免长期缺货后一次性产生极端需求。

## 7. 价格承受能力

基础商品拥有：

- 参考价格
- 正常承受价格
- 紧缺承受价格
- 最高承受价格

价格越高，人口愿意购买的数量越少。

```text
可购买数量 = 周期剩余需求 × 价格需求系数
```

初始测试示例：

| 基础商品价格 | 需求系数 |
|---:|---:|
| 3 | 1.20 |
| 4 | 1.10 |
| 5 | 1.00 |
| 6 | 0.80 |
| 7 | 0.55 |
| 8 | 0.30 |
| 9 及以上 | 0 |

数值仅用于第一轮测试。

## 8. 人口买单生成

人口每周期只生成少量价格阶梯，不按居民数量生成独立订单。

以参考价 5 为例：

| 买价 | 数量 | 说明 |
|---:|---:|---|
| 5 | 8 | 即时需求 |
| 4 | 7 | 价格敏感需求 |
| 3 | 5 | 低价补充需求 |

所有买单共享本周期剩余预算和剩余需求。

人口订单在统一市场中显示为“城市需求”或“居民消费”，资产类型为基础商品。

人口不会购买工厂，工厂交易只发生在玩家之间。

## 9. 撮合与结算

人口买单成交后：

```text
玩家冻结库存减少
-> 玩家获得成交货币
-> 人口剩余预算减少
-> 人口剩余需求减少
-> 基础商品退出流通
-> 生成成交记录与资金流水
```

人口订单支持部分成交，但不能超额使用同一预算或需求。

## 10. 需求满足率

```text
需求满足率 = 已消费数量 / 周期目标需求
```

| 满足率 | 状态 |
|---:|---|
| 90% 以上 | 供应充足 |
| 60%～89% | 供应正常 |
| 30%～59% | 供应紧张 |
| 30% 以下 | 严重短缺 |

满足率可以影响：

- 下一周期需求
- 居民可接受价格
- 城市繁荣度
- 后续人口增长
- 玩家建厂与扩产判断

MVP 只用于趋势展示和下一周期需求调整，不加入复杂城市惩罚。

## 11. 供给过剩

当工厂产量和市场卖单远高于人口需求时：

- 人口不会无限增加购买量。
- 低价卖单优先成交。
- 高价卖单可能无法成交。
- 基础商品价格自然下降。
- 工厂内部成品和企业仓库可能积压。
- 玩家需要暂停工厂、降价、减少扩建或等待需求恢复。

即使价格很低，额外消费也必须有上限。

## 12. 商品短缺

当人口需求长期无法满足时：

- 少量未满足需求可以结转。
- 居民最高承受价可有限提高。
- 页面显示供应紧张或严重短缺。
- 更高利润会吸引玩家建厂、收购工厂或扩大产能。

短缺不能直接生成玩家商品或工厂。

## 13. 人口与工厂经济

人口需求会影响工厂经营价值：

- 需求旺盛时，工厂预期收益和玩家收购意愿提高。
- 供给过剩时，工厂可能被暂停或以较低价格挂牌。
- 工厂交易价格由玩家决定，人口不直接购买工厂。
- 工厂系统估值可以参考近期商品价格和产能，但不能保证利润。

## 14. 居民预算与货币发行

人口成交会向经济中注入新货币，因此需要持续监测：

- 人口消费发行货币量
- 点击工作发行货币量
- 工厂建造、运营、升级和扩建回收量
- 玩家持有货币增长速度
- 基础商品通胀或通缩
- 工厂成交价格变化

人口预算是有限的系统货币发行来源，不能无限增长。

## 15. 玩家可见信息

统一市场中的人口需求区域展示：

- 当前城市人口
- 本周期预计需求
- 本周期已消费数量
- 需求满足率
- 下次需求刷新时间
- 当前价格承受范围
- 供应状态
- 下一周期趋势
- 影响需求的事件

不公开下一批每档买单的精确数量，避免完全无风险地提前抢占系统需求。

## 16. 防止套利

- 不设置固定永久保底价。
- 所有人口订单共享有限周期预算。
- 成交后立即扣减剩余预算和需求。
- 高于最高承受价的卖单不会被人口购买。
- 极低价格也不能产生无限额外需求。
- 用于经验、任务和排行榜的有效人口成交量可以设周期上限。
- 人口需求不能被用于工厂产权转移或玩家间变相转账。

## 17. 2 核 2G 约束

- 人口按聚合周期计算。
- MVP 只模拟一个城市、一个消费群体和一种基础商品。
- 每周期只生成 3～5 档人口买单。
- 不模拟居民个人账户、职业、家庭和购物路径。
- 人口图表、动画和完整需求曲线由客户端计算。
- 服务器只维护正式预算、需求、订单和成交结果。

## 18. MVP 初始建议

| 项目 | 建议值 |
|---|---:|
| 城市展示人口 | 10,000 |
| 有效消费人口 | 1,000 |
| 消费周期 | 5 分钟 |
| 基础商品目标需求 | 每周期 20 个 |
| 基础商品参考价 | 5 |
| 正常承受价 | 6 |
| 最高承受价 | 8 |
| 周期消费预算 | 100 货币 |
| 未使用预算结转上限 | 下一周期预算的 25% |
| 未满足需求结转上限 | 下一周期需求的 50% |

## 19. 验收标准

1. 人口按周期生成基础商品需求。
2. 买单总额不超过周期预算。
3. 买单总量不超过周期目标需求和有限结转需求。
4. 人口订单进入玩家基础商品订单簿。
5. 人口订单支持部分成交。
6. 成交商品退出流通。
7. 卖方资金和流水正确变化。
8. 预算与需求随成交正确减少。
9. 人口不会购买超过最高承受价的商品。
10. 供给过剩时人口不会无限购买。
11. 周期结束后旧人口订单能够正确过期。
12. 新周期根据价格和上一周期满足率重新计算。
13. 页面能够展示人口需求趋势。
14. 并发成交不会重复使用预算。
15. 人口成交发行货币可以统计和审计。
16. 人口不会参与工厂购买。
17. 人口系统暂停不会影响玩家已有订单和资产。
18. 系统不存在永久、固定、无限数量的保底收购价。

## 20. 当前开发决策

- 人口消费是基础商品基础买单的主要来源。
- 人口买单进入统一市场中的商品订单簿。
- 人口不购买工厂。
- 人口购买后的基础商品退出流通。
- 人口预算、需求和价格承受能力均有限。
- MVP 只模拟一个城市和一种基础商品。
- 人口采用聚合周期，不模拟单个居民。
