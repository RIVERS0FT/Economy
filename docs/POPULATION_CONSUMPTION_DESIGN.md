# Economy 人口消费与基础买单设计

> 状态：多人市场需求侧设计基线  
> 适用范围：网页端多人实时交易版本  
> 更新时间：2026-07-10

## 1. 设计目的

共享市场不能只依赖在线玩家互相买卖，否则在玩家数量较少、生产者多于消费者或新服务器刚开放时，市场容易出现没有买单、商品无法出售和经济循环停滞的问题。

游戏通过模拟城市人口产生持续消费需求，为商品市场提供有经济含义的基础买单。

人口消费不是固定价格商店，也不是无限回收系统。系统买单必须来源于人口规模、居民收入、消费偏好、商品库存缺口和市场价格承受能力。

本设计取代《网页端多人实时交易游戏设计》中原有的“系统基础流动性”概念。后续提到系统基础买单时，均指人口消费需求生成的订单。

## 2. 核心经济循环

```text
玩家生产商品
-> 玩家向共享市场提交卖单
-> 城市人口产生消费需求
-> 消费需求形成基础买单
-> 市场按照统一规则撮合
-> 商品成交后被居民消费并退出流通
-> 卖方获得资金
-> 玩家继续扩大生产和经营
```

人口系统解决需求侧问题，玩家企业解决供给侧问题。

## 3. 核心原则

### 3.1 人口是真实需求来源

每一笔人口买单都应对应一定数量居民在一定时间内产生的消费预算与商品需求。

### 3.2 不保证玩家盈利

人口只会在其价格承受范围内购买。玩家报价过高时，订单可以长期无法成交。

### 3.3 不无限购买

每个周期的人口、收入和需求都有限。需求满足后，本周期不会继续无上限购买同类商品。

### 3.4 与玩家使用同一市场规则

人口买单进入与玩家相同的订单簿，遵循价格优先、时间优先、部分成交和订单过期规则。

### 3.5 商品成交后退出流通

人口购买的商品视为被居民消费，不进入任何可再次出售的系统库存，以形成稳定商品消耗出口。

### 3.6 需求变化必须可解释

人口增加、收入变化、物价变化、节日或短缺事件可以改变需求，但页面应向玩家展示主要原因。

## 4. 城市人口模型

市场对应一个持续运行的虚拟城市或经济区域。

城市基础状态包括：

- 总人口
- 就业人口
- 居民平均收入
- 居民可支配消费预算
- 各商品基础需求
- 各商品需求满足率
- 生活成本指数
- 城市繁荣度
- 当前人口事件

MVP 不需要模拟每一个居民。人口按总体统计或消费群体进行计算。

## 5. 消费群体

后续可以将人口分为不同消费群体：

| 群体 | 主要特点 | 价格敏感度 |
|---|---|---|
| 基础居民 | 优先购买食品和日用品 | 高 |
| 中等收入居民 | 需求较稳定，购买多类商品 | 中 |
| 高收入居民 | 更愿意购买高价和高级商品 | 低 |

MVP 只有食品包一种商品，可以先将所有人口视为基础居民，但数据规则必须允许未来增加群体。

## 6. 人口基础需求

每种商品配置人均基础需求。

MVP 食品包建议按一个消费周期计算：

```text
食品包基础需求 = 有效人口 × 人均食品包需求
```

“有效人口”是本周期实际参与消费的人口，不必等于展示人口总数，可以通过比例缩放，避免订单数量过大。

示例：

```text
城市展示人口：10,000
有效消费比例：10%
有效人口：1,000
人均食品包需求：0.02
本周期基础需求：20 个食品包
```

具体人口单位和缩放比例应以游戏体验为准，不要求模拟现实世界精确比例。

## 7. 消费周期

人口需求按固定消费周期产生。

MVP 建议一个周期为 5 分钟。

每个周期执行：

1. 计算人口可支配预算。
2. 计算食品包目标需求量。
3. 扣除本周期已经满足的需求。
4. 根据当前市场价格生成买单。
5. 未成交订单在周期结束时过期或重新评估。
6. 记录本周期消费量、均价和需求满足率。

消费周期不代表市场只能每 5 分钟成交。买单生成后仍可实时与玩家卖单成交。

## 8. 人口消费预算

人口不能凭空无限产生资金。

每个消费周期拥有独立消费预算：

```text
周期消费预算 = 有效人口 × 人均可支配预算 × 商品预算占比
```

食品包预算示例：

```text
有效人口：1,000
人均周期可支配预算：0.2
食品预算占比：50%
食品包周期预算：100 资金
```

系统本周期所有食品包成交总额不得超过该预算。

未使用预算的处理方式：

- 默认不完整累积，防止长期无人供货后一次性产生极端大买单。
- 可以保留少量至下一周期，形成短期积压需求。
- 累积预算必须有上限。

## 9. 人口价格承受能力

居民不会以任意价格购买商品。

每种商品拥有：

- 参考价格
- 正常承受价格
- 紧缺承受价格
- 最高承受价格

食品包 MVP 示例：

| 价格区间 | 消费行为 |
|---|---|
| 参考价以下 | 需求强，愿意购买完整目标量 |
| 参考价附近 | 正常购买 |
| 高于正常承受价 | 减少购买数量 |
| 接近最高承受价 | 只满足最低生存需求 |
| 高于最高承受价 | 不创建人口买单 |

价格越高，人口愿意购买的数量越少。

建议使用需求曲线而不是单一固定回收价：

```text
可购买数量 = 基础需求 × 价格需求系数
```

示例系数：

| 食品包价格 | 需求系数 |
|---:|---:|
| 3 | 1.20 |
| 4 | 1.10 |
| 5 | 1.00 |
| 6 | 0.80 |
| 7 | 0.55 |
| 8 | 0.30 |
| 9 及以上 | 0 |

表中数值仅作为初始调试建议。

## 10. 基础买单生成

人口需求不应只生成一档价格的大订单，建议形成小型买单阶梯。

以参考价 5 为例，一个周期可能生成：

| 买价 | 数量 | 说明 |
|---:|---:|---|
| 5 | 8 | 正常即时需求 |
| 4 | 7 | 价格敏感需求 |
| 3 | 5 | 低价储备需求 |

买单总数量不能超过周期剩余需求，总金额不能超过周期剩余预算。

人口买单在订单簿中可以显示为“居民消费”或“城市需求”，不显示虚构个人账号。

## 11. 撮合与结算

人口买单遵循与玩家相同的撮合规则：

- 价格优先
- 同价时间优先
- 允许部分成交
- 周期结束可以过期
- 市场暂停时停止新增订单

人口买单成交后：

```text
玩家冻结库存减少
玩家获得成交资金
人口消费预算减少
人口剩余需求减少
成交商品退出市场流通
生成成交记录与卖方资产流水
```

人口买单不产生可出售库存，也不允许系统再次卖回同一商品。

## 12. 需求满足率

每个商品记录当前消费周期的需求满足率：

```text
需求满足率 = 已消费数量 / 目标需求数量
```

状态建议：

| 满足率 | 状态 |
|---:|---|
| 90% 以上 | 供应充足 |
| 60%～89% | 供应正常 |
| 30%～59% | 供应紧张 |
| 30% 以下 | 严重短缺 |

需求满足率影响：

- 下一周期人口买单数量
- 人口愿意接受的价格
- 城市繁荣度
- 后续人口增长
- 市场事件触发概率

MVP 可先只展示并用于调整下一周期需求，不立即加入复杂城市惩罚。

## 13. 供给过剩

当市场卖单远多于人口需求时：

- 人口不会无限提高购买量。
- 低价卖单优先成交。
- 高价卖单可能无法成交。
- 商品价格自然下降。
- 玩家需要减少生产、降价或等待下一周期需求。

可允许低价刺激少量额外消费，但必须存在上限。

```text
额外需求 ≤ 基础需求 × 价格刺激上限
```

## 14. 商品短缺

当人口需求长期无法满足时：

- 下一周期保留有限积压需求。
- 人口可提高最高接受价格，但不得无限上升。
- 页面显示“供应紧张”或“严重短缺”。
- 后续可以影响城市繁荣度和人口增长。

短缺不能直接凭空增加玩家库存，只能通过更高需求吸引玩家扩大生产。

## 15. 人口增长与减少

后续版本中，人口可以根据城市经济状态变化。

人口增长因素：

- 基础商品供应稳定
- 物价处于合理区间
- 城市繁荣度提高
- 玩家企业提供足够就业
- 连续多个周期需求满足率较高

人口减少或增长放缓因素：

- 长期食品短缺
- 物价长期过高
- 城市繁荣度下降
- 特殊灾害或经济事件

MVP 建议人口总量保持稳定，先验证基础消费买单是否能够支撑市场循环。人口动态增长放到后续版本。

## 16. 居民收入与资金来源

人口消费预算代表城市居民通过游戏世界中的工资、公共收入和经济活动获得的可支配资金。

MVP 不需要逐个模拟居民工作，但必须限制系统资金注入速度。

可以将人口预算视为服务器每周期允许进入市场的基础货币发行量。

需要持续观察：

- 人口消费注入资金总额
- 玩家升级和费用回收的资金总额
- 玩家持有资金增长速度
- 商品价格通胀

后续可通过以下资金出口平衡经济：

- 设施升级费用
- 仓库升级费用
- 交易手续费
- 企业维护费用
- 城市税费
- 生产许可或研究费用

MVP 可以暂不收取交易手续费，但必须记录人口消费累计注入资金，便于调整。

## 17. 玩家可见信息

市场页面增加“城市消费”区域，展示：

- 当前城市人口
- 当前商品需求状态
- 本周期预计需求
- 本周期已消费数量
- 需求满足率
- 下次需求刷新时间
- 当前居民可接受价格区间
- 影响需求的事件

不向玩家公开精确的下一批每档买单数量，避免玩家完全无风险地提前挂在最高系统价位。

可以公开趋势，例如：

```text
食品包需求：旺盛
供应状态：紧张
居民价格承受能力：中等
下个周期需求预计：上升
```

## 18. 防止利用人口买单套利

### 18.1 禁止固定保底价

人口买价会随预算、价格、需求满足率和市场库存变化，不能成为永远固定的最低售价。

### 18.2 限制周期总预算

所有人口买单共享周期预算，不能通过拆单获得额外系统资金。

### 18.3 需求按成交扣减

人口订单被不同玩家成交后，剩余需求同步下降，不能重复购买同一份需求。

### 18.4 价格异常保护

明显高于居民最高承受价的卖单不会被人口购买。

### 18.5 低价需求上限

即使价格极低，人口额外消费也存在上限，避免玩家通过低价循环制造无限成交量或经验。

### 18.6 经验和排行榜限制

与人口成交可以获得正常经营收益，但用于经验、任务和排行榜的有效成交量可以设置周期上限，防止对刷系统需求。

## 19. 玩家与人口的市场关系

人口是基础消费者，但不是唯一消费者。

完整市场由以下参与者构成：

- 玩家生产者
- 玩家贸易商
- 玩家消费者或加工企业
- 城市居民人口
- 后续公共机构或城市项目

玩家买单可以高于人口买价，从而优先获得商品；玩家卖单可以低于其他卖家，从而优先满足人口需求。

## 20. 后续扩展

### 20.1 多商品消费篮子

人口会在食品、日用品、住房相关商品和电子产品之间分配预算。

### 20.2 替代品

价格过高时，人口可以从某种商品转向替代品。

### 20.3 城市区域

不同区域拥有不同人口、收入和消费偏好，但仍可进入统一或分区市场。

### 20.4 公共采购

学校、医院、城市建设等公共机构产生大宗限时需求。

### 20.5 节日与事件

节日提高特定商品需求，灾害造成短缺，经济繁荣提升总体预算。

### 20.6 玩家影响城市

玩家企业提供就业、缴纳费用、完成公共订单，从而影响人口和繁荣度。

## 21. MVP 规则

MVP 只实现：

1. 固定城市人口。
2. 食品包一种人口消费商品。
3. 固定长度消费周期。
4. 每周期有限食品消费预算。
5. 根据价格变化的需求数量。
6. 多档人口限价买单。
7. 人口买单进入共享订单簿。
8. 人口成交后的商品销毁。
9. 周期需求满足率。
10. 人口订单过期与下一周期重新计算。
11. 市场页面显示消费趋势。
12. 人口消费资金注入统计。

MVP 暂不实现：

- 动态人口增长
- 多收入群体
- 多商品消费篮子
- 替代品
- 居民个人账户
- 公共机构采购
- 城市税收
- 复杂宏观经济模型

## 22. MVP 初始数值建议

以下数值仅用于第一轮测试：

| 项目 | 建议值 |
|---|---:|
| 城市展示人口 | 10,000 |
| 有效消费人口 | 1,000 |
| 消费周期 | 5 分钟 |
| 食品包目标需求 | 每周期 20 个 |
| 食品包参考价 | 5 |
| 食品包正常承受价 | 6 |
| 食品包最高承受价 | 8 |
| 食品包周期预算 | 100 |
| 未满足预算结转上限 | 下一周期预算的 25% |
| 未满足需求结转上限 | 下一周期基础需求的 50% |

测试重点不是立即确定最终数值，而是验证：

- 新玩家生产的商品能够合理成交。
- 市场不会变成固定价格回收商店。
- 供给增加时价格会下降。
- 商品短缺时价格会提高。
- 人口资金注入不会造成失控通胀。

## 23. 验收标准

1. 人口能够按消费周期生成食品包需求。
2. 每个周期人口买单总额不超过周期预算。
3. 人口买单进入与玩家相同的订单簿。
4. 玩家卖单可以与人口买单实时成交。
5. 人口订单支持部分成交。
6. 成交后商品从经济中退出，不能再次出售。
7. 成交后卖方资金和资产流水正确变化。
8. 人口剩余预算和需求随成交正确减少。
9. 人口不会购买超过最高承受价的商品。
10. 供给过剩时人口不会无限购买。
11. 需求不足时高价卖单可以无法成交。
12. 未满足需求只能有限结转。
13. 消费周期结束后旧人口订单能够正确过期。
14. 新周期会根据价格和上一周期满足率重新计算需求。
15. 页面能够展示人口、需求趋势和满足率。
16. 重复或并发结算不会导致人口预算被重复使用。
17. 人口成交产生的资金注入可以被统计和审计。
18. 人口系统停止时，玩家已有订单和资产不会丢失。
19. 市场暂停时不会继续生成可成交的人口订单。
20. 系统不存在永久固定、无限数量的保底收购价。

## 24. 当前开发决策

- 人口消费是共享市场基础买单的主要来源。
- 原有无背景的系统基础订单设计被本设计取代。
- 人口买单必须受人口、需求、预算和价格承受能力限制。
- 人口购买后的商品直接退出流通。
- 人口与玩家使用同一订单簿和撮合规则。
- MVP 人口固定，只消费食品包。
- MVP 先验证市场循环与货币注入，不实现复杂城市模拟。
