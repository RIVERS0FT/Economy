# Economy 权威倒计时与到期确认设计

> 状态：当前客户端倒计时、到期刷新与服务器确认基线
> 适用项目：`RIVERS0FT/Economy`
> 更新时间：2026-07-21

## 1. 唯一职责

本文档只负责客户端倒计时到达零点后的状态同步行为，包括：

- 服务器绝对截止时间的本地展示；
- `GET state` 响应时间与共享单调服务器时钟；
- 到期时立即刷新权威状态；
- 服务器结算尚未完成时的确认重试；
- 到期确认期间的按钮和文案状态；
- 新倒计时接入统一注册表的规则。

经济结算规则仍以产品、产业、拍卖和排行榜对应文档为准；普通状态轮询、修订号、分区补丁和服务器容量以 `SERVER_ARCHITECTURE_AND_DEPLOYMENT_DESIGN.md` 为准。

## 2. 倒计时分类

### 2.1 本地资格倒计时

截止时间本身已经足以判断操作资格，归零时不要求服务器删除对象或转换状态。

当前实例：

- 工作冷却 `work.cooldownUntil`。

客户端必须使用服务器返回的绝对截止时间计算剩余时间。归零后按钮立即恢复，不等待下一次轮询；服务器仍在提交操作时再次校验真实时间。

### 2.2 权威状态转换倒计时

归零只表示已经到达服务器应当结算的时点，客户端不能自行推断结算结果，也不能仅删除本地对象。

当前实例：

- 工厂施工完成 `facilityConstruction.completesAt`；
- 运行中工厂集群当前周期结束 `cycleStartedAt + activeRecipe.cycleMs`；
- 进行中资产拍卖结束 `assetAuction.endsAt`；
- 排行榜周期结算 `leaderboards.period.endsAt`。

这些截止时间必须登记在 `src/utils/authoritativeCountdowns.ts`。页面不得各自维护另一套到期刷新定时器。

## 3. 服务器时间来源

每个 `GET /api/game/state` 响应必须在六分区 envelope 顶层返回响应生成时的 `serverNow`。即使全局修订号未变化、响应只有轻量确认，也必须返回新的 `serverNow`。该字段不属于 `EconomyState`，不得进入 `catalog`、`player`、`market`、`auction`、`contract` 或 `leaderboard` 分区，也不得改变分区内容哈希。

客户端收到 `serverNow` 时记录浏览器单调时钟接收点，并建立共享单调服务器时钟：`serverNow + 接收后的单调经过时间`。新响应只能向前校准；当迟到响应或旧快照携带的时间低于当前估算值时，必须保留当前估算值，倒计时不得增加、回退或重新开始。页面共享 `useNow` 和全局到期协调器必须读取同一时钟。

`game.lastProcessedAt` 只表示世界状态最后实际保存或处理的时间，可作为旧响应或兼容测试的初始回退值；它不是 HTTP 响应时间。普通 3／5／10 秒轮询不得在每次接收状态时把 `lastProcessedAt` 重新解释为“当前服务器时间”，否则工作冷却、施工、生产和拍卖倒计时会被旧快照重新计时。

## 4. 统一到期确认流程

`AuthoritativeCountdownRefresh` 挂载在已登录游戏应用外壳中，覆盖所有页面：

1. 从统一注册表取得最近一个权威截止时间；
2. 使用共享单调服务器时钟估算当前服务器时间；
3. 在截止时间到达时立即调用 `model.refresh({ mode: 'authoritative', expectedDeadline })`；
4. 权威刷新必须中止正在等待的普通轮询，不能因为普通刷新锁或本地经济动作正在进行而永久跳过；
5. 状态读取固定使用 8 秒客户端超时；失败或服务器尚未删除到期对象时，每 `1,000ms` 继续确认；
6. 下一次确认只能在上一次请求结束后通过 `setTimeout` 安排，不得使用可能并发堆积请求的固定 `setInterval`；
7. 收到新状态后 React 依赖变化，旧定时器必须清理，并重新登记下一个截止时间；共享时钟向前校准时也必须重新核对最近截止时间；
8. 浏览器从后台恢复可见时立即重新判断截止时间；页面切换不得中止到期确认，因为协调器位于 `GameApp`，不属于具体页面。

到期确认只发送状态读取，不得提交重复经济动作，不得绕过全局修订号、服务器幂等规则或已确认动作的状态补拉。普通轮询可以复用进行中的刷新 Promise；权威刷新只能复用进行中的权威请求，遇到普通请求必须先中止旧请求再发起确认。

### 4.1 到期状态的分区替换语义

权威刷新取得的 `patches[name]` 是服务器为该分区生成的完整快照，不是分区对象内部的字段级补丁。客户端必须长期保存 `catalog`、`player`、`market`、`auction`、`leaderboard` 五个独立缓存分区；收到变化分区后整块替换同名分区，再按固定顺序重新组成页面使用的完整 `EconomyState`。

分区快照中缺失的可选字段表示服务器已经删除该字段。客户端不得把新分区浅合并进旧完整状态，也不得因为新快照省略字段而保留旧值。例如施工完成时，服务器会在同一次结算中把农场 `group.count` 从 11 增加到 12，并删除 `facilityConstruction`；客户端必须同时显示 12 座农场并结束“确认完工中”，不得只更新数量而残留旧施工对象。

未变化分区继续复用缓存快照；低于当前全局修订号的迟到响应不得替换任何分区。空对象同样是合法的完整分区快照，必须能够清除该分区原有的全部字段。

## 5. 页面表现

权威状态转换倒计时归零后、服务器仍返回旧状态期间：

- 与该状态冲突的操作继续禁用；
- 页面显示“确认中”“确认完工中”或“等待服务器结算”等明确文案；
- 不得继续显示负数时间；
- 不得为了让按钮提前恢复而在客户端删除施工、拍卖或生产状态；
- 服务器确认后立即恢复正常按钮或展示结算结果，不再等待默认 3／5／10 秒轮询间隔。

施工卡固定在归零后显示“确认完工中…”并保持建设按钮禁用，直到服务器删除 `facilityConstruction`。即使某次普通轮询、动作后状态补拉或代理连接未返回，8 秒超时也必须释放请求协调状态，使下一次权威确认能够继续执行。

## 6. 性能与时钟边界

- 根游戏模型不得维护每秒变化的 React 时间状态。
- 页面可继续通过共享 `useNow` 局部更新可见倒计时，但必须读取共享单调服务器时钟。
- `serverNow` 只属于状态交付 envelope；不得写入世界 JSON、`EconomyState` 或六分区缓存。
- 接收新 `serverNow` 可以通知当前可见倒计时和全局协调器重新校准，但不得触发全部静态页面每秒重渲染。
- 全局协调器只为最近一个权威截止时间维护一个 `setTimeout`；到期后最多维护一个串行确认请求和一个下一次重试 `setTimeout`。
- 默认状态轮询继续存在；统一到期确认只补足截止时点的即时同步，不替代普通轮询。
- 浏览器后台节流导致定时器延迟时，`visibilitychange` 恢复可见后必须立即确认当前已过期的最近截止时间。
- 普通状态读取超时为 8 秒，经济写请求超时为 12 秒；不得依赖浏览器或反向代理的无限默认等待。

## 7. 新倒计时接入规则

新增可见倒计时前必须先判断：

1. 归零后是否仅改变操作资格；若是，使用服务器绝对截止时间在页面本地解锁。
2. 归零后是否需要服务器结算、删除对象、转移资产、推进周期或发放奖励；若是，必须把截止时间加入 `authoritativeCountdownDeadlines()`。
3. 所有页面倒计时必须通过共享服务器时钟读取当前时间，不得自行用 `Date.now()` 或重新锚定 `lastProcessedAt`。
4. 权威倒计时归零后的页面文案和禁用状态必须明确。
5. 不得新增页面私有的到期轮询、递归经济动作或第二个全局秒级时钟。
6. 必须更新本设计文档和 `scripts/verify-authoritative-countdowns.mjs`。

## 8. 防回退

不得恢复：

- 仅依赖默认 3／5／10 秒轮询处理已显示倒计时的权威到期事件；
- 施工倒计时归零后仍长期显示“已有工厂正在施工”且不主动刷新；
- 拍卖结束后继续允许出价或取消；
- 页面自行把施工、生产、拍卖或排行榜标记为已结算；
- 为每张工厂卡或每场拍卖创建独立确认定时器；
- 把工作冷却错误登记为必须等待服务器状态转换的倒计时；
- 使用 `lastProcessedAt + 本次状态接收后的经过时间` 为每次轮询重新建立本地时钟；
- 允许较旧的 `serverNow` 或迟到响应使共享时钟、工作冷却或其他倒计时回退；
- 把 `serverNow` 放入任一状态分区，导致每次轮询都产生分区补丁；
- 到期确认失败后停止所有后续确认，只能等待下一次普通轮询；
- 删除统一注册表、共享单调服务器时钟、应用外壳协调器或每秒确认上限；
- 恢复无超时 `fetch`、单一布尔刷新锁、固定 `setInterval` 并发确认，或让权威刷新因普通轮询／动作请求而永久跳过；
- 把变化分区浅合并进旧完整状态，导致服务器已删除的 `facilityConstruction`、拍卖状态、待切换配方或其他可选字段继续残留。

`scripts/verify-authoritative-countdowns.mjs` 必须加入 `verify:architecture`，锁定注册表、共享单调服务器时钟、应用外壳协调、请求超时、权威刷新模式、后台恢复、串行每秒确认、施工确认文案、工作冷却例外、拍卖等待结算文案、完整分区替换和本文档规则。
