# Economy 本地活动日志设计

> 状态：当前浏览器本地日志存储与展示基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-12  
> 本地文档版本：`v3`

## 1. 核心原则

服务器不得持久化玩家活动日志。

服务器只保存权威经济状态，包括资金、商品库存、仓库、工厂类型集群、施工任务、统一生产计划、订单、工厂数量挂牌、市场、统计和幂等数据。

以下记录仅保存在当前浏览器：

- 资金与冻结资金变化；
- 商品可用与冻结库存变化；
- 仓库等级与容量变化；
- 工厂集群数量、统一状态、施工、计划和挂牌变化；
- 生产费用、原料消耗和产成品直接入仓；
- 商品与工厂数量成交；
- 下单和撤单。

本地日志不参与资产计算、工厂数量、仓库容量、生产、订单、产权或排名判断。

## 2. 本地存储

当前键：

```text
economy.local-activity.v3.<userId>
```

每个账号保存：

- 最后一次服务器权威状态的紧凑快照；
- 最多 480 条资产事件；
- 最多 240 条成交记录。

v1 日志记录可以保留读取，但旧实例快照不得参与新差异计算。首次进入 v2 时只建立新的集群快照，不生成虚假迁移事件。

## 3. v3 权威快照

本地快照只保存生成展示日志所需字段：

```ts
credits
frozenCredits
inventories
inventoryCapacity
warehouseLevel
facilityGroups
facilityConstruction
own orders
own facilityListings
products
market lastPrice
```

不保存：

```text
单座 facilities
factory instance id
internalGoods
internalCapacity
服务器 Cookie 或会话令牌
```

## 4. 生成方式

客户端比较连续两次服务器权威状态：

1. 资金和冻结资金；
2. 商品库存；
3. 仓库等级与容量；
4. 工厂集群数量、统一状态、统一计划和 `completedQuantity`；
5. 施工任务；
6. 自有商品订单；
7. 自有工厂数量挂牌。

首次加载只建立快照；相同状态不重复生成日志；幂等响应不会重复创建事件。浏览器存储错误不得阻断游戏操作。

## 5. 工厂集群事件

工厂变化按 `facilityTypeId` 比较，不存在实例 ID。

事件可以记录：

- 开始施工；
- 施工完成导致集群数量增加；
- 市场收购或出售导致数量变化；
- 统一启动或停止；
- 统一状态变化；
- 统一生产计划修改；
- 数量挂牌和撤销挂牌。

数量变化字段：

```ts
beforeCount: number;
afterCount: number;
countDelta: number;
```

本地日志不得创建、恢复或展示单座工厂实例。

## 6. 直接入仓生产事件

生产完成通过以下差异识别：

```text
FacilityGroup.completedQuantity 增量 > 0
```

事件记录：

- 工厂类型；
- 输出商品；
- 集群完成产量；
- 商品共享仓库变化；
- 资金变化；
- 描述“工厂集群产出已进入共享仓库”。

新事件不得写入 `internalGoodsDelta`，不得创建 `collectFacility` 动作。

旧 v1 浏览器历史中的 `collected` 或 `internalGoodsDelta` 可以继续作为旧文本记录读取，但不得调用服务器或改变权威状态。

## 7. 工厂数量成交

浏览器通过集群数量和资金差异生成数量成交：

- `type: facility`；
- `facilityTypeId`；
- 买入或卖出；
- 成交数量；
- 单座参考价格；
- 总额；
- 来源“工厂数量市场”。

运行中集群收购的新数量显示为数量增加，但正式 `pendingJoinCount` 始终来自服务器。

## 8. 仓库扩容事件

扩容事件记录资金扣款、扩容前后等级、扩容前后容量和容量增量：

```ts
warehouseChange?: {
  beforeLevel: number;
  afterLevel: number;
  beforeCapacity: number;
  afterCapacity: number;
  capacityDelta: number;
};
```

仓库实物库存、买单预占和剩余空间始终来自服务器状态，不从本地日志推导。

## 9. 服务器边界

SQLite 写入前必须删除：

```text
trades
ledger
assetEvents
```

日志不得参与资产估值、仓库占用、订单校验、工厂数量、生产结算或排名。

## 10. 清除、丢失与错误

清除日志时保留当前权威快照，避免把全部资产、工厂数量或完成量误记为新变化。

更换设备、清除网站数据、无痕模式或存储空间不足时日志允许丢失。日志丢失不得影响服务器状态。

读取或写入 localStorage 失败时继续接受服务器状态，不阻断下单、撤单、建设、统一启停、扩容、挂牌或购买。

## 11. 展示规则

- 市场页显示商品和工厂数量本地成交记录；
- 资金页显示资金、商品、仓库、工厂集群和生产变化；
- 新生产记录显示“产出入仓”；
- 概览页最多显示最近 6 条本地成交；
- 所有日志界面必须标注“本地记录”。

市场页本地成交记录必须遍历全部当前已加载的成交数据，不得使用 `slice`、分页、折叠、固定可见行数或内部垂直滚动区。记录面板不设置 `max-height`，随记录数量自然增长，并由市场页统一主滚动区域承载；窄屏表格只允许横向滚动列。

“面板随记录自然增长”不改变本地存储上限。当前浏览器仍最多保留最近 240 条成交记录，超过上限时按现有存储策略淘汰更早记录。

## 12. 测试与不可回退规则

必须验证：

- API 和 SQLite 不包含日志数组；
- v2 快照使用 `facilityGroups`，不使用单座实例；
- v1 快照不生成虚假迁移事件；
- 状态无变化时不生成事件；
- 工厂数量变化按类型生成；
- 生产通过完成量和库存差异生成；
- 数量成交记录数量与总额正确；
- 扩容记录包含等级和容量变化；
- localStorage 异常不阻断游戏；
- 市场页遍历全部当前 `localTrades`；
- 市场本地成交表格没有内部垂直高度限制、分页或截断；
- 本地成交存储上限仍为 240 条。

不得恢复服务器历史日志、单座工厂快照、领取动作、使用本地日志修改权威状态，或给市场本地成交记录恢复内部固定高度与记录截断。

未更新本设计和防回退检查的日志存储修改不应合并。

## 13. 权威成交明细迁移

本地日志 v3 不再根据订单剩余数量和市场 `lastPrice` 猜测成交价。服务器为玩家订单保存逐笔 `fills`；客户端只为相邻快照中新出现的 fill ID 创建成交记录，价格和总额直接使用 `fill.price` 与 `fill.total`。

同一订单在一轮请求中扫过多个价格时，每个撮合价格分别生成一条记录，不得合并为加权平均价或最终市场价。v1、v2 本地成交记录可能包含反推价格，因此升级到 v3 时保留资产事件但清空旧成交记录，并先建立新快照，避免继续展示错误价格或生成迁移假成交。

防回退检查必须禁止成交记录读取商品或工厂市场的 `lastPrice`。
