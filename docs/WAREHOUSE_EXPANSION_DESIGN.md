# Economy 共享仓库与扩容设计

> 状态：当前共享仓库容量、买单预占、生产空间与扩容规则基线
> 适用项目：`RIVERS0FT/Economy`
> 更新时间：2026-07-22
> 客户端状态版本：17
> 世界状态版本：15

生产与工厂集群以 `INDUSTRY_AND_PRODUCTION_DESIGN.md` 为准；页面归属以 `PAGE_CONTENT_AND_NAVIGATION_DESIGN.md` 为准。仓库商品卡密度、移动端四列、容器断点和商品插画主视觉尺寸只以本文第 7.1 节为准；其他文档中的重复旧密度或旧图标尺寸描述不得继续作为权威规则。

## 1. 核心规则

- 所有商品共用一个玩家仓库。
- 仓库等级、容量、占用和剩余空间均为服务器权威状态。
- 完整仓库管理入口只位于生产页面。
- 工厂产成品直接进入共享仓库。
- 工厂没有内部商品、内部容量或领取操作。
- 商品买单与商品拍卖最高出价预占容量；捆绑拍卖按资产包中全部商品数量之和预占，工厂买单和工厂拍卖不预占仓库。
- 仓库空间不足时，开启中的工厂进入异常，空间恢复后自动从新周期运行。
- 扩容费用不再回收，完整转入人口就业：基础人口 50%、技术人口 40%、专业人口 10%。
- 扩容费用必须由当前实际总容量线性计算。
- 仓库等级只能决定容量增量，不能直接决定扩容费用。
- 仓库内容只展示可用或冻结数量大于零的商品，不提供全部商品视图。
- 仓库没有玩家可见的最高等级，等级 12 后仍可继续扩容。

## 2. 无限等级、容量与费用

```text
初始等级：1
初始容量：500
当前等级 L 升到 L + 1 的容量增量：250 + 50 × (L - 1)
升级费用：150 + ceil((当前实际总容量 - 500) × 0.6)
```

费用整数实现使用等价比例 `3 / 5`，不依赖浮点累积：

```text
超出基础容量 = max(0, 当前实际总容量 - 500)
升级费用 = 150 + ceil(超出基础容量 × 3 / 5)
```

容量与费用职责必须分离：

```text
仓库等级 → 决定本次增加多少容量
当前实际总容量 → 决定本次扩容需要多少费用
```

标准等级容量与对应下一次费用：

| 等级 | 当前总容量 | 升到下一级增加 | 下一次扩容费用 |
|---:|---:|---:|---:|
| 1 | 500 | 250 | 150 |
| 2 | 750 | 300 | 300 |
| 3 | 1050 | 350 | 480 |
| 4 | 1400 | 400 | 690 |
| 5 | 1800 | 450 | 930 |
| 6 | 2250 | 500 | 1200 |

容量闭式公式：

```text
capacity(level) =
500
+ 250 × (level - 1)
+ 25 × (level - 1) × (level - 2)
```

`warehouseUpgradeCost` 只在计算超出安全整数范围时返回 `null`，不代表满级。界面不得显示 `/12`、满级、最高容量或“已达最高等级”。扩容必须在 SQLite 事务中完成并使用 `Idempotency-Key`。

服务器状态摘要显示的 `warehouseUpgradeCost` 与扩容事务实际扣除的费用必须调用同一个按实际容量计算的函数。客户端只展示服务器结果，不得重新计算扩容费用。

## 3. 旧存档迁移

迁移必须满足：

```text
新等级 = max(旧等级, 能覆盖旧容量的最小公式等级)
新容量 = max(旧容量, 新等级公式容量)
```

不得降低已有容量。下一次扩容必须在当前实际容量上增加该等级对应的完整增量，不能因为旧容量高于公式值而出现增加量为 0。下一次扩容费用也必须按迁移后的当前实际容量计算，不能按等级对应的标准容量替代。发生非安全整数、费用溢出或容量溢出时，服务器拒绝本次扩容，但不得将其表现为玩家达到最高等级。

本次定价与就业规则切换不补收、退还或追溯分配历史扩容费用，不修改已有等级、容量、库存和订单；模型 7 上线后的每次扩容费在同一事务转入人口待结算收入。

## 4. 权威占用

```text
实物库存 = Σ(各商品可用数量 + 冻结数量)
订单预占 = Σ(当前玩家 open/partial 商品买单剩余数量)
拍卖预占 = Σ(当前玩家作为最高出价者的进行中单项或捆绑拍卖中全部商品数量之和)
已用容量 = 实物库存 + 订单预占 + 拍卖预占
剩余容量 = max(0, 当前容量 - 已用容量)
```

不计入买单预占：

- 工厂买单与工厂竞拍；
- 商品卖单与卖方商品拍卖（商品已经计入冻结实物库存）；
- 已完成或已撤销订单；
- 其他玩家订单。

API 返回：

```ts
warehouseLevel: number;
warehouseUpgradeCost: number | null;
warehouseNextCapacity: number;
warehouseNextCapacityIncrease: number;
warehouseStoredQuantity: number;
warehouseReservedQuantity: number;
warehouseUsedCapacity: number;
warehouseAvailableCapacity: number;
```

客户端不得重新计算正式仓库占用或扩容费用。

## 5. 下单约束

提交商品买单或成为包含商品的拍卖最高出价者前，服务器必须确认剩余容量可以覆盖新增数量。单项商品拍卖按该商品数量预占；捆绑拍卖按资产包中全部商品数量之和预占。被超价、取消或流拍后立即释放，成交后等量转换为实物库存。

```text
最大商品买入数量 =
min(仓库剩余容量, floor(可用资金 / 限价))
```

部分成交后，已成交商品从预占转为实物库存；未成交部分继续预占。撤单释放剩余预占。

## 6. 生产约束与自动恢复

一个集群完整周期的净增长：

```text
集群周期产量 = 单座产量 × participatingCount
集群周期原料 = 单座原料 × participatingCount
netStoragePerCycle = max(0, 集群周期产量 - 集群周期原料)
```

加工集群可以利用同周期原料消耗释放的空间。服务器必须以整个集群为单位检查，不能只运行部分工厂。

仓库不足时：

```text
enabled = true
status = error
statusReason = warehouse_full
```

扩容、出售商品、撤销商品买单或其他释放空间的操作完成后，统一协调函数重新检查。条件满足时从新的完整周期自动恢复，不补算异常期间错过的周期。玩家关闭开关后不得自动恢复。

## 7. 生产页面布局

共享仓库是生产页顶部的单一面板，位于生产管理区之前。

共享仓库与建设新工厂、工厂集群属于生产页同一一级平面，统一使用 `.production-surface`。大于 `720px` 时四边内边距为 `16px`，不大于 `720px` 时为 `12px`；仓库分栏、商品数量、等级标签和响应式布局不得单独改变这一级外层内边距或标题左上锚点。

桌面内部采用：

```css
grid-template-columns: minmax(220px, 1fr) minmax(0, 3fr);
```

左侧只展示：

- 等级和容量状态；
- 已用容量、总容量和进度；
- 实物库存、买单预占、剩余容量；
- 下一等级容量、容量增量、费用和扩容按钮。

不得新增费用公式、线性定价说明或其他弱提示文案；正式费用由服务器字段直接展示。

右侧从服务器商品目录中过滤并展示有库存商品：

```text
显示条件 = available > 0 或 frozen > 0
```

“仓库内容”标题下不显示“X 种商品有库存”等商品数量说明。仓库不再区分“有库存”和“全部商品”，不得显示筛选按钮，也不得在空仓状态提供“查看全部商品”。

每张商品卡采用固定的插画主导结构，但不显示库存总量或估值：

1. 商品名称位于卡片第一行左侧，使用次级文字色并保持单行省略；名称参与正常网格布局，不得使用绝对定位和额外顶部占位制造空白；
2. 中央直接使用统一 `ProductIcon` 承载商品 PNG 插画；插画是卡片第一视觉主体，位于独立主视觉行并居中展示，正常数据模式隐藏 SVG 描边，节省数据模式回退为原 SVG；
3. 插画尺寸按仓库内容区宽度递增：小于 `560px` 时为 `46px`，`560px–759px` 为 `56px`，`760px–959px` 为 `64px`，不小于 `960px` 时为 `72px`；不得由后加载样式重新缩小；
4. 插画下方居中显示醒目的“可用 N”，沿用库存主值字号，窄容器使用辅助字号以避免四列横向溢出；
5. 最底部居中显示弱化的“冻结 N”，沿用辅助文字字号。

冻结数量为零时仍显示“冻结 0”，保证商品卡结构和高度一致。整张卡进入对应商品市场。不得把插画恢复为商品名称旁的 `1em` 小图标，也不得显示“库存 N”总量行、估值、买一价、最近成交价、单项容量、买单预占或直接出售表单。

### 7.1 商品网格密度与主视觉尺寸

仓库商品网格使用 `.warehouse-content` 的容器查询，而不是只按浏览器视口判断。正式断点为：

| 仓库内容区宽度 | 每行商品卡 | 卡片最小高度 | 插画尺寸 |
|---:|---:|---:|---:|
| `< 560px` | 4 列 | `104px` | `46px` |
| `560px–759px` | 4 列 | `116px` | `56px` |
| `760px–959px` | 5 列 | `124px` | `64px` |
| `≥ 960px` | 6 列 | `132px` | `72px` |

正式密度为 4／5／6 列，任何移动或窄容器都不得少于四列。商品卡使用“名称／插画／可用／冻结”的正常四行网格，插画行获得主要可用空间；常规内边距和网格间距均为 `8px`，内容区小于 `560px` 时使用 `6px` 网格间距和 `4px` 水平内边距。商品名称、可用数量和冻结数量均保持单行省略，不得造成横向溢出；名称的左上角锚点和插画的视觉中心必须在所有卡片中保持一致。

仓库卡片几何、插画尺寸和容器断点只允许定义在 `src/styles/warehouse-expansion.css`。`src/styles/product-artwork.css` 只负责商品 PNG 路径映射、背景渲染、描边隐藏和节省数据回退，不得再次声明 `.warehouse-product-card` 高度、网格行、内边距或 `.warehouse-product-card-icon` 尺寸。

低于页面 `960px` 时仓库管理区和内容区从左右分栏改为上下排列，但内容区仍根据自身容器宽度选择四列或五列。不得恢复移动两列／三列、按视口固定列数，也不得在 `420px` 或其他移动断点强制单列。

空仓时只显示：

- `仓库中暂无商品`；
- `生产或买入商品后，商品会显示在这里。`

## 8. 页面职责

- `WarehouseUpgradeCard` 只能由 `ProductionPage` 渲染。
- 状态栏可以显示只读仓库剩余。
- 市场页可以显示仓库剩余和买单预占，但不得扩容。
- 资产页负责资产汇总和本地资产变化，不得重复显示逐商品库存与估值卡。
- 设置页不得显示仓库经营摘要。

## 9. 防回退

不得恢复：

- 固定最高等级或固定最高容量；
- 每级固定增加 250 的旧公式；
- 按仓库等级平方计算扩容费用；
- 使用等级标准容量代替玩家当前实际总容量计算费用；
- 客户端自行计算扩容费用，或让显示费用与实际扣费使用不同公式；
- 在仓库面板新增费用公式、线性定价说明或其他弱提示；
- `warehouseMaxLevel` 客户端字段；
- 每个商品独立容量；
- 工厂内部仓库或领取按钮；
- 客户端容量公式作为权威结果；
- 设置页或市场页扩容入口；
- 六个重复的大型仓库指标卡；
- 仓库满后要求玩家手动重新开启工厂；
- 商品估值或价格进入仓库商品卡；
- 仓库“有库存／全部商品”筛选、全部商品视图或“查看全部商品”空状态按钮；
- “X 种商品有库存”说明、独立“库存 N”总量行或把可用与冻结重新合并到同一行；
- 在任何移动或窄容器把仓库商品网格降为一列、两列或三列；
- 恢复固定桌面四列、普通桌面三列或只由视口宽度决定列数；
- 恢复 `300px`／`560px` 的旧增列断点、删除 4／5／6 列容器查询、把窄容器商品卡最小高度降到 `104px` 以下，或恢复窄容器 `12px` 水平内边距；
- 把商品名称移出第一行左侧、恢复绝对定位和顶部魔法占位、把商品插画恢复为名称旁的小图标，或取消居中插画主体结构；
- 把仓库插画尺寸恢复为 `54px`／`38px`、`44px`／`30px` 或其他由后加载样式覆盖的平行尺寸；
- 让 `product-artwork.css` 重新定义仓库卡片高度、网格行、内边距或插画尺寸；
- 让共享仓库在同一生产页断点使用不同于建设卡和工厂集群的一级外层内边距；
- 没有商品 SVG 回退或商品 PNG 插画的纯文字仓库卡；
- 在资产页恢复逐商品库存与估值卡。
