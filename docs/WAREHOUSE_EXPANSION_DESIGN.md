# Economy 共享仓库与扩容设计

> 状态：当前共享仓库容量、商品内容、工厂生产约束与升级规则基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-12  
> 客户端状态版本：`version: 10`  
> 世界状态版本：`version: 6`

页面归属以 `docs/PAGE_CONTENT_AND_NAVIGATION_DESIGN.md` 为准，工厂集群状态与自动恢复以 `docs/FACILITY_GROUP_AND_MARKET_V3_DESIGN.md` 为准。

## 1. 核心规则

- 所有商品共用一个玩家仓库容量。
- 仓库等级、容量、占用和剩余空间属于服务器权威状态。
- 共享仓库完整管理入口只位于生产页面。
- 所有工厂集群产成品完成后直接进入共享仓库。
- 工厂不拥有内部商品、内部容量或领取操作。
- 集群完整周期会造成净库存增加时，必须先检查仓库空间。
- 扩容费用由系统回收并计入 `stats.systemSinks`。
- 仓库内容卡只展示现有单项库存，不重新计算正式仓库占用。

## 2. 等级、容量与费用

```text
初始等级：1
初始容量：500
每级增加：250
最高等级：12
最高容量：3250
capacity(level) = 500 + (level - 1) × 250
cost(level) = 150 × level²
```

最高等级的 `warehouseUpgradeCost` 为 `null`。扩容必须在 SQLite `BEGIN IMMEDIATE` 事务中完成并使用 `Idempotency-Key`。

## 3. 权威仓库占用

```text
实物库存 = Σ(各商品可用数量 + 各商品冻结数量)
买单预占 = Σ(当前玩家 open/partial 商品买单剩余数量)
已用容量 = 实物库存 + 买单预占
剩余容量 = max(0, 当前容量 - 已用容量)
```

不计入买单预占：工厂买单、已成交订单、已撤销订单、卖单和其他玩家订单。

API 返回：

```ts
warehouseStoredQuantity: number;
warehouseReservedQuantity: number;
warehouseUsedCapacity: number;
warehouseAvailableCapacity: number;
```

客户端不得重新计算正式仓库占用。

## 4. 工厂集群直接入仓与自动恢复

一个集群完整周期的仓库净增长：

```text
集群周期产量 = 单座产量 × participatingCount
集群周期原料 = 单座原料 × participatingCount
netStoragePerCycle = max(0, 集群周期产量 - 集群周期原料)
```

- 无原料集群必须为完整集群产出预留空间。
- 加工集群可以利用同周期原料消耗释放的空间。
- 产成品直接增加到输出商品的 `available` 库存。
- 同类型所有参与数量在一个周期中整体结算，不得部分生产。
- 买单预占会减少工厂集群可用生产空间。

共享仓库无法容纳下一完整周期净增长时：

```text
enabled = true
status = error
statusReason = warehouse_full
```

释放库存或扩容后，统一协调函数重新检查生产条件。条件满足时从新的完整周期自动恢复，不补算异常期间错过的周期。玩家主动关闭开关后不得自动恢复。

## 5. 新数量下一周期加入

施工完成或市场收购发生在集群运行中时，新数量进入 `pendingJoinCount`。

当前周期仍按旧 `participatingCount` 检查仓库和结算。周期完成后待加入数量并入下一周期，不得重置当前周期或为新增数量生成部分周期产出。

## 6. 共享仓库页面布局

共享仓库仍是一个顶层 `Panel`，位于生产页标题之后、建设新工厂之前。

桌面端内部采用 `1:3` 左右布局：

```css
grid-template-columns: minmax(220px, 1fr) minmax(0, 3fr);
```

### 左侧：容量与升级

左侧只展示：

- 等级和容量超限状态；
- 已用容量、总容量和进度条；
- 当前容量、实物库存、买单预占、剩余容量；
- 下一等级容量、增加容量、升级费用；
- 扩容按钮和必要的容量警告。

不得恢复六个重复的大型指标卡。

### 右侧：仓库内容

右侧使用商品卡片网格，数据来源为服务器商品目录和现有 `inventories`：

```ts
game.products
game.inventories[productId].available
game.inventories[productId].frozen
```

默认筛选“有库存”，定义为：

```text
可用库存 > 0 或 冻结库存 > 0
```

“全部商品”显示服务器目录中的全部商品，包括零库存商品。

## 7. 仓库商品卡

每张卡只显示：

1. 产品名称；
2. 可用库存；
3. 冻结库存。

不得显示：

- 商品估值；
- 买一价、最近成交价；
- 买单预占；
- 单项容量占用；
- 直接出售表单。

整张卡是可访问的市场导航按钮，点击后选择对应商品并打开统一市场订单簿。卡片必须使用服务器商品目录动态渲染，不得写死商品名称。

桌面宽屏四列、普通桌面三列、平板和手机两列，极窄屏幕才降为单列。

## 8. 移动端

宽度低于 `960px` 时，仓库内部改为上下布局：

1. 容量与升级；
2. 仓库商品内容。

不得依赖横向滚动查看商品。升级按钮保持整行可点击，商品卡在常规手机宽度下保持两列。

## 9. 页面职责

- `WarehouseUpgradeCard` 只能由 `ProductionPage` 渲染。
- 状态栏可以显示只读仓库剩余摘要。
- 市场页可以解释买单预占，但不得扩容。
- 资金页负责商品估值，不得渲染仓库扩容卡。
- 设置页不得显示仓库摘要或操作。

## 10. 数据与版本

本次商品卡使用已有客户端状态，不新增服务器字段，不修改 SQLite 世界数据结构：

```text
客户端状态版本保持 10
世界状态版本保持 6
```

仓库总量继续使用服务器权威字段；商品卡不参与正式容量计算。

## 11. 测试与不可回退规则

必须覆盖：

1. 仓库等级、费用和幂等扩容；
2. 实物库存与商品买单预占；
3. 集群直接入仓；
4. 加工集群利用原料释放空间；
5. 仓库不足时进入异常且不扣费；
6. 释放空间或扩容后自动恢复且不补算；
7. 共享仓库桌面布局为 `1fr 3fr`；
8. 左侧仅承担容量和升级信息；
9. 右侧商品使用卡片而不是表格；
10. 商品卡只显示名称、可用库存和冻结库存；
11. 商品卡点击进入对应市场；
12. 默认隐藏零库存商品，全部模式显示零库存商品；
13. 手机端商品卡默认双列；
14. `WarehouseUpgradeCard` 仍只由生产页渲染。

不得恢复内部仓库、领取按钮、客户端容量公式、商品估值进入仓库卡、重复大型指标卡、设置页仓库入口或仓库满后要求手动重启的旧规则。
