# Economy 统一资产订单簿设计

> 状态：商品和工厂交易、成交价格、市场需求订单与估值的当前权威设计  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-19  
> 客户端状态版本：15  
> 世界状态版本：13

## 1. 统一资产模型

商品和工厂都使用 `AssetOrder`：

```ts
interface AssetOrder {
  id: string;
  assetKind: 'commodity' | 'facility';
  assetId: string;
  productId?: string;
  facilityTypeId?: string;
  side: 'buy' | 'sell';
  ownerType: 'player' | 'population';
  ownerId?: number;
  ownerName: string;
  demandGroupId?: 'food' | 'household';
  demandTier?: 'direct' | 'derived-liquidity';
  demandCycleId?: number;
  price: number;
  quantity: number;
  remaining: number;
  fills?: OrderFill[];
  status: 'open' | 'partial' | 'filled' | 'cancelled';
  createdAt: number;
}
```

`ownerType: 'population'` 是市场需求订单在持久化、撮合、手续费和既有统计中的**兼容标识**，不表示继续使用旧人口需求模型。业务名称固定为“食品市场需求”或“家庭消费市场需求”，规则以 `PRODUCT_AND_GAMEPLAY_DESIGN.md` 的市场需求模型为准。

当前市场不设置商品／工厂二级页面切换，不使用工厂固定价格挂牌卡。商品与工厂资产标签在同一横向目录中连续展示。

商品资产标签使用对应商品的 `ProductIcon`。工厂资产标签使用独立厂房 SVG `FactoryIcon`，不得使用机械商品的齿轮图标，也不得恢复 `⚙` 字符。图标不改变资产 ID、撮合或估值规则。

`FacilityListing` 只允许作为历史迁移兼容空结构存在，不得重新成为业务模型。

## 2. 市场需求订单

市场需求订单和玩家商品订单进入同一商品订单簿，遵守同一价格优先、同价时间优先和 maker price 规则。

市场需求订单：

- 只允许 `assetKind = 'commodity'` 和 `side = 'buy'`；
- `demandGroupId` 只允许 `food` 或 `household`；
- `demandTier` 只允许 `direct` 或 `derived-liquidity`；
- `demandCycleId` 标记唯一需求周期；
- 订单价格来自服务器市场需求参考价；
- 每组同一周期的直接需求和派生流动性已提交金额之和不得超过该组总预算；
- 新周期开始时取消上一周期尚未完成的同组市场需求订单；
- 未成交预算不结转；
- 市场需求不创建卖单。

市场需求订单内部字段不得暴露给普通玩家。公开订单视图必须删除：

- `ownerType`；
- `ownerId`；
- `ownerName`；
- `demandGroupId`；
- `demandTier`；
- `demandCycleId`；
- 对手订单 ID 和非本人逐笔成交信息。

## 3. 下单与冻结

所有订单由服务器校验：

- 价格和数量必须为有效整数范围；
- 单笔数量为 1～10,000；
- 每名玩家最多 10 笔未完成订单；
- 玩家买单冻结对应资金；
- 商品卖单冻结对应可用库存；
- 工厂卖单冻结对应工厂数量；
- 商品与工厂单项或捆绑拍卖也冻结卖方资产；
- 拍卖冻结与订单簿冻结按资产 ID 合并计算；
- 撤单释放未成交部分资产；
- 部分成交只保留剩余数量对应的冻结资产；
- 同一玩家不得自成交；
- 所有写操作使用 `Idempotency-Key`。

商品买单和商品拍卖最高出价都会预占共享仓库容量；捆绑拍卖按资产包中全部商品数量之和预占，并与商品买单合并计算。工厂买单和工厂竞拍不占仓库。

市场需求订单不冻结玩家资金；其成交总额属于服务器市场需求发行，并受需求组周期预算约束。

## 4. 工厂运行中卖出

运行中的工厂允许进入卖单。服务器提交工厂卖单时：

1. 结算已经完成的完整周期；
2. 从待加入数量和当前参与数量中冻结卖出数量；
3. 立即降低当前或下一周期参与数量；
4. 冻结数量停止产生商品和运营成本；
5. 保持剩余未冻结工厂的运行意图和周期语义。

撤销工厂卖单时，未成交数量解冻：

- 集群停止时立即恢复可用数量；
- 集群运行时进入 `pendingJoinCount`，从下一周期加入；
- 不重置当前周期，不生成部分周期产出。

## 5. 拍卖与订单簿隔离

拍卖成交不属于订单簿成交。商品、工厂或捆绑拍卖的起拍价、当前最高价和成交总价不得进入：

- `AssetOrder`；
- 订单簿买卖盘；
- `lastPrice`；
- 价格历史；
- 主动方向成交量；
- 市场需求真实成交 VWAP；
- 最高非本人买价估值。

混合资产包总价不得拆分为虚构单价。订单簿冻结与拍卖冻结只在可用资产校验和工厂参与数量上合并；撮合、价格发现和成交统计保持独立。

拍卖取消或流拍释放对应冻结，成交时原子转移整个资产包，不创建伪造订单或成交明细。托管记录不得作为第二份资产余额重复计入总资产。

## 6. 撮合

撮合固定遵循：

1. 资产种类和资产 ID 必须相同；
2. 价格优先；
3. 同价时间优先；
4. 最高买价大于或等于最低卖价时成交；
5. 允许部分成交；
6. 未成交部分继续保留；
7. 禁止玩家自成交；
8. 成交、冻结、双方资产和统计在同一事务中更新。

商品订单和工厂订单不得互相撮合；不同商品或不同工厂类型也不得互相撮合。市场需求订单只能与同商品的玩家卖单成交。

## 7. maker price 与逐笔成交

每次成交使用撮合前已经位于订单簿中的对手方挂单价格，即 maker price。

服务器在相关玩家订单 `fills` 中逐笔保存：

```ts
interface OrderFill {
  id: string;
  quantity: number;
  price: number;
  total: number;
  fee: number;
  netTotal: number;
  counterparty: string;
  createdAt: number;
  makerOrderId: string;
  takerOrderId: string;
  liquidity: 'maker' | 'taker';
}
```

同一订单扫过多个价格时必须生成多条独立成交明细，不得合并为平均价。浏览器不得使用市场 `lastPrice` 反推玩家成交价，也不得以最终市场价覆盖真实逐笔价格。

订单簿和价格曲线属于市场状态；玩家成交记录属于订单级权威明细。两者不得互相替代。

## 8. 玩家卖出手续费

统一订单簿只向玩家卖方收取手续费，买方仍按 `quantity × maker price` 支付完整成交总额。商品卖单、工厂卖单以及玩家卖给市场需求的成交使用同一规则；拍卖不收费。

```text
累计应收手续费 = max(1, ceil(累计成交总额 × 1%))
本次 fill 手续费 = 累计应收手续费 - 此前累计已收手续费
本次卖方实收 = 本次成交总额 - 本次 fill 手续费
```

累计成交总额为 0 时手续费为 0。手续费按单张卖单累计，不得对每个 fill 重复应用最低值。

`fill.total` 始终保持成交总额；卖方 fill 的 `fee` 表示本笔增量手续费，`netTotal` 表示本笔实际到账；买方 fill 的 `fee` 为 0，`netTotal` 等于 `total`。

市场需求成交继续计入既有 `populationIssued` 统计字段，以保持存档兼容；该字段在业务含义上表示市场需求发行总额。手续费进入卖方 `stats.systemSinks`，净货币变化等于市场需求发行总额减手续费。

上线迁移不得追收既有成交。服务器必须在同一成交事务完成扣费、fill 标记和系统回收；幂等重试不得重复扣费。

## 9. 行情主动方向

每笔成交必须保存吃单方方向：

- 主动买单吃掉已有卖单时记为主动买入；
- 主动卖单吃掉已有买单时记为主动卖出；
- 不得把成交双方数量直接相减，也不得按价格涨跌猜测主动方向。

概览页与市场页共用最近 24 小时、6 分钟行情分段。分段收盘价取最后一笔成交价，总成交量为全部成交数量之和，净主动量为主动买入量减主动卖出量。

旧成交没有主动方向字段时必须保留为中性成交量，新成交逐步替换该窗口，禁止伪造迁移方向。

市场需求价格模型只能使用真实成交方向和成交量；未成交市场需求订单不得作为真实成交。

## 10. 单列订单簿

每个选中资产固定显示：

```text
卖盘标题
价格最低的 5 笔有效卖单
无文字中性分隔线
买盘标题
价格最高的 5 笔有效买单
```

每行只显示：

```text
方向｜价格｜剩余数量
```

规则：

- 不显示玩家身份或市场需求身份；
- 不按价格档位聚合；
- 买卖盘各最多 5 笔；
- 订单簿内部不显示买一、卖一、价差、最近成交或指标卡式成交线；
- 桌面和移动端都按内容自然高度显示；
- 订单不足时不生成虚假空行。

## 11. 下单界面与默认价格

玩家界面统一将订单输入字段称为“价格”，不得显示“限价”字段、标题或表头。底层订单仍是限价订单模型，服务器撮合、冻结和校验规则不变。

默认价格可以读取当前资产的参考价或订单簿，但不得自动提交订单，也不得让未成交高价挂单直接改写市场需求参考价。

## 12. 迁移与防回退

升级到市场需求模型时：

- 取消旧未完成的人口需求订单；
- 保留玩家订单及其时间优先级；
- 新市场需求订单在当前周期按新模型重建；
- 世界版本继续保持 13，模型迁移由 `marketDemand.modelVersion = 1` 保证只执行一次；
- 不修改玩家资金、库存、工厂、冻结资产、拍卖托管和历史成交。

不得恢复固定价格工厂挂牌、系统自动卖单、企业采购、公开市场需求组、公开对手身份、按价格档位聚合、按最终市场价覆盖真实 fill、将拍卖成交写入行情、或让市场需求订单绕过统一撮合与卖方手续费。
