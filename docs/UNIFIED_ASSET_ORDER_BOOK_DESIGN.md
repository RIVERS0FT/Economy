# Economy 统一资产订单簿设计

> 状态：商品和工厂交易、成交价格与估值的当前权威设计  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-17
> 客户端状态版本：15
> 世界状态版本：12

## 1. 统一资产模型

商品和工厂都使用 `AssetOrder`：

```ts
interface AssetOrder {
  id: string;
  assetKind: 'commodity' | 'facility';
  assetId: string;
  productId?: string;
  facilityTypeId?: string;
  side: 'buy' | 'sell';
  ownerType: 'player' | 'population';
  ownerId?: number;
  ownerName: string;
  demandGroupId?: 'food' | 'household';
  demandTier?: 'raw' | 'intermediate' | 'final';
  demandCycleId?: number;
  price: number;
  quantity: number;
  remaining: number;
  fills?: OrderFill[];
  status: 'open' | 'partial' | 'filled' | 'cancelled';
  createdAt: number;
}
```

当前市场不设置商品／工厂二级页面切换，不使用工厂固定价格挂牌卡。商品与工厂资产标签在同一横向目录中连续展示。

商品资产标签使用对应商品的 `ProductIcon`。工厂资产标签使用独立厂房 SVG `FactoryIcon`，不得使用机械商品的齿轮图标，也不得恢复 `⚙` 字符。图标只承担视觉语义，不改变资产 ID、撮合或估值规则。

`FacilityListing` 只允许作为历史迁移兼容空结构存在，不得重新成为业务模型。

人口买单使用 `demandGroupId`、`demandTier` 和 `demandCycleId` 标记需求组、产业层级与周期，但仍遵守同一价格优先、同价时间优先规则。预算与双向价格传导以 `PRODUCT_AND_GAMEPLAY_DESIGN.md` 为唯一权威来源。

## 2. 下单与冻结

所有订单均由服务器校验：

- 价格和数量必须为有效整数范围。
- 单笔数量为 1～10,000。
- 每名玩家最多 10 笔未完成订单。
- 买单冻结对应资金。
- 商品卖单冻结对应可用库存。
- 工厂卖单冻结对应工厂数量。
- 撤单释放未成交部分资产。
- 部分成交只保留剩余数量对应的冻结资产。
- 同一玩家不得自成交。
- 所有写操作使用 `Idempotency-Key`。

商品买单还会预占共享仓库容量；工厂买单不占仓库。

## 3. 工厂运行中卖出

运行中的工厂允许进入卖单。

服务器提交工厂卖单时：

1. 结算已经完成的完整周期。
2. 从待加入数量和当前参与数量中冻结卖出数量。
3. 立即降低当前或下一周期参与数量。
4. 冻结数量停止产生商品和运营成本。
5. 保持剩余未冻结工厂的运行意图和周期语义。

撤销工厂卖单时，未成交数量解冻：

- 集群停止时立即恢复可用数量；
- 集群运行时进入 `pendingJoinCount`，从下一周期加入；
- 不重置当前周期，不生成部分周期产出。

## 4. 撮合

撮合固定遵循：

1. 资产种类和资产 ID 必须相同。
2. 价格优先。
3. 同价时间优先。
4. 最高买价大于或等于最低卖价时成交。
5. 允许部分成交。
6. 未成交部分继续保留。
7. 禁止自成交。
8. 成交、冻结、双方资产和统计在同一事务中更新。

商品订单和工厂订单不得互相撮合；不同商品或不同工厂类型也不得互相撮合。

## 5. maker price 与逐笔成交

每次成交使用撮合前已经位于订单簿中的对手方挂单价格，即 maker price。

服务器在相关订单 `fills` 中逐笔保存：

```ts
interface OrderFill {
  id: string;
  quantity: number;
  price: number;
  total: number;
  counterparty: string;
  createdAt: number;
  makerOrderId: string;
  takerOrderId: string;
  liquidity: 'maker' | 'taker';
}
```

同一订单扫过多个价格时必须生成多条独立成交明细，不得合并为平均价。浏览器不得使用市场 `lastPrice` 反推玩家成交价，也不得以最终市场价覆盖真实逐笔价格。

订单簿和价格曲线属于市场状态；玩家成交记录属于订单级权威明细。两者不得互相替代。

### 5.1 行情主动方向

商品和工厂每笔成交除价格、数量和时间外，还必须保存吃单方（taker／incoming order）的买卖方向。主动买单吃掉已有卖单时记为主动买入，主动卖单吃掉已有买单时记为主动卖出；不得把成交双方的买卖数量直接相减，也不得按价格涨跌猜测主动方向。

概览页与市场页共用同一最近 24h、6 分钟行情分段。分段收盘价取最后一笔成交价，总成交量为全部成交数量之和，净主动量为主动买入量减主动卖出量。成交量柱高表示总成交量，颜色只表达净主动方向：净买入使用成功色，净卖出使用危险色，均衡或旧历史方向未知使用中性色。旧成交没有主动方向字段时必须保留为中性成交量，新成交逐步替换该窗口，禁止伪造迁移方向。

## 6. 单列订单簿

每个选中资产固定显示：

```text
卖盘标题
价格最低的 5 笔有效卖单
无文字中性分隔线
价格最高的 5 笔有效买单
买盘标题
```

每行只显示：

```text
方向｜价格｜剩余数量
```

规则：

- 不显示玩家身份。
- 不按价格档位聚合。
- 买卖盘各最多 5 笔。
- 订单簿内部不显示买一、卖一、价差、最近成交或指标卡式成交线。
- 桌面订单簿拉伸到所在网格行高度；订单不足时不生成虚假空行。
- 移动端按内容自然增长。

## 7. 下单界面与默认价格

玩家界面统一将订单输入字段称为“价格”，不得显示“限价”字段、标题或表头。底层订单仍是限价订单模型，服务器撮合、冻结和校验规则不变。

界面顺序固定为：

```text
方向
价格
数量
1/4 仓｜1/2 仓｜全仓
订单总额
提交订单
当前资产未完成订单与撤单
```

默认价格只从客户端当前已经加载的 `game.orders` 本地快照计算，不得为填写默认价格调用 `getGameState`、`refresh`、新增 API 或发起任何额外网络请求。

只考虑同资产种类、同资产 ID、`open` 或 `partial`、剩余数量大于零且价格为正整数的订单：

```text
买入：最低卖出价
      → 没有卖单时使用最高买入价
      → 买卖盘均为空时使用 1

卖出：最高买入价
      → 没有买单时使用最低卖出价
      → 买卖盘均为空时使用 1
```

重新填写默认价格的时机：

- 首次进入市场页；
- 从其他页面重新进入市场页；
- 主动切换商品或工厂资产；
- 主动切换买入／卖出方向。

自动刷新、下单响应、成交、撤单或其他权威状态同步只更新本地订单快照，不得直接覆盖当前价格输入。玩家下一次重新进入市场、切换资产或切换方向时，使用届时最新的本地订单快照重新计算。

在市场页内再次点击当前已经选中的资产或当前方向时，不重新填写价格，保留玩家手动输入。默认价格不得使用 `lastPrice`、商品 `basePrice`、工厂 `systemValue` 或正式估值价格作为回退。

商品买入快捷数量：

```text
maxBuy = min(warehouseAvailableCapacity, floor(credits / price))
```

商品卖出快捷数量使用当前可用库存。冻结商品不能出售。

工厂买卖快捷数量使用当前可用资金或未冻结工厂数量，不使用共享仓库容量。

## 8. 订单来源

商品订单只允许玩家或人口需求作为所有者：

- 玩家可以提交商品买单和卖单。
- 人口需求只提交买单，且名称只能是“饮食需求”或“家庭用品需求”。
- 不提供系统流动性买单或卖单，不生成企业采购或普通人口需求。
- 商品订单簿允许单侧为空或完全为空；服务器不得为了填充盘口制造订单、成交量或最近成交价。
- 迁移必须删除旧 `ownerType = market` 商品订单和旧企业采购／普通人口需求订单，同时保留玩家订单及其冻结资产。
- 新世界首次状态处理与需求模型升级迁移必须在同一事务立即生成当前周期人口买单；不得删除旧订单后把订单簿留空到下一周期，也不得因补发重复使用当期预算。

工厂订单仍只能由玩家提交；服务器周期、迁移、刷新和进入游戏都不得创建工厂订单。

## 9. 估值与排行榜

每项商品和工厂的估值价格使用：

```text
该资产最高非本人有效买入价
```

- 排除资产所有者自己的买单。
- 没有非本人有效买单时估值为 0。
- 商品估值数量包含可用和冻结库存。
- 工厂估值数量包含玩家拥有的全部数量，包括卖单冻结数量。
- 总资产、资产摘要和排行榜由服务器统一计算。
- 客户端不得使用系统参考价、玩家自己的买单或固定挂牌价替代正式估值。

## 10. 本地成交

浏览器本地成交记录只比较相邻权威状态中新出现的 fill ID，并使用 `fill.price`、`fill.quantity`、`fill.total` 和 `fill.counterparty`。

本地日志 v3 升级时清空由旧 `lastPrice` 推导的成交记录；v4 将生产记录升级为多输入结构并建立包含工厂目录的新快照。

## 11. 防回退

不得恢复：

- 商品和工厂二级市场切换；
- 工厂固定价格挂牌或即时购买卡；
- 工厂必须停止后才能挂卖单；
- 按价格档位聚合订单簿；
- 双列买卖盘；
- 使用 `lastPrice` 反推成交价格；
- 使用系统参考价计算正式总资产；
- 客户端撮合或客户端解冻资产；
- 工厂资产标签使用机械齿轮图标、`⚙` 字符或与机械商品相同的 SVG 路径；
- 在玩家界面恢复“限价”文案；
- 使用最近成交价、基础价、系统价值或估值价填写默认价格；
- 自动刷新覆盖玩家已输入的价格；
- 为默认价格计算新增服务器请求；
- 从其他页面重新进入市场时继续保留旧价格而不按最新本地订单簿重算；
- 使用商品最近成交价生成系统商品买卖订单；
- 系统商品订单越过当前最佳买卖价或制造立即成交；
- 创建任何系统工厂买单、系统工厂卖单或固定价格工厂供给；
- 保留旧世界中的系统工厂订单。

## 普通玩家订单与成交匿名化

服务器内部订单继续保存真实所有者、需求组和 maker/taker 关系，用于撮合、结算、人口货币发行统计和管理员审计。普通玩家状态必须经过集中式公开订单序列化：本人订单仅增加 `isOwn: true` 并返回由 `id / quantity / price / total / createdAt` 组成的匿名 fills；其他订单返回 `isOwn: false` 且不返回 fills。

普通玩家状态中的所有订单都不得返回真实 `ownerId`、`ownerName`、`ownerType`、`demandGroupId`、`demandTier` 或 `demandCycleId`。公开 fill 不得返回 `counterparty`、`makerOrderId`、`takerOrderId` 或 `liquidity`。人口需求订单与玩家订单在普通玩家 JSON 中必须使用相同结构，不能通过字段差异判断来源。

客户端通过 `isOwn` 识别本人订单，并只显示订单部分成交、全部成交、成交数量、价格、总额和时间。隐藏界面列不能替代 API 脱敏。
