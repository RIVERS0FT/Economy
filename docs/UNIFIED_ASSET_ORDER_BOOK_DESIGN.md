# Economy 统一资产订单簿设计

> 状态：商品和工厂交易、成交价格、市场需求订单与估值的当前权威设计
> 适用项目：`RIVERS0FT/Economy`
> 更新时间：2026-07-23
> 客户端状态版本：17
> 世界状态版本：15
> 市场需求模型版本：9

模型 9 的稳定需求补充在订单创建前进入人口真实钱包；直接与派生订单仍必须逐笔冻结真实资金，成交、价差退款和撤单释放规则不变，订单簿不得把补充金额当作成交发行。

> 人口政策调控只改变后续人口稳定需求预算、钱包缺口补充和下一完整生产周期的人口工资系数，不得取消、改价或重建当前人口订单，也不得改变玩家订单、储备订单、生产成本或真实成交统计。市场需求模型保持版本 9，人口经济内部版本为 3。


模型 9 保持 `food` / `household` 两个内部需求组和既有订单字段，但 `household` 的产品名称升级为社会消费市场。全部 31 种商品均可形成 `direct` 买单；成交后商品被终端消耗，不进入人口库存。模型升级必须先释放旧 `direct` / `derived-liquidity` 订单占用的人口冻结资金，再删除旧系统订单；市场储备的真实资金、库存和累计交易数据继续保留。

## 1. 统一资产模型

商品和工厂都使用 `AssetOrder`：

```ts
interface AssetOrder {
  id: string;
  assetKind: 'commodity' | 'facility';
  assetId: string;
  productId?: string;
  facilityTypeId?: string;
  side: 'buy' | 'sell';
  ownerType: 'player' | 'population';
  ownerId?: number;
  ownerName: string;
  demandGroupId?: 'food' | 'household';
  demandTier?: 'direct' | 'derived-liquidity' | 'liquidity-buy' | 'liquidity-sell';
  demandCycleId?: number;
  populationModelId?: 'basic' | 'skilled' | 'professional';
  fundingPool?: 'direct' | 'derived';
  price: number;
  quantity: number;
  remaining: number;
  fills?: OrderFill[];
  status: 'open' | 'partial' | 'filled' | 'cancelled';
  createdAt: number;
  lastFilledAt?: number;
}
```

`ownerType: 'population'` 是消费需求与市场储备商品订单在持久化、撮合、手续费和既有统计中的兼容标识，不表示继续使用旧人口需求模型。两类系统订单都沿用需求组兼容名称“食品市场需求”或“家庭消费市场需求”，必须通过 `demandTier` 区分消费与储备角色；预算、替代、互补、储备资产和价格传导以 `PRODUCT_AND_GAMEPLAY_DESIGN.md` 为唯一权威来源。

当前市场不设置商品／工厂二级页面切换，不使用工厂固定价格挂牌卡。商品与工厂资产标签在同一横向目录中连续展示。

商品资产标签使用对应商品的 `ProductIcon`。工厂资产标签使用独立厂房 SVG `FactoryIcon`，不得使用机械商品的齿轮图标，也不得恢复 `⚙` 字符。图标只承担视觉语义，不改变资产 ID、撮合或估值规则。

横向资产目录中的商品和工厂价格都必须读取对应市场的 `lastTradePrice` 并标注为“最近成交价”；从未发生真实订单簿成交时显示“—”，无障碍名称使用“暂无成交”。`lastPrice` 只可继续用于行情曲线的内部回退，不得作为资产标签成交价、工厂利润或正式估值。

`FacilityListing` 只允许作为历史迁移兼容空结构存在，不得重新成为业务模型。

消费需求与市场储备订单使用 `demandGroupId`、`demandTier` 和 `demandCycleId` 标记组别、角色与周期，但仍遵守同一价格优先、同价时间优先规则。直接需求和派生流动性按 70%／30% 共享消费预算；库存不得扩大消费预算，派生需求不得沿产业链重复发行。市场储备独立使用真实资金和库存挂出双边订单，每周期撤销重挂，不参与消费买单的 50% 保留和 3% 阶梯提价。

### 1.1 人口资金字段隐私

`populationModelId` 与 `fundingPool` 只用于服务器持久化、冻结和结算。集中式公开订单序列化必须删除这两个字段，并继续删除所有订单来源、需求组和角色字段；普通玩家、本地成交记录和市场页面不得判断订单属于哪类人口。

## 2. 下单与冻结

所有订单均由服务器校验：

- 价格和数量必须为有效整数范围。
- 单笔数量为 1～10,000。
- 每名玩家最多 10 笔未完成订单。
- 玩家买单冻结对应资金。
- 消费需求订单不冻结玩家资金，但必须从所属 `populationModelId` 的真实人口可用余额转入冻结资金；直接与派生资金池通过 `fundingPool` 标记。成交退回限价与 maker price 的差额，撤单或缩量释放未成交冻结资金。
- 市场储备买单创建时从储备可用资金转入冻结资金，成交按 maker price 支出并退回价差，撤单释放未成交部分。
- 市场储备卖单创建时从储备可用库存转入冻结库存，成交扣除真实冻结库存并增加储备资金，撤单释放未成交部分。
- 商品卖单冻结对应可用库存。
- 工厂卖单冻结对应工厂数量。
- 商品与工厂单项或捆绑拍卖也冻结卖方资产；拍卖冻结与订单簿冻结必须按资产 ID 合并计算，任何一方都只能使用扣除另一方冻结后的可用数量。
- 撤单释放未成交部分资产。
- 部分成交只保留剩余数量对应的冻结资产。
- 同一玩家不得自成交。
- 玩家提交商品或工厂订单时，若新买价不低于本人同资产最低未完成卖价，或新卖价不高于本人同资产最高未完成买价，服务器必须在冻结任何资金、商品或工厂之前拒绝新订单，并提示先撤销原反向订单。
- 相等价格也属于交叉；不得只在撮合阶段跳过本人订单后，让两张交叉订单同时留在公开订单簿。
- 所有写操作使用 `Idempotency-Key`。

商品买单和商品拍卖最高出价都会预占共享仓库容量；捆绑拍卖按资产包中全部商品数量之和预占，并与商品买单合并计算。工厂买单和工厂竞拍不占仓库。

## 3. 工厂运行中卖出

运行中的工厂允许进入卖单。

服务器提交工厂卖单时：

1. 结算已经完成的完整周期。
2. 从待加入数量和当前参与数量中冻结卖出数量。
3. 立即降低当前或下一周期参与数量。
4. 冻结数量停止产生商品和运营成本。
5. 保持剩余未冻结工厂的运行意图和周期语义。

撤销工厂卖单时，未成交数量解冻：

- 集群停止时立即恢复可用数量；
- 集群运行时进入 `pendingJoinCount`，从下一周期加入；
- 不重置当前周期，不生成部分周期产出。

## 3.1 拍卖与订单簿隔离

拍卖成交不属于订单簿成交。商品、工厂或捆绑拍卖的起拍价、当前最高价和成交总价不得进入 `AssetOrder`、订单簿买卖盘、`lastPrice`、`lastTradePrice`、价格历史、主动方向成交量、市场需求真实成交 VWAP 或资产估值；混合资产包总价不得拆分为虚构单价。订单簿冻结与拍卖冻结只在可用资产校验和工厂参与数量上合并，撮合、价格发现和成交统计保持独立。

商品拍卖最高出价者按资产包中全部商品数量之和预占仓库；同一资产包中的多种工厂与工厂卖单冻结数量共同退出生产。拍卖取消或流拍释放对应冻结，成交时原子转移整个资产包，不创建伪造订单或成交明细。冻结资金、冻结商品和冻结工厂继续存在于各自权威余额中；托管记录不得作为第二份资产余额重复计入总资产。

## 4. 撮合与非交叉不变量

撮合固定遵循：

1. 资产种类和资产 ID 必须相同。
2. 价格优先。
3. 同价时间优先。
4. 最高买价大于或等于最低卖价时成交。
5. 允许部分成交。
6. 未成交部分继续保留。
7. 禁止自成交。
8. 成交、冻结、双方资产和统计在同一事务中更新。

商品订单和工厂订单不得互相撮合；不同商品或不同工厂类型也不得互相撮合。不同主体之间可成交的交叉订单必须在新订单进入订单簿时立即按 maker price 撮合，直到新订单完成、对手盘耗尽或最高买价严格低于最低卖价；不得先公开展示可成交交叉盘口再等待下一次轮询处理。

消费需求买单只能与同商品的玩家卖单成交；市场储备买单和卖单只能与玩家订单成交，任何两个 `ownerType = population` 的系统订单都必须排除，禁止消费需求与储备、不同储备方向或同组系统订单互相制造成交。消费需求成交必须记录 `lastFilledAt`，用于下一周期停止该商品的阶梯提价；无人卖出时旧消费买单保留 50%，新买单在保留最高价基础上上调 3%，有效消费买单总价值最多为当前周期预算的 3 倍。

系统订单不能用系统自成交清除交叉，因此报价生成器必须主动维护以下不变量：

```text
同一商品同时存在系统买盘和系统卖盘时：
最高系统买价 < 最低系统卖价
```

具体规则：

- 储备卖价必须至少为当前最高系统买价加 1 个整数价格单位。
- 储备买价必须至多为当前最低系统卖价减 1 个整数价格单位。
- 报价仍必须位于商品基础价 50%～300% 的合法区间，并保留储备自身买价低于卖价的动态价差。
- 无法在合法区间内避让时，不创建对应方向的储备订单；不得压低消费需求订单、制造系统成交或突破价格边界。
- 每轮系统订单生成后必须重新检查系统盘口；若异常状态仍交叉，优先取消当前周期的储备卖单，其次取消当前周期的储备买单，并原额释放冻结库存或冻结资金。
- 交叉修复只允许重新定价、跳过报价或撤销储备订单，不得生成 fill、成交量、价格历史、估值变化、手续费或排行榜交易额。

公开订单簿的长期稳定状态不得包含可成交的不同主体交叉订单、同一玩家交叉订单或系统交叉订单。瞬时函数调用内部可以在新订单创建后立即进入撮合，但事务返回和客户端状态序列化之前必须恢复非交叉状态。

### 4.1 共享撮合内核

`server/src/order-matching.js` 是商品与工厂限价订单的唯一撮合状态机。商品 `balanced-market.js` 与工厂 `facility-groups.js` 必须调用同一个 `matchIncomingOrder`，不得各自重新实现候选排序、价格交叉判断或逐笔成交循环。

共享内核统一负责：

- 同资产、反方向和有效未完成订单筛选；
- 价格优先、同价时间优先；
- 同一玩家自成交的最后一道阻断；
- resting order 作为 maker，并使用 maker price；
- 部分成交数量、`remaining`、`status` 与 `lastFilledAt` 推进；
- `makerOrderId`、`takerOrderId`、`liquidity`、手续费与净额字段的逐笔 fill；
- 统一调用玩家卖方手续费规则。

资产适配器只允许负责自身业务结算与行情记录：商品适配器转移玩家库存、真实人口冻结资金和市场储备资产；工厂适配器转移工厂集群数量并维护生产参与状态。共享内核不得导入商品目录、工厂目录、仓库或生产规则，适配器也不得重新实现排序、maker/taker、fill 或订单状态推进。历史迁移和运行时新订单必须复用同一内核，避免两类资产形成不同撮合语义。

## 5. maker price 与逐笔成交

每次成交使用撮合前已经位于订单簿中的对手方挂单价格，即 maker price。

服务器在相关订单 `fills` 中逐笔保存：

```ts
interface OrderFill {
  id: string;
  quantity: number;
  price: number;
  total: number;
  fee: number;
  netTotal: number;
  counterparty: string;
  createdAt: number;
  makerOrderId: string;
  takerOrderId: string;
  liquidity: 'maker' | 'taker';
}
```

同一订单扫过多个价格时必须生成多条独立成交明细，不得合并为平均价。浏览器不得使用市场 `lastPrice` 反推玩家成交价，也不得以最终市场价覆盖真实逐笔价格。

订单簿和价格曲线属于市场状态；玩家成交记录属于订单级权威明细。两者不得互相替代。

### 5.1 玩家卖出手续费

统一订单簿只向玩家卖方收取手续费，买方仍按 `quantity × maker price` 支付完整成交总额。商品卖单、工厂卖单以及玩家卖给“食品市场需求”“家庭消费市场需求”“食品市场储备”或“家庭消费市场储备”买单的成交使用同一规则；玩家从储备卖单买入时玩家不是卖方，不收取玩家卖出手续费。拍卖结算不属于订单簿，不收取统一订单簿玩家卖出手续费。

手续费按单张卖单自规则启用后的累计成交总额精确计算，不设置最低手续费：

```text
累计应收手续费 = floor(累计成交总额 × 1%)
本次 fill 手续费 = 累计应收手续费 - 此前累计已收手续费
本次卖方实收 = 本次成交总额 - 本次 fill 手续费
```

累计成交额不足 100 时手续费为 0；累计达到 100 时收取 1，达到 200 时累计收取 2。拆单或部分成交不能改变同一卖单的累计费率。

`fill.total` 始终保持成交总额，卖方 fill 的 `fee` 表示本笔增量手续费，`netTotal` 表示本笔实际到账。手续费不再进入 `systemSinks`，而是按基础人口 20%、技术人口 60%、专业人口 20% 转入市场服务待结算收入。人口消费成交从真实人口冻结资金支付，禁止增加 `populationIssued`。

上线迁移不得追收既有成交：旧卖单第一次发生新成交时，已有 fill 保留原值，V2 累计字段从 0 开始，只对启用后的新 fill 收费。服务器必须在同一成交事务完成扣费、就业转移和 fill 标记，幂等重试或状态轮询不得重复扣费。

### 5.2 行情主动方向

商品和工厂每笔成交除价格、数量和时间外，还必须保存吃单方（taker／incoming order）的买卖方向。主动买单吃掉已有卖单时记为主动买入，主动卖单吃掉已有买单时记为主动卖出；不得把成交双方的买卖数量直接相减，也不得按价格涨跌猜测主动方向。

概览页与市场页共用同一最近 24h、6 分钟行情分段。分段收盘价取最后一笔成交价，总成交量为全部成交数量之和，净主动量为主动买入量减主动卖出量。成交量柱高表示总成交量，颜色只表达净主动方向：净买入使用成功色，净卖出使用危险色，均衡或旧历史方向未知使用中性色。旧成交没有主动方向字段时必须保留为中性成交量，新成交逐步替换该窗口，禁止伪造迁移方向。

## 6. 单列价格档位订单簿

每个选中资产固定显示：

```text
卖盘标题
价格最低的 5 个有效卖出档位（最低卖价靠近中线）
无文字中性分隔线
买盘标题
价格最高的 5 个有效买入档位（最高买价靠近中线）
```

每行只显示：

```text
方向｜价格｜合计剩余数量
```

规则：

- 同资产、同方向、同价格的有效订单按当前剩余数量聚合为价格档位。
- 有效订单只包括 `open` 或 `partial`、`remaining > 0` 且价格为正整数的订单；已成交、已取消和零剩余订单不得进入档位。
- 聚合只累加 `remaining`，不得累加原始 `quantity`，并记录档位内独立订单笔数用于无障碍名称。
- 必须先聚合全部有效订单，再按买盘价格降序或卖盘价格升序排序，聚合完成后再按最优价格截取 5 档；不得先截取 5 笔订单再聚合。
- 卖盘视觉上将已选出的最优 5 档反向排列，使最低卖价靠近中线；买盘最高买价直接靠近中线。
- 档位聚合只属于客户端匿名展示，不改变服务器 `AssetOrder`、价格优先、同价时间优先、部分成交、冻结、逐笔 fill、撤单和单张卖单手续费。
- 不显示玩家身份或市场需求身份；玩家订单与市场需求订单同价时使用相同档位结构。
- “我的当前资产未完成订单”和“我的订单与成交”继续逐单显示，使用独立订单 ID 单独撤销，不得按档位合并。
- 订单簿内部不显示买一、卖一、价差、最近成交或指标卡式成交线。
- 桌面订单簿按内容自然高度显示；订单不足时不生成虚假空行，也不得与行情卡强制等高。
- 移动端按内容自然增长。

## 7. 下单界面与默认价格

玩家界面统一将订单输入字段称为“价格”，不得显示“限价”字段、标题或表头。底层订单仍是限价订单模型，服务器撮合、冻结和校验规则不变。

界面顺序固定为：

```text
方向
价格
数量
1/4 仓｜1/2 仓｜全仓
订单总额
提交订单
当前资产未完成订单与撤单
```

默认价格只从客户端当前已经加载的 `game.orders` 本地快照计算，不得为填写默认价格调用 `getGameState`、`refresh`、新增 API 或发起任何额外网络请求。

只考虑同资产种类、同资产 ID、`open` 或 `partial`、剩余数量大于零且价格为正整数的订单：

```text
买入：最低卖出价
      → 没有卖单时使用最高买入价
      → 买卖盘均为空时使用 1

卖出：最高买入价
      → 没有买单时使用最低卖出价
      → 买卖盘均为空时使用 1
```

重新填写默认价格的时机：

- 首次进入市场页；
- 从其他页面重新进入市场页；
- 主动切换商品或工厂资产；
- 主动切换买入／卖出方向。

自动刷新、下单响应、成交、撤单或其他权威状态同步只更新本地订单快照，不得直接覆盖当前价格输入。玩家下一次重新进入市场、切换资产或切换方向时，使用届时最新的本地订单快照重新计算。

在市场页内再次点击当前已经选中的资产或当前方向时，不重新填写价格，保留玩家手动输入。默认价格不得使用 `lastPrice`、商品 `basePrice`、工厂 `systemValue` 或正式估值价格作为回退。

商品买入快捷数量：

```text
maxBuy = min(warehouseAvailableCapacity, floor(credits / price))
```

商品卖出快捷数量使用当前可用库存。冻结商品不能出售。

工厂买卖快捷数量使用当前可用资金或未冻结工厂数量，不使用共享仓库容量。

## 8. 订单来源与模型迁移

商品订单只允许玩家、消费需求或市场储备作为所有者：

- 玩家可以提交商品买单和卖单。
- 消费需求只提交买单，名称只能是“食品市场需求”或“家庭消费市场需求”，`demandTier` 只能是 `direct` 或 `derived-liquidity`。
- 市场储备可以提交商品买单和卖单，继续沿用所属需求组的兼容名称，`demandTier` 分别为 `liquidity-buy` 或 `liquidity-sell`；不得根据 `ownerName` 判断订单角色。
- 持久化字段 `ownerType = 'population'` 只是两类系统商品订单的兼容标识。
- 储备买单必须由真实储备可用资金全额冻结；储备卖单必须由真实储备可用库存全额冻结。不得创建无资产支持的系统订单，不得在普通周期补发储备资产。
- 消费需求和市场储备订单不得互相成交，任何系统订单之间都不得成交。
- 消费需求仍执行 70%／30% 预算、旧买单 50% 保留与 3% 阶梯提价；市场储备每周期释放旧冻结并重新报价，不继承这些规则。
- 不生成企业采购或普通人口需求，也不创建任何系统工厂买单、系统工厂卖单或固定价格工厂供给。

模型迁移规则：

- 模型 3 升级到 4 时一次性初始化储备资金和商品种子，并立即生成消费买单与储备买卖单。
- 模型 4 升级到 5 时必须删除全部旧系统商品订单，先把每个储备组的冻结资金原额退回可用资金、把每种商品的冻结库存原额退回可用库存，再按非交叉规则重建系统盘口。
- 模型 5 迁移不得改变玩家资金、商品、工厂、玩家订单、玩家订单冻结资产、成交记录或价格历史，也不得重复发行储备资金和商品种子。

工厂订单仍只能由玩家提交；服务器周期、迁移、刷新和进入游戏都不得创建工厂订单。

## 9. 估值与排行榜

每项商品和工厂的估值价格使用：

```text
该资产最近一次订单簿真实成交价（lastTradePrice）
```

- 只有统一订单簿撮合生成、带真实买卖主动方向的成交才能更新 `lastTradePrice`；玩家与市场储备的成交属于真实成交并更新估值，但在需求价格传导中只使用 50% 信号权重。
- 新世界的模拟价格历史、商品 `basePrice`、工厂 `systemValue`、当前 `lastPrice`、买卖挂单价格和拍卖价格都不得充当最近真实成交价。
- 从未发生订单簿真实成交时估值为 0；界面必须显示货币 0，不得显示 `--`。
- 商品估值数量包含可用和冻结库存。
- 工厂估值数量包含玩家拥有的全部数量，包括卖单和拍卖冻结数量。
- 总资产、资产摘要和财富排行榜由服务器统一计算。
- 客户端不得使用系统参考价、任意挂单价、基础价、系统价值或拍卖价替代正式估值。

## 10. 本地成交

浏览器本地成交记录只比较相邻权威状态中新出现的 fill ID，并使用匿名 `fill.price`、`fill.quantity`、`fill.total`、`fill.fee`、`fill.netTotal` 和 `fill.createdAt`；不得读取或推断对手信息。

本地日志 v3 升级时清空由旧 `lastPrice` 推导的成交记录；v4 将生产记录升级为多输入结构并建立包含工厂目录的新快照。

## 11. 防回退

不得恢复：

- 商品和工厂二级市场切换；
- 工厂固定价格挂牌或即时购买卡；
- 工厂必须停止后才能挂卖单；
- 按独立订单逐行显示匿名公共买卖盘；
- 先截取 5 笔订单再聚合、用原始 `quantity` 代替当前 `remaining`，或跨买卖方向合并档位；
- 把价格档位作为服务器订单保存，或用档位破坏同价时间优先、逐单撤单、逐笔成交和单张卖单手续费；
- 合并“我的订单”或撤单入口；
- 双列买卖盘；
- 使用 `lastPrice` 反推成交价格；
- 使用系统参考价计算正式总资产；
- 客户端撮合或客户端解冻资产；
- 工厂资产标签使用机械齿轮图标、`⚙` 字符或与机械商品相同的 SVG 路径；
- 在玩家界面恢复“限价”文案；
- 使用最近成交价、基础价、系统价值或估值价填写默认价格；
- 自动刷新覆盖玩家已输入的价格；
- 为默认价格计算新增服务器请求；
- 从其他页面重新进入市场时继续保留旧价格而不按最新本地订单簿重算；
- 使用最近成交价直接替代价格传导参考价生成储备报价，或让库存偏移超过 ±10%；
- 创建无真实储备资金支持的商品买单、无真实储备库存支持的商品卖单，或在迁移后重复补发储备资产；
- 允许系统商品订单彼此成交、允许最高系统买价不低于最低系统卖价，或绕过 4%～24% 动态价差与基础价 50%～300% 边界；
- 通过跳过自成交候选，让同一玩家的交叉商品或工厂订单长期同时留在公开订单簿；
- 让不同主体的可成交交叉订单等待下一次轮询，而不是在入簿时立即撮合；
- 创建任何系统工厂买单、系统工厂卖单或固定价格工厂供给；
- 保留旧世界中的系统工厂订单；
- 让库存价值扩大市场需求预算、让直接与派生需求超过总预算，或在同一周期跨越多条生产边；
- 对同一卖单的每条部分成交重复收取最低手续费、向买方加收手续费、追收上线前成交，或对拍卖收取统一订单簿玩家卖出手续费。

## 普通玩家订单与成交匿名化

服务器内部订单继续保存真实所有者、需求组和 maker/taker 关系，用于撮合、结算、市场需求货币发行统计和管理员审计。普通玩家状态必须经过集中式公开订单序列化：本人订单仅增加 `isOwn: true` 并返回由 `id / quantity / price / total / fee / netTotal / createdAt` 组成的匿名 fills；其他订单返回 `isOwn: false` 且不返回 fills。

普通玩家状态中的所有订单都不得返回真实 `ownerId`、`ownerName`、`ownerType`、`demandGroupId`、`demandTier` 或 `demandCycleId`。公开 fill 不得返回 `counterparty`、`makerOrderId`、`takerOrderId` 或 `liquidity`。消费需求、市场储备与玩家订单在普通玩家 JSON 中必须使用相同结构，不能通过字段差异判断来源。

客户端通过 `isOwn` 识别本人订单，并只显示订单部分成交、全部成交、成交数量、价格、总额、手续费、实收和时间。隐藏界面列不能替代 API 脱敏。

## 市场页面布局与可用性

市场页保持商品和工厂同一连续资产目录，不新增二级市场页面或隐藏筛选。目录在桌面使用两行横向网格，商品和工厂之间必须有非交互分组标记，并提供前后滚动按钮、边缘可滚动提示和明确的“当前”可见状态；选择远端资产后必须自动滚动到可见范围。

市场交易主区按实际内容宽度响应：

- 内容宽度不小于 `1280px` 时固定为 `320px` 下单列、`360px` 订单簿列和不小于 `620px` 的行情主列；
- 内容宽度为 `820px–1279px` 时下单与订单簿同排，行情图和账户记录跨整行；
- 内容宽度小于 `820px` 时全部单列；
- 响应式必须使用市场内容容器宽度，不能只依赖浏览器视口或侧栏外的全局断点。

完整行情图继续使用 `960×540` 视图框，但屏幕高度必须由图表自身宽度和 `16:9` 比例决定，不得使用基于 `vw` 的固定高度。图表是宽屏第一排的最大模块；底部统计使用可重排的两列网格，窄卡片下降为单列。

行情底部“成交笔数”只统计与 240 个六分钟分段完全相同的最近 24 小时窗口。零涨跌使用中性色并显示 `0`，不得显示绿色 `+0`。

下单界面在当前条件无法提交时必须显示明确的文字原因，并同步禁用数量输入、快捷仓位与提交按钮。不得通过 `Math.max(1, maxTradeQuantity)` 伪造可交易上限；资金不足、仓库不足、商品无库存和工厂无可售数量必须分别说明。

单列订单簿的视觉与 DOM 顺序固定为：

```text
卖盘标题｜最低价前 5 档
卖出档位（最低卖价靠近中线）
中性分隔线
买盘标题｜最高价前 5 档
买入档位（最高买价靠近中线）
```

标题必须位于对应档位之前。订单不足时卡片按内容自然高度显示，不得与行情卡强制等高或制造虚假空白。

## 市场需求模型 V6 订单簿约束

- 市场需求模型版本为 6。消费需求订单在每个周期开始时先结算上一周期整个时间窗内的成交，再发布新订单。
- 同一商品的新消费预算按 50%／30%／20% 形成三档买盘；旧消费订单最多存在两个周期，无成交时只保留 35%，不得固定每周期提高 3%。
- 单商品消费买盘价值上限为本周期该商品分配预算的 1.5 倍，需求组总买盘价值上限为周期预算的 2.5 倍。
- 成交价格历史内部记录 `marketRole = player | consumption | liquidity`；预算活跃度只统计 `player`，价格观察允许 `consumption` 100% 和 `liquidity` 50% 权重。普通玩家 API 不得暴露该内部角色。
- 商品供需压力允许低于 1 表示公开供给过剩，不得恢复只取历史最大压力的单向实现。
- 市场储备仍使用真实可用资金和库存，任何系统订单之间都不得成交，且最高系统买价必须严格低于最低系统卖价。
