# Economy 礼品兑换、资产拍卖与管理员设计

> 状态：礼品码、藏品、藏品／商品／工厂拍卖、宝石邀请封禁与管理员后台的当前权威设计
> 适用项目：`RIVERS0FT/Economy`
> 更新时间：2026-07-24

## 1. 礼品兑换

设置页提供独立礼品兑换卡。兑换码规范化后在礼品码业务表中只以 SHA-256 哈希保存；同一账号对同一礼品只能兑换一次，并受启用状态、开始时间、过期时间和最大次数限制。兑换与资金增加、giftIssued 统计及兑换记录必须在同一 SQLite 事务内完成。

管理员既可以创建单个礼品码，也可以一次批量生成 1～50,000 个随机礼品码。批量中的所有礼品码共用奖励金额、每码最大兑换次数、开始时间、过期时间和管理备注，并必须在同一个 SQLite 事务中完成：任意非冲突类写入失败时整批回滚，不允许只创建部分结果。随机码与数据库中既有礼品码冲突时应重新生成，最终返回数量必须与请求数量完全一致。

单个创建和批量创建只在创建响应中返回明文；批量结果由管理员页面保存在当前页面内，并支持下载 UTF-8 `.txt` 文件，格式为每行一个兑换码。礼品码业务表、世界状态和管理员记录列表不得保存明文。为保证管理员写操作的幂等重试，创建响应可以沿用通用 `economy_idempotency` 缓存，在现有 24 小时 TTL 内短期保存并按期删除；不得将该缓存改造成长期礼品码明文档案。批量创建必须使用 `Idempotency-Key`；同一请求重试必须返回第一次创建的同一批明文，不得重复插入新礼品码。

## 2. 藏品模型

- 每件藏品是世界状态中的唯一实例，使用不可复用的内部 `collectibleId`。
- 每件藏品只允许一个当前持有人；未分配藏品的持有人为 `null`。
- 服务器保存当前归属和不可覆盖的归属历史；创建、管理员初始分配和拍卖成交都必须追加记录。
- 藏品图片源固定为芝加哥艺术博物馆 IIIF，正式图片 URL 由服务器按 `imageId` 生成：`https://www.artic.edu/iiif/2/{imageId}/full/{width},/0/default.jpg`。
- 管理员上传格式只保存芝加哥艺术博物馆藏品 ID、`imageId`、标题、艺术家、年代、材质、尺寸、说明、公版标记和可选初始持有人 ID，不接受任意图片 URL 或其他图片域名。
- 只有明确提供 `isPublicDomain: true` 的记录可以导入；重复的芝加哥艺术博物馆藏品 ID 必须拒绝。
- 图片外链加载失败时只影响展示，不得改变归属、拍卖或资金状态。
- 藏品目前不参与商品／工厂订单簿、资产估值或排行榜；后续若加入估值，必须在统一资产订单簿设计中定义唯一估值来源。

### 2.1 管理员 JSON 格式

管理员页面接受 JSON 数组或带 `items` 数组的对象，每批 1～100 条：

```json
[
  {
    "sourceArtworkId": 28560,
    "title": "The Bedroom",
    "artist": "Vincent van Gogh",
    "dateDisplay": "1889",
    "mediumDisplay": "Oil on canvas",
    "dimensions": "73.6 × 92.3 cm",
    "description": "可选说明",
    "imageId": "芝加哥艺术博物馆 image_id",
    "isPublicDomain": true,
    "initialOwnerId": 123
  }
]
```

`initialOwnerId` 可省略或设为 `null`；提供时必须对应已经存在的玩家。导入操作必须使用 `Idempotency-Key`，同一请求重试不得重复创建藏品。

拍卖成交属于独立资产包结算，不得更新商品或工厂的 `lastTradePrice`、订单簿价格历史、主动成交方向或财富估值；捆绑总价不得拆成虚构单件价格。

## 3. 资产拍卖

藏品、商品和工厂使用同一公开竞价、最高出价冻结和服务器自动结算模型。一场拍卖是一个不可拆分的资产包：单项拍卖是只含一个项目的资产包，捆绑拍卖可以同时包含多件藏品、多种商品和多种工厂。持久化继续沿用 `collectibleAuctions` 数组兼容旧世界，新记录以 `items[]` 为权威结构，并保留首项 `assetKind`、`assetId`、`quantity` 兼容别名；客户端返回通用 `assetAuctions`、`itemSummaries`、`itemCount` 与 `isBundle`，仅含一件藏品的旧调用继续通过 `collectibleAuctions` 别名读取。资产包结构继续兼容；当前世界版本统一为 14，客户端状态版本为 16。

- 资产包至少 1 项、最多 20 个规范化资产项目。每项包含 `assetKind`、`assetId` 和 `quantity`；藏品数量固定为 1，同种商品或同种工厂重复加入时合并数量，同一藏品不得重复。
- 只有当前所有者可以发布。发布必须先完整预检所有项目，再整体冻结；任一项目不存在、归属异常、可用数量不足或已在其他拍卖中时，整个请求失败且不得冻结任何项目。
- 商品从 `available` 转入 `frozen`；工厂拍卖冻结数量与订单簿卖单冻结合并计算，并立即退出当前参与数量和后续生产；藏品保持卖方归属但进入逻辑托管。
- 冻结资产继续计入卖方总资产。冻结只限制资产的交易、生产或使用，不产生所有权转移；拍卖记录只描述托管，不得在现金、库存或工厂权威余额之外重复计价。
- 起拍价为正整数，表示整个资产包总价；拍卖时长为 1～168 小时。卖家不能竞拍自己的资产包，竞拍者不能只购买其中一部分。
- 首次出价不得低于起拍价；后续出价至少提高 1。最高出价资金转入冻结资金并继续计入竞拍者总资产；被超价时原最高出价立即完整释放，同一最高出价者只冻结提高报价的差额。
- 最高出价者的仓库预占必须覆盖资产包中全部商品数量之和；多种商品合并占用。被超价、取消、流拍或结算后释放预占；成交时等量转换为实物库存。
- 没有出价的拍卖可由卖家取消并整体解冻；出现任何有效出价后不得取消。资产包发布后不得增删项目、修改数量或拆分成交。
- 拍卖结束由服务器自动结算。结算先验证全部归属、冻结、仓库和资金，再一次性转移全部项目和总价；任一检查或执行失败必须回滚已执行变化、取消拍卖并尽可能释放双方冻结，不得出现部分成交。
- 捆绑拍卖中的藏品归属历史记录共享同一个 `auctionId` 和整包成交总价，但不得把整包总价重复写成每件藏品的单件 `price`。
- 工厂成交给正在运行的买方集群时从下一完整周期加入。卖方剩余工厂保持原运行意图；全部工厂曾被冻结导致异常时，解冻后从新的完整周期恢复，不补算冻结期间产出。
- 拍卖成交价不得写入商品或工厂订单簿行情，不更新 `lastPrice`、`lastTradePrice`、价格历史、最近真实成交估值或统一订单簿成交量。混合资产包总价不得按比例分摊为商品或工厂单价。拍卖结算不收取统一订单簿玩家卖出手续费；未来新增拍卖费用必须在本权威设计中单独定义，不得复用订单簿最低手续费。
- 进行中的出售拍卖或最高出价不得被客户端清除；玩家不提供经济状态重置、清空进度或重新开始能力。

## 4. 玩家页面职责

- “藏品”页只展示当前账号持有的藏品，使用图片卡、作品信息、馆藏来源链接和拍卖状态。
- “拍卖”页使用资产包编辑器连续加入藏品、商品和工厂，按整包总价竞价，查看进行中拍卖、取消无出价拍卖以及始终可见的最近结算区域。
- “设置”页展示宝石余额、专属分享链接、永久邀请码、邀请统计、最近邀请以及注册后 24 小时内的一次性邀请码填写入口。
- 页面不得从浏览器直接修改归属、最高出价、结束时间、冻结资金、宝石余额、邀请关系或封禁状态。

## 5. 管理员后台

管理员唯一地址固定为 `https://game.riversoft.top/economy/admin`；原 `/economy/admin/bans` 独立页面与路由已移除，不提供重定向。后台使用“概况／社区／藏品／礼品码／账号封禁”五分区导航，封禁复核直接合入统一后台。普通主导航不显示管理员入口，设置页仅对 `role=admin` 显示一个管理员后台入口。所有 `/api/game/admin/` 接口必须由服务器再次校验管理员角色，普通用户返回 403。管理员账号即使因同 IP 规则处于封禁状态，也只能继续访问管理员接口完成复核，不能绕过封禁进入普通游戏接口。

桌面端复用统一 `SignedInShell`、`SidebarFrame`、共享 `ScrollArea` 和 `game-shell-layout.css`：侧栏外距、侧栏与工作区间距、桌面玻璃工作栏外距、页面内容边缘、一级内容网格间距与贴右侧页面滚动条必须和玩家端读取同一个 `--desktop-layout-gutter`。管理员桌面工作栏由 `AdminDesktopBar` 复用 `desktopStatusBar` 液态玻璃预设，承载当前分区标题、说明、管理员账号、世界／API 状态和刷新操作；管理员页面不得创建第二套桌面玻璃材质、根级 padding、居中限宽页面框或原生主滚动容器。移动端复用统一 `MobileBottomNavigationFrame`、`mobileNavigation` 液态玻璃预设、`48px × 48px` 导航按钮、唯一原生横向滚动 `<nav>`、`68px` 底栏高度和安全区几何；管理员移动端不渲染顶部状态栏或桌面玻璃工作栏。管理员移动页面层与 `.mobile-chrome-overlay.admin-mobile-chrome-layer` 必须保留在 `SignedInShell` 工作区内的同一单格 Grid Overlay：页面层使用 `order: 1`，Chrome 层使用 `order: 2`，以网格绘制顺序保证一级卡片先绘制、管理员 Chrome 后绘制；Chrome 层保持 `position: relative` 与 `z-index: auto`，不得传送到 `document.body`，也不得用正向层级掩盖错误结构。桌面和移动端必须共享同一个 `activeSection`，切换导航不得重建管理员应用或清空其他分区状态。不得恢复顶部横向 `panel` 导航条、管理员专属液态玻璃参数、第二套移动导航滚动容器或独立底栏尺寸。

后台提供：

- 世界概况，并在概况内部提供人口经济指标、完整当前人口政策参数与持续时间，以及受控人口政策调控；
- 单个礼品码创建、一次批量生成最多 50,000 个礼品码、TXT 下载、停用与兑换记录；
- 按固定 JSON 格式导入芝加哥艺术博物馆公版藏品；
- 查看藏品图片、馆藏编号、当前持有人、拍卖状态和归属历史；
- 查看同注册 IP 封禁事件、事件成员、注册来源、注册时间和当前封禁状态；
- 填写管理员备注后解禁单个账号、解禁整个事件或重新封禁账号。

批量礼品码页面不得把最多 50,000 条明文逐项渲染到 DOM；只显示本次生成数量和单个码预览，并通过 Blob 下载一行一个兑换码的 TXT。创建按钮在请求期间必须禁用，避免管理员重复提交。明文只保留在当前页面状态和通用幂等缓存的既有 TTL 内，礼品码历史列表继续只展示不可逆的元数据。

藏品列表、礼品码列表、归属历史和兑换记录可能持续增长，必须复用共享 `VirtualList`。藏品、归属和封禁事件继续使用窗口化结构；礼品码历史与单码兑换记录必须同时使用服务端游标分页和客户端 `VirtualList`：

- 礼品码与兑换记录默认每页 100 条、最多 200 条，响应返回当前页数组、完整总数和 `nextCursor`；
- 首屏只请求第一页，管理员明确点击“加载更多”后才使用游标继续读取，不得全表返回后仅靠窗口化隐藏 DOM；
- 已加载页面合并进客户端状态，DOM 只渲染可视区域和少量预加载行；
- 表头与虚拟记录使用统一列定义，并保持横向滚动对齐；
- 礼品码游标按递减 ID 稳定翻页；兑换记录按兑换时间和玩家 ID 组合游标稳定翻页；
- 每条记录必须使用服务器稳定 ID；兑换记录使用玩家 ID 与兑换时间组合键；
- 分页和窗口化不得改变查看归属、停用礼品码、查看兑换记录、馆藏链接或封禁复核等操作。

后台不提供任意修改玩家普通货币、直接删除玩家、编辑世界 JSON、覆盖归属历史或上传二进制图片。管理员解禁与重新封禁必须使用 `Idempotency-Key`，写入操作人、时间、备注和不可删除的审计记录。解禁不得自动补发被封禁阻止的邀请宝石，也不得删除原事件或清空注册 IP 指纹或修改玩家既有资产。

### 5.1 人口经济概况

管理员导航仍固定为五分区，不新增“人口经济”第六分区。人口经济区域位于“概况”现有世界指标下方，展示三类人口可用／冻结／待结算资金、五档消费状态（奢靡／繁荣／正常／拮据／生存）、状态持续周期、食品／家庭预算、收入健康度、基础收入覆盖、状态判定钱包覆盖、施工就业托管、生产／建造／仓储／市场服务累计收入、C1～C7 生产工资以及各类发行统计，并在同一区域提供完整当前人口政策参数、持续时间和受控“人口政策调控”。其中状态判定钱包覆盖必须显示服务端保存的自动补充前指标，不能用刷新后钱包重新推算替代。人口政策不显示、不查询也不新增调控记录。

人口政策允许调整稳定需求比例、生产工资系数、目标钱包周期、单周期补充比例、三类人口倍率和政策有效周期。业务参数不设上限，只保留下限：稳定需求比例与单周期补充比例不低于 0%，生产工资系数与三类人口倍率不低于 50%，目标钱包与有效期不低于 1 个周期。所有参数必须是安全整数，比例以整数基点保存；到期周期、稳定预算、目标钱包、补充额度和生产工资等计算结果必须处于系统安全整数范围，超出时拒绝请求或结算。三类倍率调整后的稳定预算总额不得突破全局稳定需求比例确定的总额。生产工资系数只调整人口获得的生产工资，不改变玩家生产成本；已开始周期锁定原系数。非默认政策到期后由服务器自动恢复稳定需求 12%／生产工资 100%／3 周期／补充比例 100%／三类 100% 的模型默认值，服务重启不得延长有效期。

管理员可以按当前政策立即补充全部或单类人口，但只能补目标钱包缺口，可用与冻结资金共同计入钱包覆盖；自动补充和管理员补充共享同一五分钟周期上限，更换幂等键或重复点击不得绕过。立即补充只增加人口可用资金，从后续正常需求处理参与消费，不重建当前订单，不改变收入 EMA、就业收入、消费状态或近期收入峰值。后台仍不得任意填写或扣除人口余额、修改就业比例、强制消费状态、工作发行、商店发行或直接编辑世界 JSON。

管理员状态标签颜色固定为：奢靡紫色、繁荣蓝色、正常绿色、拮据橙色、生存红色。桌面比较矩阵和移动端人口卡必须同时显示状态持续周期与“健康／收入覆盖／判定钱包”三项服务端指标；当前钱包覆盖可以继续显示补充后的实时钱包，但必须与状态判定钱包明确区分。管理员不得增加强制选择消费状态的控件。

更新政策、恢复默认和立即补充都必须要求 `Idempotency-Key`，但不要求管理备注，也不写入人口调控审计记录。写操作必须在世界状态保存与通用幂等响应的同一 SQLite 事务内完成；更换幂等键不得绕过同周期补充约束。既有 `economy_population_policy_audit` 历史表只做数据库兼容保留，不再写入、不再提供管理员查询接口，也不在客户端展示。管理员刷新复用现有概况刷新按钮，不增加独立轮询；市场页面不得复用或展示该区域。

### 5.2 运营控制台布局方案

管理员界面采用“运营控制台”而不是表单纵向堆叠方案，最终样式标识固定为 `ADMIN_CONSOLE_SCHEME: command-center`，由 `src/styles/admin-navigation.css` 在统一外壳和基础组件之上负责管理员专属的信息编排，不创建第二套按钮、面板、表单、页面滚动视口或玻璃材质。

- 桌面玻璃工作栏固定承载当前分区标题、说明、管理员身份、世界／API 摘要和刷新操作；桌面 `PageLayout` 标题行隐藏，避免重复标题。移动端不渲染该工作栏，继续显示 `PageLayout` 标题、说明和全宽刷新按钮。
- “概况”保持全宽结构，但世界指标必须压缩为高密度网格；世界版本与 API 状态由共享桌面工作栏承载，概况内容不得重复占用大卡片。

### 5.3 玩家运营统计

管理员导航继续固定为五分区，不新增“玩家统计”分区。玩家运营统计位于“概况”世界指标与人口经济之间，采用“运营快照／新增与经济活跃趋势／留存与注册来源／经营成长漏斗／当前经营参与／财富分布／需要关注的玩家群体”五层只读诊断结构；7 日、30 日和 90 日范围切换只刷新玩家统计，不重复加载人口经济、礼品码或藏品。

经济活跃的唯一口径是成功经济写操作：工作、建设或控制工厂、订单、仓库扩容、礼品兑换、宝石兑换、拍卖与长期合同等动作只有在服务器事务成功并更新 `lastEconomicActivityAt` 时才计入。状态轮询、失败动作、幂等重放、管理员读取以及后台生产、成交或合同自动结算不得计入经济活跃；后台结算可以独立计入生产／成交／合同参与，但不能刷新玩家活跃时间。

玩家日活动与首次里程碑保存在独立 SQLite 分析表，不进入世界 JSON、客户端状态版本、六分区或普通玩家接口。日活动、动作幂等结果和世界变更必须处于同一个 SQLite 事务；事务回滚时分析记录同步回滚。精确日活动覆盖起点由服务器首次创建分析表时固定保存，覆盖前或首个不完整自然日不得伪造 DAU、留存和阶段耗时；客户端必须显示 `coverageStartsAt`，并把未覆盖日期和不完整日期明确标记为部分历史。

留存固定按北京时间 `Asia/Shanghai` 自然日计算：某日建档玩家在注册后的 D1／D7／D30 目标自然日内至少完成一次成功经济写操作才算留存。成长漏斗依次为建档、首次经济操作、第一座工厂、首次生产和首次订单簿成交；阶段人数可读取当前权威资产和累计统计，注册到阶段的中位耗时只允许使用覆盖期内精确记录的首次里程碑，旧玩家时间未知时显示“历史耗时待积累”。注册来源只聚合直接建档、分享链接、手动邀请码和奖励被阻止四类，不显示邀请码、邮箱或 IP。

经营参与展示持有工厂、运行中工厂、未完成订单、本周生产、本周订单簿成交、进行中合同、进行中拍卖、仓库已满和异常工厂人数与占比。财富分布必须复用排行榜 `wealthAssetsFor()` 的唯一估值口径：现金按余额，商品和工厂只按最近一次统一订单簿真实成交价，从未真实成交的资产估值为 0；不得使用基础价、系统价值、拍卖总价或挂单价。财富区域展示总额、人均、中位数、P25／P75／P90／P99、前 1%／10% 占比、现金／商品／工厂构成、冻结资产占比和固定财富区间。

关注群体只允许使用可解释的客观条件聚合，包括注册超过 24 小时未激活、经营尚未启动、全部开启工厂受阻、仓库满且仍有生产或采购、高冻结比例、30 日沉睡、7 日活跃但本周无订单簿成交和区间新增封禁。聚合接口不得返回玩家邮箱、完整或截断 IP 指纹、邀请码、管理员备注、订单对手或逐玩家资产；后续增加明细时必须使用独立管理员接口、服务端游标分页和 `VirtualList`，不得把玩家数组塞入聚合响应。

玩家运营统计完全只读，不得把统计结果用于扩张人口需求预算、调整稳定需求、价格传导、市场储备、排行榜或任何玩家资产。`lastPlayerScaleBudget` 与 `lastInventoryBoost` 仅为旧世界兼容字段，必须保持停用和零值；活跃玩家数、库存价值、财富分位数和留存都只是管理员诊断。桌面宽屏允许业务卡片两列，721px～1179px 和移动端回到单列；移动端不得新增页面级横向滚动、第二纵向滚动视口或独立管理员 Chrome。

## 6. 宝石邀请与账号封禁管理

宝石是与普通货币分离的整数资产。每名玩家宝石初始为 0；分享链接注册或手动填写邀请码成功时，只向邀请人立即发放 10 宝石。宝石不参与商品、工厂、仓库、人口需求、总资产或排行榜。

邀请和封禁权威表包括：

- `economy_invite_codes`：每名玩家唯一且永久的 8 位邀请码；
- `economy_invitation_relations`：每名被邀请人最多一条邀请关系，记录分享链接或手动邀请码来源；
- `economy_gem_ledger`：邀请宝石与管理员宝石调整流水；
- `economy_ip_ban_incidents` 与 `economy_ip_ban_members`：同 IP 事件和成员；
- `economy_account_bans`：账号当前封禁或解禁状态；
- `economy_ban_audit`：系统封禁、管理员解禁和重新封禁审计。

管理员手动解禁后账号恢复普通游戏访问，但被 `blocked_same_ip` 或 `blocked_banned_account` 拦截的邀请关系不自动改成已奖励。服务启动扫描只为没有人工复核的新冲突建立封禁；人工解禁不得因重启自动失效。相同注册 IP 后续出现新的账号时，系统重新激活事件并再次封禁该指纹下全部账号。

批量接口请求体包含 `count`、`rewardCredits`、`maxRedemptions`、可选 `startsAt`、可选 `expiresAt` 和可选 `note`。`count` 必须为 1～50,000 的整数；响应包含 `createdCount` 与 `codes` 明文数组，仅供本次 TXT 导出和幂等重试。

封禁写接口请求体只接受可选管理备注 `note`，并要求 `Idempotency-Key`。查询响应可以显示指纹前 12 位作为事件区分，但不得返回完整指纹或注册 IP 明文。

社区入口默认使用 `https://qm.qq.com/q/eN8hya0Yn0`。管理员后台必须允许保存侧边栏“加入 QQ 群”按钮地址；服务端只接受长度不超过 2048、无账号信息的 HTTPS URL，写入 `economy_settings` 并要求 `Idempotency-Key`。普通玩家只通过 `GET /api/game/community-link` 读取有效配置，不得由浏览器本地存储覆盖服务器配置。

## 8. 防回退

不得：

- 将批量礼品码上限降到 50,000 以下，或把一批创建拆成可能部分成功的多个独立事务；
- 将礼品码明文写入 `economy_gift_codes`、世界状态、管理员日志或礼品码历史列表，或把通用幂等缓存的明文保留期延长为长期存档；
- 在管理员页面一次渲染最多 50,000 条礼品码明文，或移除 TXT 下载能力；
- 对相同 `Idempotency-Key` 的批量重试生成第二批礼品码；
- 把藏品降级为仅保存在浏览器的图片列表；
- 允许一件藏品同时属于多个玩家；
- 直接信任客户端提交的图片 URL、当前持有人、最高出价、结束状态、宝石余额、邀请归因或封禁状态；

- 为规避图片流量而把远程作品自动复制到游戏服务器；
