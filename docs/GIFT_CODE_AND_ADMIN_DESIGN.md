# Economy 礼品兑换、藏品拍卖与管理员设计

> 状态：礼品码、藏品、拍卖和管理员后台的当前权威设计  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-14

## 1. 礼品兑换

设置页提供独立礼品兑换卡。兑换码规范化后只以 SHA-256 哈希保存；同一账号对同一礼品只能兑换一次，并受启用状态、开始时间、过期时间和最大次数限制。兑换与资金增加、giftIssued 统计及兑换记录必须在同一 SQLite 事务内完成。

管理员既可以创建单个礼品码，也可以一次批量生成 1～50,000 个随机礼品码。批量中的所有礼品码共用奖励金额、每码最大兑换次数、开始时间、过期时间和管理备注，并必须在同一个 SQLite 事务中完成：任意非冲突类写入失败时整批回滚，不允许只创建部分结果。随机码与数据库中既有礼品码冲突时应重新生成，最终返回数量必须与请求数量完全一致。

礼品码明文不得写入数据库或管理员记录列表。单个创建和批量创建只在创建响应中返回明文；批量结果由管理员页面保存在当前页面内，并支持下载 UTF-8 `.txt` 文件，格式为每行一个兑换码。刷新、离开页面或丢失本次响应后不得从服务器恢复明文。批量创建属于管理员写操作，必须使用 `Idempotency-Key`；同一请求重试必须返回第一次创建的同一批明文，不得重复插入新礼品码。

## 2. 藏品模型

- 每件藏品是世界状态中的唯一实例，使用不可复用的内部 `collectibleId`。
- 每件藏品只允许一个当前持有人；未分配藏品的持有人为 `null`。
- 服务器保存当前归属和不可覆盖的归属历史；创建、管理员初始分配和拍卖成交都必须追加记录。
- 藏品图片源固定为芝加哥艺术博物馆 IIIF，正式图片 URL 由服务器按 `imageId` 生成：`https://www.artic.edu/iiif/2/{imageId}/full/{width},/0/default.jpg`。
- 管理员上传格式只保存芝加哥艺术博物馆藏品 ID、`imageId`、标题、艺术家、年代、材质、尺寸、说明、公版标记和可选初始持有人 ID，不接受任意图片 URL 或其他图片域名。
- 只有明确提供 `isPublicDomain: true` 的记录可以导入；重复的芝加哥艺术博物馆藏品 ID 必须拒绝。
- 图片外链加载失败时只影响展示，不得改变归属、拍卖或资金状态。
- 藏品目前不参与商品／工厂订单簿、资产估值或排行榜；后续若加入估值，必须在统一资产订单簿设计中定义唯一估值来源。

### 2.1 管理员 JSON 格式

管理员页面接受 JSON 数组或带 `items` 数组的对象，每批 1～100 条：

```json
[
  {
    "sourceArtworkId": 28560,
    "title": "The Bedroom",
    "artist": "Vincent van Gogh",
    "dateDisplay": "1889",
    "mediumDisplay": "Oil on canvas",
    "dimensions": "73.6 × 92.3 cm",
    "description": "可选说明",
    "imageId": "芝加哥艺术博物馆 image_id",
    "isPublicDomain": true,
    "initialOwnerId": 123
  }
]
```

`initialOwnerId` 可省略或设为 `null`；提供时必须对应已经存在的玩家。导入操作必须使用 `Idempotency-Key`，同一请求重试不得重复创建藏品。

## 3. 藏品拍卖

- 只有当前持有人可以发起拍卖，同一藏品同时最多存在一场进行中拍卖。
- 起拍价为正整数，拍卖时长为 1～168 小时。
- 卖家不能竞拍自己的藏品。
- 首次出价不得低于起拍价；后续出价至少比当前最高价高 1。
- 当前最高出价者的出价金额从可用资金转入冻结资金；被新的最高出价超过时，原最高出价立即完整释放。
- 同一最高出价者提高报价时只冻结差额。
- 没有出价的拍卖可由卖家取消；出现任何有效出价后不得取消。
- 拍卖结束由服务器在状态读取、游戏操作或管理员读取时自动结算。存在最高出价时，冻结资金支付给卖家、藏品归属转移给最高出价者并追加归属历史；没有出价时标记为流拍。
- 结算时若藏品归属异常、玩家不存在或冻结资金不足，拍卖取消并尽可能释放冻结资金，不得生成负余额或错误归属。
- 玩家存在进行中的出售拍卖或作为最高出价者参与拍卖时，禁止重置经济状态。

## 4. 玩家页面职责

- “藏品”页只展示当前账号持有的藏品，使用图片卡、作品信息、馆藏来源链接和拍卖状态。
- “拍卖”页负责发起拍卖、查看进行中拍卖、提交竞价、取消无出价拍卖以及查看最近结算结果。
- 页面不得从浏览器直接修改归属、最高出价、结束时间或冻结资金。

## 5. 管理员后台

管理员地址固定为 `https://game.riversoft.top/economy/admin`。普通主导航不显示管理员入口，设置页仅对 `role=admin` 显示入口。所有 `/api/game/admin/` 接口必须由服务器再次校验管理员角色，普通用户返回 403。

后台提供：

- 世界概况；
- 单个礼品码创建、一次批量生成最多 50,000 个礼品码、TXT 下载、停用与兑换记录；
- 按固定 JSON 格式导入芝加哥艺术博物馆公版藏品；
- 查看藏品图片、馆藏编号、当前持有人、拍卖状态和归属历史。

批量礼品码页面不得把最多 50,000 条明文逐项渲染到 DOM；只显示本次生成数量和单个码预览，并通过 Blob 下载一行一个兑换码的 TXT。创建按钮在请求期间必须禁用，避免管理员重复提交。明文只保留在当前页面状态中，礼品码历史列表继续只展示不可逆的元数据。

藏品列表、礼品码列表、归属历史和兑换记录可能持续增长，必须复用共享 `VirtualList`：

- API 返回的完整数组仍保留在客户端状态中；
- DOM 只渲染可视区域和少量预加载行；
- 表头与虚拟记录使用统一列定义，并保持横向滚动对齐；
- 每条记录必须使用服务器稳定 ID；兑换记录使用玩家 ID 与兑换时间组合键；
- 不得用分页、截断、`slice` 或只显示最近记录替代窗口化；
- 窗口化不得改变查看归属、停用礼品码、查看兑换记录或馆藏链接等操作。

后台不提供任意修改玩家资金、删除玩家、编辑世界 JSON、覆盖归属历史或上传二进制图片。管理员写操作必须使用 `Idempotency-Key`。

## 6. API

玩家接口：

- `POST /api/game/collectible-auctions`
- `POST /api/game/collectible-auctions/:auctionId/bids`
- `POST /api/game/collectible-auctions/:auctionId/cancel`

管理员接口：

- `GET /api/game/admin/gift-codes`
- `POST /api/game/admin/gift-codes`
- `POST /api/game/admin/gift-codes/batch`
- `POST /api/game/admin/gift-codes/:giftCodeId/disable`
- `GET /api/game/admin/gift-codes/:giftCodeId/redemptions`
- `GET /api/game/admin/collectibles`
- `POST /api/game/admin/collectibles/import`
- `GET /api/game/admin/collectibles/:collectibleId/ownership`

批量接口请求体包含 `count`、`rewardCredits`、`maxRedemptions`、可选 `startsAt`、可选 `expiresAt` 和可选 `note`。`count` 必须为 1～50,000 的整数；响应包含 `createdCount` 与 `codes` 明文数组，仅供本次 TXT 导出。

## 7. 防回退

不得：

- 将批量礼品码上限降到 50,000 以下，或把一批创建拆成可能部分成功的多个独立事务；
- 将礼品码明文、批量 TXT 内容或可逆密钥写入数据库；
- 在管理员页面一次渲染最多 50,000 条礼品码明文，或移除 TXT 下载能力；
- 对相同 `Idempotency-Key` 的批量重试生成第二批礼品码；
- 把藏品降级为仅保存在浏览器的图片列表；
- 允许一件藏品同时属于多个玩家；
- 直接信任客户端提交的图片 URL、当前持有人、最高出价或结束状态；
- 导入未明确标记为公版的芝加哥艺术博物馆图片；
- 在已有出价后允许卖家取消拍卖；
- 删除、覆盖或重写既有归属历史；
- 为规避图片流量而把远程作品自动复制到游戏服务器；
- 将藏品任意价格计入总资产而没有权威市场估值规则；
- 对管理员藏品、礼品码、归属或兑换记录恢复全量 `.map()` DOM 渲染；
- 因窗口化而隐藏、丢弃或限制管理员记录。
