# Economy 市场订单与资金资产信息架构

> 状态：当前市场、订单、成交、资金与资产记录的设计基线  
> 适用项目：`RIVERS0FT/Economy`  
> 更新时间：2026-07-11  
> 客户端状态版本：`version: 6`  
> 世界状态版本：`version: 3`

本文件定义市场订单管理与资金资产记录的页面归属、服务器事件模型、旧数据迁移和不可回退规则。

涉及商品目录、工厂目录、生产计划和产业规则时，以 `docs/INDUSTRY_AND_PRODUCTION_DESIGN.md` 为准；涉及视觉与响应式规则时，以 `docs/UI_DESIGN_SYSTEM.md` 为准；涉及服务器事务、幂等和部署时，以 `docs/SERVER_ARCHITECTURE_AND_DEPLOYMENT_DESIGN.md` 为准。

## 1. 核心决策

- 市场负责交易行为：商品选择、下单、订单簿、撤单、冻结资产摘要、成交记录和工厂挂牌。
- 资金负责资产结果：现金、冻结资金、商品库存、工厂估值、生产变化和完整资产事件。
- 不再提供独立“订单”导航或独立订单页面。
- 未完成订单数量显示在“市场”导航上。
- 玩家下单后可以在当前商品订单区直接撤单，也可以在市场的全部订单区管理所有商品订单。
- 成交记录保留在市场，因为它回答交易价格、数量、方向和对手方。
- 资产事件保留在资金页，因为它回答一次服务器事务对资金、商品、工厂和生产状态造成的实际变化。
- 成交记录和资产事件可以通过来源 ID 关联，但不得合并成一张职责混乱的表格。

## 2. 导航与路由

正式导航固定为：

```text
概览
市场
生产
资金
排行
设置
```

对应稳定路由 ID：

```text
home
market
production
assets
leaderboard
settings
```

不得恢复 `records` 导航或 `records` 路由分支。

旧客户端或旧会话即使仍保存 `records` 值，也不得出现空白页面。当前客户端不持久化标签；若未来增加持久化，必须将未知标签和 `records` 统一回退到 `market`。

## 3. 市场页面职责

市场页必须包含：

1. 商品选择；
2. 行情摘要；
3. 商品限价下单；
4. 当前商品未完成订单和快捷撤单；
5. 商品订单簿；
6. 价格历史；
7. 全部未完成订单；
8. 冻结资金与冻结商品摘要；
9. 成交记录；
10. 工厂挂牌、撤销挂牌和收购。

### 3.1 当前商品快捷订单

下单区域必须显示当前选中商品的所有未完成订单：

- 方向；
- 限价；
- 剩余数量与原始数量；
- 撤单按钮。

玩家不需要离开市场页即可撤销刚提交的订单。

### 3.2 全部未完成订单

市场的“我的订单与成交”面板必须显示所有商品的未完成订单：

| 字段 | 说明 |
|---|---|
| 商品 | 稳定商品 ID 对应的显示名称 |
| 方向 | 买入或卖出 |
| 限价 | 订单限价 |
| 剩余/原始 | 未成交数量与初始数量 |
| 冻结资产 | 买单冻结资金或卖单冻结商品 |
| 状态 | 等待成交或部分成交 |
| 时间 | 服务器创建时间 |
| 操作 | 直接撤单 |

订单管理继续受每名玩家最多 10 笔未完成订单限制。

### 3.3 成交记录

成交记录继续使用 `TradeRecord`，展示：

- 商品或工厂；
- 买入或卖出；
- 数量；
- 价格；
- 总额；
- 对手方；
- 服务器成交时间。

成交记录不得搬到资金页作为唯一展示，因为资产事件不替代市场成交语义。

## 4. 资金与资产页面职责

“资金”导航进入“资金与资产”页面。页面必须包含：

- 可用资金；
- 冻结资金；
- 当前总资产；
- 商品资产；
- 工厂资产；
- 仓库使用；
- 资产配置；
- 各商品库存和估值；
- 货币发行与系统回收；
- 完整资金与资产变动。

资金页不得再提供“前往独立流水页面”的按钮，因为完整资产事件已经在当前页面。

## 5. 统一资产事件模型

客户端状态版本升级为 `version: 6`，新增：

```ts
assetEvents: AssetEvent[];
```

资产事件结构：

```ts
interface AssetEvent {
  id: string;
  category:
    | 'work'
    | 'order'
    | 'trade'
    | 'inventory'
    | 'facility'
    | 'production'
    | 'system';
  createdAt: number;
  description: string;

  cashDelta: number;
  availableCashAfter: number;
  frozenCashDelta: number;
  frozenCashAfter?: number;

  inventoryChanges: Array<{
    productId: string;
    availableDelta: number;
    frozenDelta: number;
    availableAfter: number;
    frozenAfter: number;
  }>;

  facilityChanges: Array<{
    facilityId: string;
    facilityTypeId?: string;
    facilityName?: string;
    action: string;
    beforeStatus?: string;
    afterStatus?: string;
  }>;

  productionChanges: Array<{
    facilityId: string;
    facilityName?: string;
    action: 'produced' | 'collected';
    inputProductId?: string;
    inputQuantity: number;
    outputProductId?: string;
    outputQuantity: number;
    internalGoodsDelta: number;
    completedQuantityDelta: number;
  }>;

  sourceType?: 'order' | 'trade' | 'facility' | 'production' | 'work' | 'system';
  sourceId?: string;
  legacy?: boolean;
}
```

## 6. 服务器生成规则

资产事件只能由服务器在 SQLite 事务内生成。

服务器在操作前后捕获玩家资产快照，并计算：

- 可用资金变化；
- 冻结资金变化；
- 各商品可用与冻结库存变化；
- 工厂新增、移除、挂牌、启动、停止和状态变化；
- 工厂内部产成品和计划完成量变化；
- 玩家订单状态变化；
- 新成交记录。

一次服务器事务只生成一条复合资产事件，避免把同一操作拆成多个无法关联的流水行。

### 6.1 下单

买单事件记录：

- 可用资金减少；
- 冻结资金增加；
- 新订单来源 ID。

卖单事件记录：

- 可用商品减少；
- 冻结商品增加；
- 新订单来源 ID。

### 6.2 撤单

买单撤销记录：

- 可用资金增加；
- 冻结资金减少。

卖单撤销记录：

- 可用商品增加；
- 冻结商品减少。

### 6.3 成交

成交事件根据正式状态差异记录：

- 冻结资金或冻结商品减少；
- 实际退款或卖出收入；
- 商品可用库存变化；
- 新成交来源 ID。

订单提交时已经发生的冻结不得在成交时重复记作资金扣除。

### 6.4 生产

生产结算事件可以同时记录：

- 运营费扣除；
- 原料库存减少；
- 工厂内部产成品增加；
- 定量计划完成量增加；
- 工厂阻塞或计划完成状态。

### 6.5 领取产成品

领取事件记录：

- 工厂内部产成品减少；
- 玩家对应商品可用库存增加；
- 总资产可能保持不变，但资产位置发生变化。

### 6.6 工厂产权

工厂建设、挂牌、撤销挂牌、购买和出售必须记录工厂变化；购买和出售同时记录资金变化。

## 7. 读取与离线结算

玩家读取状态时，服务器仍会先按服务器时间处理：

- 离线生产；
- 施工完成；
- 人口或企业需求成交；
- 工厂阻塞状态。

如果读取过程实际改变玩家资产，必须在同一读取事务中生成资产事件；没有资产变化时不得生成空事件。

## 8. 幂等与事务

- 所有写操作继续使用 `Idempotency-Key`。
- 相同幂等键重复请求直接返回首次响应，不得再次生成资产事件。
- 资产事件与资金、库存、工厂、订单和成交在同一个 `BEGIN IMMEDIATE` 事务中提交。
- 事务失败时资产变化和事件必须一起回滚。
- 客户端不得自行补写、修改或删除正式资产事件。

## 9. 旧流水迁移

世界状态版本升级为 `version: 3`。

旧玩家的 `ledger` 在 SQLite 读取层迁移为 `assetEvents`：

| 旧字段 | 新字段 |
|---|---|
| `id` | 保留 |
| `category` | 映射到新事件分类 |
| `amount` | `cashDelta` |
| `balanceAfter` | `availableCashAfter` |
| `description` | 保留 |
| `createdAt` | 保留 |
| 无商品或工厂明细 | 不伪造，标记 `legacy: true` |

迁移要求：

- 可重复执行；
- 不删除旧 `ledger` 兼容数据；
- 不重复生成迁移事件；
- 不改变玩家资金、商品、工厂、订单或成交；
- 下一次保存时写回世界版本 3。

## 10. 展示与筛选

资金页资产事件提供：

```text
全部
资金
商品
工厂
生产
订单冻结
```

每条事件以卡片展示，允许同时出现多个变化块。移动端不得依赖横向大表格查看资产变化。

旧流水迁移事件应标记“历史流水迁移”，并仅显示旧数据真实包含的信息。

## 12. 架构检查契约

构建必须验证：

- 导航不包含 `records`；
- 路由不导入或渲染 `RecordsPage`；
- `src/pages/RecordsPage.tsx` 不存在；
- 市场页包含快捷撤单、全部未完成订单、冻结摘要和成交记录；
- 资金页标题为“资金与资产”，并使用 `assetEvents`；
- 客户端状态版本为 6；
- 世界状态版本为 3；
- `server/src/asset-events.js` 存在；
- SQLite 读取和写入事务均调用资产事件生成；
- 旧 `ledger` 迁移保留 `legacy` 标记；
- 幂等测试证明重复请求不会重复生成资产事件；
- 本文件中的不可回退规则存在。

## 13. 不可回退规则

除非先更新本设计、迁移和测试，否则不得：

- 恢复独立“订单”导航；
- 恢复 `records` 路由或 `RecordsPage`；
- 要求玩家离开市场页才能撤单；
- 从市场页移除未完成订单或成交记录；
- 把成交记录作为资金页唯一记录；
- 把资金流水与商品、工厂、生产变化重新拆成无法关联的独立记录；
- 由客户端生成正式资产事件；
- 在 SQLite 事务外写入资产事件；
- 重复幂等请求时再次生成事件；
- 删除旧流水迁移或伪造旧记录不存在的商品、工厂明细；
- 回退客户端状态版本 6 或世界状态版本 3；
- 绕过架构检查合并上述回退。

## 14. 验收标准

1. 下单后当前市场立即显示未完成订单。
2. 玩家可以在下单区域或全部订单表直接撤单。
3. 撤单后冻结资金或冻结商品立即释放。
4. 市场展示全部商品订单和全部成交记录。
5. 导航不再显示“订单”。
6. “资金”进入“资金与资产”页面。
7. 同一次操作的资金、商品和工厂变化显示为一条复合事件。
8. 生产事件同时显示运营费、原料和产出变化。
9. 旧流水无损迁移并标记为历史记录。
10. 重复幂等请求不会重复改变资产或生成事件。
11. 桌面、紧凑桌面和移动端均可直接撤单和查看资产事件。
12. 完整架构检查、领域测试、TypeScript 和生产构建通过。

未更新本设计、迁移和测试的市场或资产信息架构回退不应合并。
